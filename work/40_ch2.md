### 一、输入输出流（iostream）回顾
#### 流类层次结构与基类抽象 【重点】
【定义】  
`iostream` 类库以抽象基类为核心，组织为如图所示的层次结构：

- `ios_base` — 最底层抽象，管理格式状态等通用功能。
- `ios` — 继承自 `ios_base`，提供指向 `streambuf` 的指针。
- `istream` — 输入流抽象基类，重载 `>>`，提供 `get`、`peek`、`ignore` 等输入操作。
- `ostream` — 输出流抽象基类，重载 `<<`，提供 `put`、`flush` 等输出操作。
- `iostream` — 多重继承 `istream` 和 `ostream`，同时支持输入、输出。
- `ifstream` / `ofstream` / `fstream` — 文件流，分别继承 `istream` / `ostream` / `iostream`。
- `istringstream` / `ostringstream` / `stringstream` — 字符串流，同理继承。

`cin` 是 `istream` 的全局对象（标准输入），`cout` / `cerr` / `clog` 是 `ostream` 的全局对象（标准输出/错误/日志）。

原理与推导  
通过面向对象的多态设计，所有流都提供了统一的接口，例如 `operator>>` 和 `operator<<`。  
- `ostream` 内部为基本类型（`int`, `double`, `const char*` 等）重载了一系列 `operator<<`，调用系统函数完成实际 I/O。  
- 用户自定义类型只需定义 `ostream& operator<<(ostream&, const T&)` 全局函数或友元，即可无缝接入流接口。  
- 链式调用依赖 `<<` 左结合性和返回 `ostream&` 的引用，形如：  

```cpp
// 简化的 ostream 内部实现示意
class ostream {
public:
    ostream& operator<<(char c) {
        printf("%c", c);
        return *this;
    }
    ostream& operator<<(const char* str) {
        printf("%s", str);
        return *this;
    }
} cout;
```

执行 `cout << "hello" << ' ' << "world";` 时，各次调用均返回 `cout` 自身引用，实现连续输出。

【解题步骤】  
1. 识别类层次：根据需求（文件/字符串/标准 I/O）选择对应的流类。  
2. 编写输出运算符重载：若 `Test` 类需要 `cout << obj`，定义 `std::ostream& operator<<(std::ostream& out, const Test& src)`。  
3. 确保返回 `ostream&` 以支持链式调用。  
4. 若需同时支持输入，对称定义 `istream& operator>>(istream& in, Test& dst)`。

【易错点】  
- 将 `operator<<` 定义为成员函数会使得左操作数必须为自定义类，破坏链式调用习惯；应定义为友元或全局函数。  
- 返回类型必须是 `ostream&` 而非 `ostream`，否则不能连续 `<<`。  
- 忘记 `const` 修饰自定义类型参数可能导致右值实参无法绑定。  
- 混淆 `istream` 和 `ostream` 的继承关系，误用 `cin` 的类型。

**例题**：（自编）  
**题目**：阅读代码，指出编译错误的原因。  
```cpp
#include <fstream>
using namespace std;
int main() {
    ifstream ifs("input.txt");
    ostream out = ifs;   // 错误1
    cout = out;          // 错误2
    return 0;
}
```
**思路**：错误1：`ifstream` 继承自 `istream`，而 `ostream` 与 `istream` 是兄弟类，无法将 `ifstream` 对象隐式转换为 `ostream` 或拷贝。错误2：`cout` 是 `ostream` 对象，`ostream` 的拷贝赋值运算符被删除，禁止复制。  
**解答**：  
- 错误1：`ifstream` 仅继承自 `istream`，非 `ostream` 的派生类，无法进行类型转换。若需要输出，应使用 `ofstream` 或 `fstream`。  
- 错误2：所有流对象（包括 `cout`）均不可复制，其拷贝构造和拷贝赋值被删除。即使类型匹配，也不能将一个 `ostream` 对象赋值给另一个。  
**技巧**：牢记流类继承层次：`ios_base` → `ios` → `istream`/`ostream` → `iostream` → `fstream`/`stringstream`；`ifstream` 仅继承 `istream`，不能作输出流。流对象不可复制，必须以引用形式传递和返回。

#### 流操纵算子原理与自定义实现 【高频】
【定义】  
流操纵算子（Stream Manipulator）是一类可改变流格式状态的特殊对象或函数，它们通过 `operator<<`（或 `operator>>`）被插入到流中，例如 `endl`、`fixed`、`setprecision`、`setw` 等。

原理与推导  
实现流操纵算子有两种典型模式：

1. **无参数函数式算子（C++标准定义）**：`endl` 实际是函数 `ostream& endl(ostream& os)`，它输出 `\n` 并调用 `os.flush()`。为了支持 `cout << endl`，`ostream` 重载了函数指针版本的 `operator<<`：

```cpp
// endl 函数定义
ostream& endl(ostream& os) {
    os.put('\n');
    os.flush();
    return os;
}

// ostream 中对函数指针的重载
ostream& operator<<(ostream& (*fn)(ostream&)) {
    return (*fn)(*this);
}
```

这样 `cout << endl` 就等价于 `endl(cout)`。

2. **有参数对象式算子（编译器自定义）**：某些操纵算子（如 `setprecision`）需要参数，通常借助辅助类实现。

```cpp
// 辅助类
class setprecision {
    int precision;
public:
    setprecision(int p) : precision(p) {}
    friend class ostream;   // 允许 ostream 访问私有成员
};

// ostream 中对应重载
class ostream {
    int precision;
public:
    ostream& operator<<(const setprecision& m) {
        precision = m.precision;
        return *this;
    }
};
```

使用 `cout << setprecision(2);` 时，`setprecision(2)` 构造临时对象，`operator<<` 提取其参数并修改流的精度状态。

【解题步骤】  
1. 记住常用的预定义操纵算子：`endl`（换行并刷缓冲区）、`fixed` / `scientific` / `defaultfloat`（浮点格式）、`setprecision(n)`、`setw(n)`、`setfill(ch)`、`oct` / `hex` / `dec` 等。  
2. 理解两层重载：`endl` 型算子依赖函数指针重载；`setprecision` 型算子依赖自定义类对象及对应 `operator<<`。  
3. 编写自定义无参操纵算子时，模仿 `endl` 的函数签名和对应 `operator<<`；编写有参算子时，定义辅助类并为其提供 `operator<<` 重载。  
4. 注意 `iomanip` 头文件提供 `setprecision`、`setw` 等，使用时需 `#include <iomanip>`。

【易错点】  
- 混淆 `endl` 和 `'\n'`：`endl` 额外调用 `flush()`，用于需要立即输出的场景（如交互式程序）；仅想换行而不强制刷新时用 `'\n'` 效率更高。  
- 认为所有操纵算子的实现方式都由标准严格定义：实际上 `setprecision` 的实现因编译器而异（辅助类或其它技术），但效果一致。  
- 在 `<<` 链中忘记操纵算子返回流的引用，导致链式调用中断（实现者易错）。  
- 自定义操纵算子时未在正确命名空间（`std`）中声明，可能找不到匹配的重载。

**例题**：  
**题目**：（来源 [L11-iostream&函数对象和智能指针-2026.pptx P10]）  
```cpp
cout << setprecision(2) << 1.05 << endl;
```
请问输出结果是什么？

**思路**：`setprecision(2)` 设置输出精度为2位有效数字（默认浮点格式下）；`1.05` 用默认格式输出，取2位精度应四舍五入为 `1.1`；`endl` 换行并刷新。

**解答**：  
```
1.1
```
（`1.05` 保留两位精度后四舍五入得 `1.1`，小数点后的零不显示。）

**技巧**：`setprecision` 影响的是有效数字位数，若与 `fixed` 结合则变成小数位数，需注意区分。

#### 格式化输出控制 【重点】
【定义】  
利用 `<iomanip>` 中操纵算子（也可直接调用 `ios_base` 的 `precision()` 等成员函数）控制输出格式，包括浮点格式、进制、域宽、填充字符等。

原理与推导  
流内部维护了一系列格式状态标志，`operator<<` 会根据这些标志转换数据。操纵算子通过修改状态标志达到格式化效果。常用操纵算子及其作用：

| 操纵算子 | 作用 |
|--------|------|
| `fixed` / `scientific` / `defaultfloat` | 定点/科学计数法/恢复默认浮点格式 |
| `setprecision(n)` | 设置浮点输出精度（配合 `fixed` 为小数位数，否则为有效位数） |
| `setw(n)` | 设置下一次输出的最小域宽 |
| `setfill(ch)` | 配合 `setw`，指定填充字符（默认空格） |
| `oct` / `hex` / `dec` | 八进制/十六进制/十进制输出整数 |
| `left` / `right` / `internal` | 域内对齐方式 |
| `endl` / `flush` / `ws` | 换行并冲缓冲/只冲缓冲/提取空白符 |

所有操作都通过插入到 `cout` 等流中完成。需注意 `setw` 只对紧随其后的一次输出有效，而 `fixed`、`setprecision` 等持续有效直到被修改。

【解题步骤】  
1. 确定需要控制的格式类型（浮点、进制、对齐）。  
2. 选择对应操纵算子，将其插入到输出链中，位置放在待输出量之前。  
3. 需要域宽时，`setw` 紧挨在目标数据前，且注意其“一次性”特性。  
4. 结合 `fixed` 与 `setprecision` 控制小数位数是常见组合。  
5. 结束特殊格式后常用 `defaultfloat`、`dec` 等恢复默认状态，避免影响后续输出。

【易错点】  
- 误以为 `setprecision` 总是设置小数位数；仅在启用 `fixed` 或 `scientific` 时才是小数位数，默认是有效数字。  
- `setw` 仅影响下一次输出，之后域宽恢复默认（0）；忘写 `setw` 导致域宽不对齐。  
- 进制转换后忘记用 `dec` 恢复十进制，使后续整数都以八进制或十六进制展示，发现 bug 困难。  
- 填充字符需要用 `setfill` 配合 `setw`，单独 `setfill` 不起显示效果。  

**例题**：  
**题目**：（来源 [L11-iostream&函数对象和智能指针-2026.pptx P9]）  
请写出以下代码的输出结果：

cout << oct << 12 << " " << hex << 12 << endl;
cout << dec;
cout << setw(3) << setfill('*') << 5 << endl;
```

**思路**：逐行分析格式状态与数据：第一行定点格式，默认精度6位小数；第二行科学计数法；第三行 `defaultfloat` 恢复默认浮点，接着 `setprecision(2)` 设置有效数字位数2；第四行八进制和十六进制输出12；第五行十进制恢复后，`setw(3)` 配合 `setfill('*')` 产生带星号填充的宽度3的整数5。

2.018000e+03 1.000000e-04

14 c

```

**技巧**：遇到复杂格式化输出时，从流的“当前状态”出发，列出每个操纵算子对状态的影响，顺序分析即可避免遗漏。

#### 不能复制的对象语义 【高频】
【定义】  
`ostream`（以及所有流对象）不可复制，只支持移动。其复制构造函数为 `delete`，提供了移动构造函数。

```cpp
ostream(const ostream&) = delete;
ostream(ostream&& x);
```

`cout` 是一个全局唯一的 `ostream` 对象，不能被复制。

原理与推导  
为什么流对象不能复制？  
1. **资源唯一性**：每个流对象关联着一个唯一的底层缓冲区（`streambuf`），如果允许多个流对象共享同一个缓冲区，会造成输出混乱或竞态，复制语义难以管理。  
2. **避免开销**：流的内部状态（格式标志、异常状态、绑定流等）复杂，拷贝成本高且语义不清晰。  
3. **面向对象思想**：一个全局 `cout` 对象封装了标准输出，符合“单一职责”和“全局接入点”的设计。

重载 `operator<<` 时，必须返回 `ostream&` 引用，从而：  
- 避免产生无效的拷贝请求。  
- 支持链式调用，且始终操作同一个流对象。

实际上，如果 `operator<<` 返回值类型不是引用，会触发已删除拷贝构造函数的编译错误。

【解题步骤】  
1. 编写流输出运算符时，第一参数为 `std::ostream&`，返回类型为 `std::ostream&`。  
2. 返回语句通常是 `return out;`，在函数内调用 `out << a << b;` 后返回引用。  
3. 理解 `cout` 单例性质，避免试图创建另一个 `ostream` 对象（除非通过已提供的其他全局对象如 `cerr`）。  
4. 当需要将 `ostream` 作为参数传递时，一律使用引用（或指针）。

【易错点】  
- 误将 `operator<<` 返回值写为 `ostream` 导致编译错误（尝试复制）。  
- 混淆 `cin`、`cout` 是类还是对象：它们是对象，`cin` 是 `istream` 对象，`cout` 是 `ostream` 对象。  
- 移动语义允许通过 `std::move` 改变流的所有权，但一般极少使用，多数情况下只是将 `ifstream` 等作为局部变量传递（如 `std::ifstream ifs("file")`），函数返回时自动关闭。  
- 对 `cout` 使用 `auto x = std::move(cout);` 会使得 `cout` 处于未指定状态，后续不能再安全使用，属于危险操作。

**例题**：

### 二、字符串流（stringstream）回顾
#### 缓冲区操作与状态位管理 【高频】
【定义】  
`std::stringstream`（定义在 `<sstream>`）内部维护一个 `std::string` 缓冲区，同时提供输入（读取）和输出（写入）功能。它继承自 `iostream`，因而同时具备 `istream` 和 `ostream` 的方法。

通过 `operator<<` 向缓冲区写入数据，通过 `operator>>` 从缓冲区读取数据，就像操作标准输入输出一样，但数据全部驻留在内存中。

原理与推导  
写入时数据被追加到内部缓冲区末尾（类似 `operator+=`），读取时按照流位置前进，从缓冲区头开始逐个读取解析。  
- 状态位（`eofbit`, `failbit`, `badbit`）同样适用：当读取超出缓冲区末尾时，将置 `eofbit`；格式解析失败则置 `failbit`。  
- 可以使用 `clear()` 重置状态位，用 `str("")` 清空缓冲区，用 `peek()` 查看下一个字符而不提取。

常用模式：  

```cpp
stringstream ss;
ss << 10 << " " << 20;   // 写入 "10 20"
int a, b;
ss >> a >> b;            // 读取成功, a=10, b=20
```

【解题步骤】  
1. 包含 `<sstream>`，构造 `stringstream` 对象（可传入初始字符串）。  
2. 将数据格式化为字符串：用 `<<` 连续写入，完全如 `cout`。  
3. 从字符串解析数据：用 `>>` 读取，注意类型匹配，失败时处理状态位。  
4. 若需要复用流，先 `clear()` 再 `str("")` 重置。  
5. 多次读取时留意流内位置，必要时使用 `seekg(0)` 回到开头。

【易错点】  
- 读取失败后流状态变为 `fail`，必须调用 `clear()` 才能继续操作，否则后续读写被忽略。  
- 缓冲区和流位置是独立的概念：写入后读取位置在缓冲区头，无需手动定位；但若先部分读取再追加写入，可能需要 `clear()` + `seekp(0, ios::end)` 保证一致性。  
- `operator>>` 以空白字符为分隔，读取字符串数据时不保留内部的空格。  
- 忘记包含 `<sstream>` 头文件导致编译错误。

**例题**：（自编）  
**题目**：以下程序期望分别读取两个整数 10 和 20，但实际运行结果未知。请分析可能的问题并修正。  
```cpp
#include <sstream>
#include <iostream>
using namespace std;
int main() {
    stringstream ss;
    ss << 10;
    int a, b;
    ss >> a;
    ss << 20;
    ss >> b;
    cout << a << ' ' << b << endl;
    return 0;
}
```
**思路**：第一次 `ss << 10` 后缓冲区为 `"10"`，`ss >> a` 读取成功后缓冲区为空，流状态变为 `eof`。此时流状态非 `good`，后续 `ss << 20` 写入操作被忽略，导致 `b` 未被正确赋值。  
**解答**：  
程序可能输出 `10` 和一个随机值，因为 `b` 未初始化或读取失败。修正：在读取 `a` 之后、写入 `20` 之前调用 `ss.clear()` 清除错误状态，并可选 `ss.str("")` 清空缓冲区，保证流处于可写状态。  
```cpp
ss >> a;
ss.clear();
ss.str("");  // 清空缓冲区
ss << 20;
ss >> b;
```
**技巧**：`stringstream` 复用必须同时清空错误状态和缓冲区内容，`clear()` 和 `str("")` 缺一不可。

#### 类型转换与模板函数结合 【必背】
【定义】  
`stringstream` 经常作为类型转换的工具：字符串到数值（`"42"` → `int`），数值到字符串等。结合模板函数可写出通用的转换代码。

原理与推导  
输入输出流对基本类型已经重载了 `>>` 和 `<<`，因此：

```cpp
// 数值 -> 字符串
template<typename T>
std::string toString(const T& val) {
    std::ostringstream oss;
    oss << val;
    return oss.str();
}

// 字符串 -> 数值
template<typename T>
T fromString(const std::string& str) {
    T val;
    std::istringstream iss(str);
    iss >> val;
    return val;
}
```

利用 `ostringstream`（只输出）和 `istringstream`（只输入）分别实现转换，避免了 `sprintf`/`scanf` 的类型不安全和格式指定问题。

【解题步骤】  
1. 数值→字符串：`ostringstream oss; oss << value; return oss.str();`。  
2. 字符串→数值：`istringstream iss(str); T val; iss >> val;`，注意检查 `iss.fail()` 来处理格式错误。  
3. 封装成模板函数，提高代码复用性。  
4. 结合 `std::stoi`、`std::stod` 等现代 C++ 函数也可以完成转换，但 `stringstream` 更通用且适合自定义类型。

【易错点】  
- 使用 `stringstream` 同时读写时，若未将流位置重置容易读取到错误数据。最好分别用 `ostringstream` 和 `istringstream` 各司其职。  
- 转换失败时不检查状态位，导致未初始化的 `val` 被当做有效值。记得添加 `if(iss.fail())` 处理。  
- 若字符串含有空格，`>>` 只会读取第一个单词，需要用 `std::getline(iss, val)` 读取完整一行。  
- 与 `std::to_string` / `std::stoi` 混淆：前者直接转换数值到字符串（便捷），后者将字符串转为数值并可能抛出异常；`stringstream` 方式更灵活（可处理自定义类型，适用于流格式化）。

**例题**：  
**题目**：（来源 [L11-iostream&函数对象和智能指针-2026.pptx P23]）  
```cpp
#include <sstream>
using namespace std;

int main() {
    stringstream ss;
    ss << "10";
    ss << "0 200";

    int a, b;
    ss >> a >> b;   // a = ?, b = ?
    return 0;
}
```
问：变量 `a` 和 `b` 的值分别是多少？

**思路**：`ss` 初始为空，写入 `"10"` 后缓冲区为 `"10"`，再写入 `"0 200"`，缓冲区变为 `"100 200"`。随后提取两个 `int`，第一个读到 `100`，第二个读到 `200`。

**解答**：  
`a` = 100，`b` = 200。

**技巧**：`stringstream` 的写入是连续追加的，多个 `<<` 操作直接拼接字符串，无自动分隔。本题中 `"10"` 和 `"0"` 直接邻接变成 `"100"`，因此解析时作为一个完整数字。使用时应留意这种拼接效应。

#### 头文件与继承关系 【重点】
【定义】  
字符串流相关的头文件为 `<sstream>`，其中定义了三个类模板（并用 `char` 实例化出我们常用的类型）：

| 类 | 继承关系 | 用途 |
|----|----------|------|
| `std::istringstream` | 继承 `std::istream` | 输入字符串流，只读 |
| `std::ostringstream` | 继承 `std::ostream` | 输出字符串流，只写 |
| `std::stringstream` | 继承 `std::iostream` | 输入输出字符串流，可读写 |

`iostream` 又由 `istream` 和 `ostream` 多重继承而来，因此 `stringstream` 接口涵盖全部输入输出操作。

原理与推导  
继承结构使得 `stringstream` 对象可以作为 `istream&` 或 `ostream&` 参数传递给任何接受这些基类的函数。例如，`getline(ifs, str)` 同样可写为 `getline(std::cin, str)` 或 `getline(ss, str)`，因为函数接受 `istream&`。

与文件流类似，字符串流通过重写基类虚函数，操作内部 `stringbuf`（派生自 `streambuf`）实现内存中的 I/O。

【解题步骤】  
1. 只在需要内存字符串格式化时引入 `<sstream>`。  
2. 根据只需要读、写、还是都要选用 `istringstream` / `ostringstream` / `stringstream`。  
3. 将字符串流对象当作 `istream` 或 `ostream` 使用即可，无需关注底层。  
4. 若要将自定义类型也支持 `<<`/`>>` 与字符串流配合，只需保证该自定义类型已有对应的重载运算符。

【易错点】  
- 只包含 `<iostream>` 无法使用 `stringstream`，必须包含 `<sstream>`。  
- 混淆文件流头文件 `<fstream>` 与字符串流头文件 `<sstream>`。  
- 用 `stringstream` 却忘记其继承自 `iostream`，导致误认为不能使用某些 `istream`/`ostream` 的成员函数（实际全部可用）。

**例题**：  
**题目**：（来源 [L11-iostream&函数对象和智能指针-2026.pptx P27]）  
下列说法错误的是（　）。  
A. `scanf` 有可能写入非法内存  
B. `stringstream` 是继承自 `iostream` 的子类  
C. `stringstream` 既可以作为输入流，也可以作为输出流  
D. `ifstream` 是 `iostream` 的子类，能同时从文件中读入数据和写出数据  

**思路**：逐项检查：A 正确，`scanf` 若格式控制不当可能写入非法地址；B 正确；C 正确；D 错误，`ifstream` 继承自 `istream`，只能读，不能写。  
**解答**：错误答案是 **D**。  
**技巧**：抓住流类继承体系核心——只有 `iostream` 的派生类（`stringstream`、`fstream`）才同时拥有输入输出能力，`ifstream` 仅继承 `istream`。

### 三、文件流（fstream）回顾
#### 文件打开模式与操作生命周期 【重点】
【定义】  
文件流操作声明于 `<fstream>`，主要类及其继承关系：

- `std::ifstream` — 输入文件流，继承 `istream`
- `std::ofstream` — 输出文件流，继承 `ostream`
- `std::fstream` — 输入输出文件流，继承 `iostream`

打开模式用标志位组合（`std::ios_base::openmode`），常用模式：

| 模式 | 含义 |
|------|------|
| `in` | 读模式（`ifstream` 默认） |
| `out` | 写模式（`ofstream` 默认） |
| `binary` | 二进制模式 |
| `ate` | 打开时定位到文件末尾 |
| `app` | 追加模式（每次写入都定位到文件末尾） |
| `trunc` | 打开前清空文件（`out` 默认） |

构造时可直接指定文件名和模式，如：  
```cpp
ifstream ifs("input.txt");                     // 默认文本读
ifstream ifs("data.bin", ifstream::binary);   // 二进制读
ofstream ofs("output.txt", ofstream::out | ofstream::app); // 追加写
```

原理与推导  
文件流对象在构造时打开文件，内部调用操作系统的文件打开函数，建立文件缓冲区。  
- 对象析构时自动关闭文件（RAII）。  
- 也可显式调用 `open()` / `close()`；`close()` 后可以再次打开另一个文件。  
- 判断文件是否成功打开：可以检查流对象本身（它重载了 `operator bool`），如 `if(!ifs) { /* 处理错误 */ }`。

【解题步骤】  
1. 包含 `<fstream>`，声明文件流对象并传递文件名/模式。  
2. 使用前检查流对象是否有效（`if(ifs)` 或 `ifs.is_open()`）。  
3. 通过 `<<` / `>>` 或 `getline` 进行读写，操作方式与 `cin` / `cout` 一致。  
4. 操作完毕，自动析构或显式 `close()` 关闭文件，保证数据写回磁盘。  
5. 需要改变关联文件时，先 `close()` 再 `open("other.txt")`。

【易错点】  
- 忘记检查文件打开状态，导致读取数据为空或写文件无输出。  
- 写入文件后未关闭或刷新缓冲区，程序异常终止时文件内容可能丢失（除非显式 `flush()` 或 `close()`）。  
- 如果在文本模式下读二进制文件，可能因 `0x0D` `0x0A` 转换导致数据错误，务必使用 `binary` 模式。  
- 对象被复制会使多个对象关联同一文件句柄？流对象禁止复制，但移动会造成悬空状态。通常只需作为局部变量即可。

**例题**：（自编）  
**题目**：以下程序希望每次运行时向日志文件追加一行记录，但实际每次运行后文件只有最新的一行。分析原因并修正。  
```cpp
#include <fstream>
using namespace std;
int main() {
    ofstream log("log.txt");
    log << "New log entry" << endl;
    return 0;
}
```
**思路**：`ofstream` 默认打开模式为 `out | trunc`，每次打开文件时会清空原有内容。追加写入应使用 `app` 模式。  
**解答**：  
修改打开方式为追加模式：
```cpp
ofstream log("log.txt", ios::app);
```
或使用 `fstream` 并指定 `ios::app`。这样每次写入自动定位到文件末尾，不清空已有数据。  
**技巧**：文件流默认模式：`ifstream` 默认 `in`，`ofstream` 默认 `out | trunc`。需要追加时必须显式增加 `app` 标志。

#### 输入读取策略与流状态检查 【高频】
【定义】  
从文件流读取数据的常用手段：`>>`（格式化提取）、`getline`（读一行）、`get` / `peek` / `ignore`（细粒度控制）。  
流状态标志（`good()`, `eof()`, `fail()`, `bad()`）用于判断读取是否成功。

原理与推导  
`ifstream` 继承了 `istream` 的全部输入成员。重要成员函数：

- `operator bool()` — 对流是否处于可读状态进行隐式判断，通常用作 `while(ifs)`
- `ifs >> ws` — 提取前导空白，但不读取有效字符（`ws` 是操纵算子）
- `ifs.peek()` — 返回下一个字符而不提取，`EOF` 表示结束
- `ifs.get()` — 读取一个字符，失败返回 `EOF`
- `ifs.ignore(n, delim)` — 跳过最多 `n` 个字符或直到遇见 `delim`

典型读取策略示例（P18 演示）：

```cpp
ifstream ifs("input.txt");
while (ifs) {                // 只要流状态为 good
    ifs >> ws;               // 跳过空白
    int c = ifs.peek();      // 偷看下一个字符
    if (c == EOF) break;
    if (isdigit(c)) {        // 是数字？
        int n;
        ifs >> n;
        cout << "Read a number: " << n << endl;
    } else {
        string str;
        ifs >> str;
        cout << "Read a word: " << str << endl;
    }
}
```

这段代码展示了安全读取的方式：先去除空白，预览，再根据字符类型选择不同读取策略。

【解题步骤】  
1. 打开文件后进行循环读取：`while (ifs)` 或 `while (getline(ifs, line))`。  
2. 利用 `peek()` 预判数据类型，避免用错读取方式。  
3. 每次格式化输入 (`>>`) 后检查 `ifs.fail()`，若类型不匹配会导致 `failbit`，需中断或跳过。  
4. 使用 `ifs.clear()` 重置错误状态，`ifs.ignore(n, '\n')` 跳过错误行，继续处理。  
5. 读取二进制文件时用 `read(char*, streamsize)`，并结合 `gcount()` 获取实际读取字节数。

【易错点】  
- 习惯 `while (!ifs.eof())` 控制循环：文件末尾标志只有在尝试读取越过末尾后才设置，会导致最后一次读取被处理两次或产生错误。正确方式是 `while (ifs >> var)` 或 `while (getline(ifs, line))`。  
- `peek()` 返回 `int` 类型，不可直接赋给 `char`，否则 `EOF` 值可能无法区分（比较前先转为 `int`）。  
- 混合使用 `getline` 和 `>>` 时，`>>` 会遗留换行符在缓冲区，导致后续 `getline` 读取空行；需要用 `ifs.ignore()` 消耗掉残留分隔符。  
- 忘记 `ifs >> ws` 去除空白导致 `peek()` 看到空格而误判。

**例题**：  
**题目**：（来源 [L11-iostream&函数对象和智能指针-2026.pptx P18]）设有文件 `input.txt` 内容如下：  
```

```
请写出以下程序的完整输出。  
```cpp

    while (ifs) {

            ifs >> n;

**思路**：循环利用 `peek` 预判下一个字符类型，决定用 `>> int` 还是 `>> string` 读取；`>> ws` 跳过空白。文件内容按顺序读取，依次遇到数字、单词、数字、单词。

```

### 四、函数对象（Function Object）回顾

#### 函数指针与函数对象的概念统一 【高频】
- 【定义】C++ 中，函数调用既可以通过函数指针实现，也可以通过**函数对象**（重载了 `operator()` 的对象）实现。STL 算法（如 `sort`）通常将其比较准则的类型参数化为模板类型 `Compare`，可以接受任何可调用实体，包括函数指针和函数对象。这种统一依托于模板的“鸭子类型”：只要 `comp(a,b)` 语法有效，即可编译通过。
- 原理与推导：  
  函数指针示例：`bool (*comp)(int,int) = myCompare;`，然后 `comp(x,y)` 调用。  
  函数对象示例：`greater<int> compObj;`，然后 `compObj(x,y)`。  
  `sort` 声明为：
  ```cpp
  template<class Iterator, class Compare>
  void sort(Iterator first, Iterator last, Compare comp);
  ```
  `Compare` 既可以是 `bool(*)(int,int)`，也可以是 `greater<int>` 类类型。编译器在实例化模板时检查 `comp(*i,*j)` 是否合法，而不关心具体类型。  
  函数对象可以携带内部状态（如调用计数），且其调用操作符允许内联优化，性能通常优于函数指针。
- 【解题步骤】
  1. 如果只需指定一种固定比较逻辑且无内部状态，使用普通函数 + 函数指针即可。
  2. 若需要更强的内联性、或要携带额外数据（如阈值），定义函数对象类并重载 `operator()`。
  3. 当需要将多种可调用类型传递给同一个泛型函数时，使用模板参数（如 `Compare`）自动适配。
- 【易错点】
  - 函数指针声明时可读性差，善用 `auto` 或 `using` 别名：`auto f = myFunc;` 或 `using Cmp = bool(*)(int,int);`。
  - 函数对象需要先实例化：`sort(v.begin(), v.end(), greater<int>())`，括号不可省略；若写成 `greater<int>` 则传递的是类型而非对象，导致编译错误。
  - 函数对象作为比较器时，应声明 `operator()` 为 `const`，因为标准算法可能要求函数调用不修改比较器本身。

| 对比维度 | 函数指针 | 函数对象 |
|----------|----------|----------|
| 类型 | `RetType (*) (Args...)` | 含有 `operator()` 的类类型 |
| 是否可带状态 | 否（静态函数） | 可以，成员变量保存状态 |
| 内联优化 | 难以内联 | 可内联，调用开销更小 |
| 典型用法 | `sort(v.begin(),v.end(),cmpFunc);` | `sort(v.begin(),v.end(),greater<int>());` |

**例题**：
**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P41] 实现一个 `mySort` 函数，使其既能接受普通比较函数，又能接受 `greater<int>()` 对象。
**思路**：模板化比较器类型。
**解答**：
```cpp
template<class Iterator, class Compare>
void mySort(Iterator first, Iterator last, Compare comp) {
    for (auto i = first; i != last; ++i)
        for (auto j = i; j != last; ++j)
            if (!comp(*i, *j))
                swap(*i, *j);
}
// 调用：
mySort(arr, arr+5, comp);           // comp 为函数指针
mySort(arr, arr+5, greater<int>()); // greater<int>() 为函数对象
```
**技巧**：STL 算法通过模板将比较准则提升为“可调用对象”概念，这使得 lambda 表达式也可直接传入（本质是匿名函数对象）。

#### operator()重载与鸭子类型 【必背】
- 【定义】函数对象的核心是重载 `operator()`。C++ 没有定义函数对象的统一接口，而是依赖“**鸭子类型**”（Duck Typing）：如果一个对象支持 `()` 调用语法，它就可以当作函数使用，无需继承特定基类。
- 原理与推导：  
  一个典型的比较函数对象如下：
  ```cpp
  template<class T>
  class greater {
  public:
      bool operator()(const T& a, const T& b) const {
          return a > b;
      }
  };
  ```
  当使用 `greater<int>()` 创建匿名对象，并将其传递给 `sort`，编译器看到 `comp(*i,*j)` 调用的是 `greater<int>::operator()`，只要该成员存在且可访问，编译就能通过。这种静态检查机制即为“鸭子类型”。  
  如果希望对象在被拷贝后仍保持行为一致，通常将 `operator()` 声明为 `const`，表示调用不会改变对象内部状态。
- 【解题步骤】
  1. 定义一个类，将行为逻辑写入 `public` 成员函数 `RetType operator() (参数列表)`。
  2. 若该对象仅作为比较器且自身无需修改，将成员函数声明为 `const`，参数尽量用 `const T&`。
  3. 创建对象：`ClassName()` 或 `ClassName obj;`，然后即可使用 `obj(...)`。
  4. 理解该机制后，可顺理成章地使用 lambda 表达式（编译器自动生成匿名函数对象类）。
- 【易错点】
  - `operator()` 必须设为 `public`，否则外部不可见。
  - 常犯错误：将 `operator()` 声明为静态 (`static`) 或虚函数，前者失去对象特性，后者破坏内联和模板调用。
  - 混淆“对象类型”和“实例”：`sort(v.begin(), v.end(), greater<int>)` 缺少括号，传递了类型而非对象，编译报错；正确为 `greater<int>()`。
  - 缺失 `const` 限定可能导致与标准库配合时编译出错，尤其当算法期望比较器为 const 可调用时。

**例题**：
**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P38] 手动实现一个 `greater` 模板类，使 `greater<int>()(2,1)` 返回 `true`，`(1,1)` 返回 `false`。
**思路**：重载 `operator()`，参数 `const T&`，返回 `bool`。
**解答**：
```cpp
template<class T>
class greater {
public:
    bool operator()(const T& a, const T& b) const {
        return a > b;
    }
};
```
**技巧**：注意三个 `const` —— 两个参数 `const` 防止误改元素，成员函数 `const` 保证比较器对象状态不变。该写法与标准库 `std::greater<T>` 一致。

#### 比较器在STL中的深度应用 【高频】
- 【定义】在 `std::sort`、`std::priority_queue`、`std::set`/`std::map` 等容器与算法中，比较器决定了元素的排序或优先级。比较器可以是函数指针或函数对象（包括预定义的 `less<T>`、`greater<T>`）。自定义类型排序时，有两个典型方案：1）重载类型自身的 `operator<`；2）提供独立的比较器类。
- 原理与推导：  
  `sort` 默认使用 `operator<` 进行升序排列。若要降序，可传入 `greater<Type>()`。  
  对于自定义类 `People`，重载 `operator<` 后，默认排序生效；但如果我们想在不同场景下按不同字段排序（年龄、体重），重载单一 `operator<` 会冲突。此时可定义比较器对象：
  ```cpp
  struct ByAge {
      bool operator()(const People& a, const People& b) const {
          return a.age < b.age;
      }
  };
  sort(v.begin(), v.end(), ByAge());
  ```
  STL 关联容器也以比较器维持有序性：`set<People, ByAge>`。
- 【解题步骤】
  1. 若只有一种全局默认排序方式，可在类内重载 `bool operator<(const T&) const`。
  2. 若需要多种排序，每种定义各自的结构体（函数对象），重载 `operator()`。
  3. 注意比较器语义必须满足严格弱序（strict weak ordering）：不能同时 `comp(a,b)` 与 `comp(b,a)` 为真；等价关系需传递。
  4. 使用标准预定义比较器：`less<T>`、`greater<T>` 位于 `<functional>`。
- 【易错点】
  - 重载 `operator<` 时，必须加 `const` 且参数使用 `const T&`，否则 STL 算法编译报错。
  - 自定义比较器若返回 `true` 表示 a 应排在 b 之前；降序时返回 `a > b`。
  - `set` 的模板参数顺序：`set<Key, Compare>`，默认为 `less<Key>`，若要传入自定义比较器，写法为 `set<People, ByAge>`，不需要加括号（因为类型即可，实例由容器内部构造）。
  - `priority_queue` 的比较器含义相反：默认大顶堆 `less` 实际上产生降序；自定义需额外注意。
  - 比较器对象一般无状态，适合短期匿名对象传递；注意生命周期（通常全程有效）。

| 方法 | 实现 | 适用场景 |
|------|------|----------|
| 重载 `operator<` | `bool operator<(const T& b) const { ... }` | 该类型天然只有一种排序规则 |
| 自定义函数对象 | `struct Cmp { bool operator()(const T&, const T&) const; };` | 多种排序规则并存，或排序标准库类型 |
| 使用标准库对象 | `greater<T>()` | 内建类型逆序 |

**例题**：
**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P42-43] `People` 类含 `age` 和 `weight`，实现按年龄升序排序的两种方式，并说明优劣。
**思路**：方法一重载 `<`，方法二独立比较器。
**解答**：
方法一（重载）：
```cpp
class People {
public:
    int age, weight;
    bool operator<(const People& b) const { return age < b.age; }
};
sort(vec.begin(), vec.end());
```
方法二（函数对象）：
```cpp
struct CmpAge {
    bool operator()(const People& a, const People& b) const {
        return a.age < b.age;
    }
};
sort(vec.begin(), vec.end(), CmpAge());
```
**技巧**：方法一只允许一种全局排序；若后续需按体重排序，需改用函数对象。推荐使用独立比较器，灵活且符合开闭原则。

#### std::function类型擦除机制 【必背】

**【定义】**
`std::function` 是一个通用多态函数包装器，定义在 `<functional>` 头文件中。它可以存储、复制、调用任何满足指定签名的可调用目标：函数、函数对象、lambda 表达式、成员函数指针等。其实现依赖类型擦除（Type Erasure）技术。

**【原理与推导】**
类型擦除的核心思想是将具体类型的信息隐藏在一个统一的接口后面。`std::function<Ret(Args...)>` 内部通常会持有：
- 一个基类（如 `callable_base`），定义了虚函数 `invoke` 用于调用目标；
- 针对每种具体可调用对象类型，派生出一个模板子类，实现 `invoke`；
- 通过一个基类指针（或类似机制）指向实际的派生对象，在调用时进行动态分发。

这样，`std::function` 只需要知道返回值类型和参数列表，不需要知道被包装对象的静态类型，从而可以在运行时统一处理不同类型的可调用实体。

使用示例：
```cpp
#include <functional>
std::function<string()> f;
f = readFromScreen;        // 函数指针
f = ReadFromFile();        // 函数对象
string result = f();       // 调用
```

**【解题步骤】**
1. 确定可调用实体的签名：返回类型 `Ret` 和参数类型 `Args...`。
2. 声明 `std::function<Ret(Args...)>` 对象。
3. 通过赋值或构造函数将可调用对象存入其中。
4. 通过 `operator()` 触发调用。

**【易错点】**

| 易混概念 | 模板参数 | std::function |
|---------|---------|---------------|
| 确定时机 | 编译期 | 运行期（通过类型擦除） |
| 运行开销 | 无额外开销（编译期内联优化） | 有虚函数调用或间接调用开销 |
| 可统一种类的类型 | 同一模板实例化只能处理确定类型，但可通过模板参数泛化 | 可以接受任何匹配签名的可调用对象 |

- 错误：试图用 `std::function` 完全替代模板以获得灵活性，忽略了性能损失。
- 内存占用：`std::function` 可能使用堆分配来存储较大的可调用对象（若大于内部缓冲区，通常一个指针大小以上）。

**例题**：
**题目**：解释为什么 `string(*readFunc2)(); readFunc2 = ReadFromFile();` 编译失败，而 `std::function<string()> readFunc; readFunc = ReadFromFile();` 成功。 [来源 L11-iostream&函数对象和智能指针-2026.pptx P53]
**思路**：函数指针类型必须严格匹配函数的静态类型，而 `ReadFromFile()` 是一个类的实例，不是函数。`std::function` 通过类型擦除抹去了具体类型差异。
**解答**：
函数指针 `string(*)(void)` 只能指向具有完全匹配签名的函数。`ReadFromFile` 是一个类类型，其对象不能隐式转换为函数指针。因此赋值非法。
`std::function<string()>` 实现了类型擦除，内部会构造一个适配器，使得 `ReadFromFile` 的 `operator()` 调用与 `string()` 接口匹配，因此赋值合法。
**技巧**：如需将不同的可调用对象统一保存（如存于容器、作为回调），使用 `std::function` 消除类型差异。

#### 模板、虚函数与std::function实现多态的对比 【必背】

**【定义】**
在 C++ 中实现多态有三种典型方式：
1. **虚函数多态**：通过基类指针或引用调用派生类重写的虚函数。
2. **模板（编译期多态）**：通过模板参数传递不同类型，在编译期间生成对应版本的函数。
3. **std::function 多态**：利用类型擦除，在运行期统一包装不同类型可调用对象。

**【原理与推导】**
- **虚函数**：基于继承和虚函数表（vtable），运行时通过查表决定调用哪个函数。需要预先定义基类接口，所有派生类必须继承自该基类。
- **模板**：在编译时根据传入参数的实际类型实例化函数或类，不同实例可调用对应类型的成员函数，无需共同基类。实现的是静态多态（鸭子类型）。
- **std::function**：通过类型擦除将调用逻辑封装在内部，向外暴露统一的函数签名接口。在运行时通过间接调用实现多态，无需使用者关心具体类型。

三者对比如下：

| 特性 | 虚函数 | 模板 | std::function |
|------|--------|------|---------------|
| 多态时机 | 运行期 | 编译期 | 运行期（类型擦除） |
| 是否需要公共基类 | 是 | 否（鸭子类型） | 否（只需签名匹配） |
| 运行时开销 | 一次虚函数查表 | 无（内联优化） | 间接调用，可能涉及堆分配 |
| 灵活性 | 受限于继承体系 | 非常高，但类型需在编译期确定 | 运行时灵活组合不同类型的可调用实体 |
| 代码膨胀 | 小 | 易致代码膨胀（大量实例化） | 较小 |

**【解题步骤】**
在需要回调或多态行为的场景，根据需求选择：
- 如果多态类型的数量在编译期就已固定，且对性能要求极高，使用模板。
- 如果多态类型需要运行时动态改变（如通过配置文件动态加载处理模块），且可以承担一定开销，使用 `std::function` 或虚函数。
- 如果已经有完善的类层次结构，且需要扩展性强，使用虚函数。

**【易错点】**
- 误以为 `std::function` 是模板，运行时零开销。实际上它通常会引入动态内存分配和间接调用。
- 模板在调用不同函数对象时，如果函数对象类型不同，会实例化出多份代码，可能导致可执行文件体积增大。
- 用 `std::function` 作为函数参数可能导致不必要的类型擦除开销，当不需要运行时多态时，应优先使用模板参数。

**例题**：（自编）  
**题目**：分别使用模板和虚函数实现一个打印容器所有元素的函数，并简述两种方式的优缺点。  
**思路**：模板版本利用 `begin()`/`end()` 的鸭子类型，无需继承；虚函数版本需要定义一个抽象的 `Printable` 基类。  
**解答**：  
模板版本：
```cpp
template <typename Container>
void print(const Container& c) {
    for (const auto& elem : c) cout << elem << ' ';
}
```
虚函数版本：
```cpp
class Printable {
public:
    virtual void print(ostream& os) const = 0;
    virtual ~Printable() {}
};
class IntVec : public Printable {
    vector<int> data;
public:
    void print(ostream& os) const override {
        for (int v : data) os << v << ' ';
    }
};
```
优缺点：模板版本零开销，但类型必须在编译期确定，且每个具体容器类型会生成一份函数，可能导致代码膨胀。虚函数版本可在运行时通过基类指针操作不同派生类对象，但引入了虚函数调用开销和继承体系约束。  
**技巧**：优先使用模板实现静态多态以获得高性能；当需要运行时动态切换行为或将不同类型存入同一容器时，使用 `std::function` 或虚函数。

### 五、智能指针与引用计数回顾

#### shared_ptr 引用计数原理与循环引用 【高频】

【定义】  
`std::shared_ptr` 是 C++11 引入的智能指针，位于 `<memory>` 头文件。它通过**引用计数**实现共享所有权：多个 `shared_ptr` 可以指向同一个对象，内部维护一个控制块记录引用计数，当计数归零时自动 `delete` 所指对象。

原理与推导  
每个 `shared_ptr` 内部存储一个指向所管理对象的裸指针和一个指向控制块的指针。控制块包含引用计数、弱引用计数及删除器等信息。  
- 构造函数 `shared_ptr<T>(new T(...))` 创建新控制块，引用计数初始化为 1。  
- 拷贝构造/拷贝赋值时，两个 `shared_ptr` 共享同一个控制块，引用计数加 1。  
- 移动构造/移动赋值时，所有权转移，引用计数不变。  
- 析构或 `reset()` 时引用计数减 1；若计数变为 0，则调用 `delete` 销毁对象并释放控制块。

**循环引用问题**：若两个对象互相持有对方的 `shared_ptr`，则引用计数永远不为 0，导致内存泄漏。  
例如：类 `Parent` 持有 `shared_ptr<Child>`，类 `Child` 持有 `shared_ptr<Parent>`，形成环状结构，离开作用域后两者均不会析构。

【解题步骤】  
分析引用计数的考题通常按以下步骤：  
1. 标明每个 `shared_ptr` 变量的生命周期（作用域）。  
2. 追踪构造、拷贝、赋值、移动、`reset()` 等操作对引用计数的影响。  
3. 判断对象销毁的时机（引用计数何时变为 0）。  
4. 若存在双向引用，识别循环引用并判断是否泄漏，应用 `weak_ptr` 解决。

【易错点】  
❌ 使用同一裸指针初始化多个 `shared_ptr`  
```cpp
int* p = new int(5);
shared_ptr<int> sp1(p);
shared_ptr<int> sp2(p);   // 错误！产生两个独立控制块，导致重复 delete
```
✅ 应直接从 `new` 表达式初始化或使用 `make_shared`。  

❌ 使用 `shared_ptr` 管理动态数组  
```cpp
shared_ptr<int> sp(new int[10]); // 错误！默认删除器使用 delete，应使用 delete[]
```
✅ 建议用 `std::vector` 或自定义删除器。  

❌ 循环引用未使用 `weak_ptr` 导致泄漏。

【例题】  
**题目**：阅读以下代码，写出程序的输出。 [L11-iostream&函数对象和智能指针-2026.pptx P61]  
```cpp
#include <memory>
#include <iostream>
using namespace std;

int main() {
    shared_ptr<int> p1(new int(4));
    cout << p1.use_count() << ' ';
    {
        shared_ptr<int> p2 = p1;
        cout << p1.use_count() << ' ';
        cout << p2.use_count() << ' ';
    }
    cout << p1.use_count() << ' ';
}
```
**思路**：根据作用域分析引用计数的变化：`p2` 在内部块内拷贝自 `p1`，离开块后销毁，计数恢复。  
**解答**：  
1. `p1` 创建，引用计数 = 1，输出 `1 `。  
2. 进入内部块，`p2` 拷贝自 `p1`，计数变为 2，`p1.use_count()` 和 `p2.use_count()` 均输出 `2 `。  
3. 离开内部块，`p2` 析构，计数减为 1，输出 `1`。  
程序最终输出：`1 2 2 1`。  
**技巧**：`shared_ptr` 的拷贝构造和赋值均会使引用计数加 1；`use_count()` 返回当前控制块的共享指针数量。

#### 手写引用计数（SmartPtr）的拷贝赋值语义 【必背】

【定义】  
通过自定义模板类 `SmartPtr` 和辅助类 `U_Ptr` 实现引用计数智能指针，理解 `shared_ptr` 底层拷贝、赋值、析构的计数管理机制。

原理与推导  
- 辅助类 `U_Ptr<T>` 存放实际数据指针 `T* p` 和引用计数 `int count`，构造函数将 `count` 初始化为 1。`SmartPtr` 持有 `U_Ptr<T>* rp`。  
- **拷贝构造** `SmartPtr(const SmartPtr& sp)`：将 `rp` 指向与源对象相同的 `U_Ptr`，并使 `rp->count` 增加 1。  
- **拷贝赋值运算符** `operator=(const SmartPtr& rhs)`：  
  ++`rhs.rp->count`；  
  if `--rp->count == 0`，则 `delete rp`；  
  `rp = rhs.rp`。  
  该顺序保证自赋值安全：先增加右侧计数，再减少自身计数，即使 `this == &rhs` 也能正确工作。  
- **析构函数**：`--rp->count` 递减，若为 0 则 `delete rp`（`U_Ptr` 析构函数会 `delete p` 释放实际对象）。  
- **解引用运算符** `*` 和成员访问运算符 `->`：分别返回 `*(rp->p)` 和 `rp->p`。  
- **友元关系**：`SmartPtr` 需要访问 `U_Ptr` 的私有成员，故将 `SmartPtr<T>` 声明为 `U_Ptr` 的友元类。

【解题步骤】  
分析自己实现的智能指针时：  
1. 确定 `U_Ptr` 对象的创建时机（智能指针构造时）。  
2. 跟踪拷贝/赋值时引用计数的增减顺序（特别注意赋值运算符的先加后减）。  
3. 判断何时删除 `U_Ptr` 指针（即删除控制块）以及何时删除实际对象（`U_Ptr` 析构时）。  
4. 验证自赋值安全性和异常安全性。

【易错点】  
- 赋值运算符若先减后加会导致自赋值时对象被提前删除。  
- 忘记友元声明会导致 `SmartPtr` 无法访问 `U_Ptr` 的私有成员 `count` 和 `p`。  
- `operator*` 返回引用，`operator->` 返回指针。  
- 在实际对象析构后继续访问（如 `use_count()` 仅表示共享指针数量，不代表对象依然存活）。

【例题】  
**题目**：给定以下自定义智能指针代码，运行 `main` 函数，写出输出结果，并解释赋值 `ptr3 = ptr2` 时引用计数的变化。 [L11-iostream&函数对象和智能指针-2026.pptx P66-68]  
```cpp
template <typename T> class SmartPtr;
template <typename T>
class U_Ptr {
    friend class SmartPtr<T>;
    U_Ptr(T *ptr) : p(ptr), count(1) {}
    ~U_Ptr() { delete p; }
    int count;
    T *p;
};

template <typename T>
class SmartPtr {
    U_Ptr<T> *rp;
public:
    SmartPtr(T *ptr) : rp(new U_Ptr<T>(ptr)) {}
    SmartPtr(const SmartPtr<T> &sp) : rp(sp.rp) { ++rp->count; }
    SmartPtr& operator=(const SmartPtr<T>& rhs) {
        ++rhs.rp->count;
        if (--rp->count == 0) delete rp;
        rp = rhs.rp;
        return *this;
    }
    ~SmartPtr() { if (--rp->count == 0) delete rp; }
    T& operator*() { return *(rp->p); }
    T* operator->() { return rp->p; }
};

int main() {
    int *pi = new int(2);
    SmartPtr<int> ptr1(pi);
    SmartPtr<int> ptr2(ptr1);
    SmartPtr<int> ptr3(new int(3));
    ptr3 = ptr2;
    cout << *ptr1 << endl;
    *ptr1 = 20;
    cout << *ptr2 << endl;
    return 0;
}
```
**思路**：  
- `ptr1` 绑定 `int(2)`，计数 1。  
- `ptr2` 拷贝自 `ptr1`，两者指向同一 `U_Ptr`，计数变为 2。  
- `ptr3` 新建 `int(3)`，计数 1。  
- `ptr3 = ptr2`：先增加 `ptr2` 的计数（变为 3），再减少 `ptr3` 原先计数（变 0），销毁 `int(3)` 的控制块；然后 `ptr3.rp` 指向 `ptr2` 的控制块。最终三个 `SmartPtr` 共享 `int(2)`，计数 3。  
- 输出 `*ptr1` 为 2；`*ptr1 = 20` 修改共享对象；`*ptr2` 也输出 20。  
**解答**：程序输出：  
```
2
20
```
赋值 `ptr3 = ptr2` 时：`rhs`（即 `ptr2`）引用计数先由 2 增为 3；`ptr3` 原引用计数由 1 减为 0，触发 `delete rp`（从而 `delete int(3)`）；最后 `ptr3.rp` 指向 `ptr2` 的控制块。  
**技巧**：赋值运算符中先增 `rhs` 计数再减自身计数，是处理自赋值的经典写法，同时保证异常安全。

#### weak_ptr 与 shared_ptr 协作打破循环 【高频】

【定义】  
`std::weak_ptr` 是一种不控制对象生命周期的智能指针，指向 `shared_ptr` 管理的对象，但不增加引用计数。主要用于监视 `shared_ptr` 对象是否存在，以及解决 `shared_ptr` 循环引用造成的内存泄漏。

原理与推导  
- `weak_ptr` 必须从 `shared_ptr` 或另一个 `weak_ptr` 构造，指向由 `shared_ptr` 管理的内存。  
- 它不参与引用计数，因此不影响对象的销毁。  
- 使用 `lock()` 返回一个临时的 `shared_ptr`，若对象还存在则返回非空指针，否则返回 `nullptr`。  
- 成员函数 `use_count()` 返回对应 `shared_ptr` 的引用计数；`expired()` 返回对象是否已被销毁（即引用计数为 0）。  
- 典型应用场景：树结构中的父指针、观察者模式、缓存等需要反向引用又不希望阻止销毁的场景。

**打破循环引用**：当两个类互相持有对方的 `shared_ptr` 时，会发生循环引用。将其中一个改为 `weak_ptr` 后，该指针不增加引用计数，打破环状依赖，保证析构链顺利进行。

【解题步骤】  
处理循环引用类时：  
1. 画出对象间持有关系，确定哪些指针必须是 `shared_ptr`，哪些可用 `weak_ptr`。  
2. 修改循环中的一方为 `weak_ptr`，并通过 `lock()` 临时获取 `shared_ptr` 来访问对象。  
3. 确保在 `lock()` 返回的 `shared_ptr` 存活期间对象不被意外销毁。  
4. 必要时使用 `expired()` 检查对象是否有效，避免空悬访问。

【易错点】  
❌ 误用 `weak_ptr` 直接解引用：`weak_ptr` 不能直接使用 `*` 或 `->`，必须先 `lock()` 转换为 `shared_ptr`。  
❌ 循环引用判断错误：只有形成双向 `shared_ptr` 持有才会泄漏，单向持有不存在问题。  
❌ 访问 `weak_ptr` 时忘记检查 `expired()` 或 `lock()` 返回值是否为 `nullptr`。

【例题】  
**题目**：以下代码在 `test()` 中创建了两个相互引用的对象，请问输出什么？为什么？如何修改才能让对象正常析构？ [L11-iostream&函数对象和智能指针-2026.pptx P70-74]  
```cpp
#include <memory>
#include <iostream>
using namespace std;

class Child;
class Parent {
    shared_ptr<Child> child;
public:
    Parent()  { cout << "parent constructing" << endl; }
    ~Parent() { cout << "parent destructing" << endl; }
    void setChild(shared_ptr<Child> c) { child = c; }
};

class Child {
    shared_ptr<Parent> parent;   // 造成循环引用
public:
    Child()  { cout << "child constructing" << endl; }
    ~Child() { cout << "child destructing" << endl; }
    void setParent(shared_ptr<Parent> p) { parent = p; }
};

void test() {
    shared_ptr<Parent> p(new Parent());
    shared_ptr<Child> c(new Child());
    p->setChild(c);
    c->setParent(p);
}

int main() {
    test();
    return 0;
}
```
**思路**：`test` 中 `p` 和 `c` 均为 `shared_ptr`，互相持有对方后引用计数均为 2，离开 `test` 时各自 `reset` 使计数降为 1，均不为 0，导致两个对象的内存泄漏。  
**解答**：  
- 原程序输出：`parent constructing` 和 `child constructing`，没有析构输出。  
- 修改方法：将 `Child` 类中的 `shared_ptr<Parent>` 改为 `weak_ptr<Parent>`，并调整 `setParent` 保持传入 `shared_ptr` 转换为 `weak_ptr`，访问 `Parent` 时通过 `lock()` 获得临时 `shared_ptr`。  
修改后程序输出：  
```
parent constructing
child constructing
parent destructing
child destructing
```
**技巧**：设计双向关联时，选择“拥有权”较弱的方使用 `weak_ptr`；被 `weak_ptr` 指向的对象不会阻止其析构。

#### unique_ptr 独占所有权与移动语义 【重点】

【定义】  
`std::unique_ptr` 是一种独占所有权的智能指针，同一时刻只能有一个 `unique_ptr` 指向特定对象。它无法拷贝，只能移动，体现资源独占语义。离开作用域时自动 `delete` 所管理的对象。

原理与推导  
- 构造函数 `unique_ptr<T>(new T(...))` 获得所有权；C++14 起推荐使用 `std::make_unique<T>(args...)` 避免裸指针暴露。  
- `unique_ptr` 禁用了拷贝构造和拷贝赋值，仅允许移动构造和移动赋值。移动操作将内部指针的所有权转移给目标 `unique_ptr`，源指针变为空（`nullptr`）。  
- `release()` 放弃所有权，返回裸指针，并且 `unique_ptr` 自身变为空。调用者需自行 `delete` 返回的指针（或将其移交给另一个智能指针）。  
- `reset()` 销毁当前管理的对象并接管新指针（或置空）。  
- 由于没有引用计数开销，性能优于 `shared_ptr`，且语义更清晰。

【解题步骤】  
使用 `unique_ptr` 时需注意：  
1. 确定资源的所有权模型——是否只能有一个拥有者。  
2. 用 `std::move` 显式转移所有权。  
3. 若需要将所有权交给外部，使用 `release()` 并确保接收方负责释放。  
4. 不应直接使用`unique_ptr`的地址进行多次释放，也不应把`release()`返回的裸指针交付给多个 `unique_ptr`。  

【易错点】  
❌ 尝试拷贝 `unique_ptr`，导致编译错误。  
```cpp
unique_ptr<int> up1 = make_unique<int>(10);
unique_ptr<int> up2 = up1;   // 错误！拷贝构造被删除
```
❌ 移动后继续使用源 `unique_ptr`（源指针已为空）。  
❌ `release()` 后忘记释放裸指针，造成内存泄漏。  
```cpp
int* raw = up.release();   // 必须负责 delete raw
```
❌ 使用 `unique_ptr` 管理数组时，默认删除器调用 `delete` 而非 `delete[]`，应使用特化 `unique_ptr<T[]>`。

【例题】  
**题目**：写出以下程序的执行结果，并说明 `release()` 和 `move` 的作用。 [L11-iostream&函数对象和智能指针-2026.pptx P80]  
```cpp
#include <memory>
#include <iostream>
using namespace std;

int main() {
    auto up1 = make_unique<int>(20);
    // unique_ptr<int> up2 = up1; // 编译错误，不能复制
    unique_ptr<int> up2 = std::move(up1);  // 移动所有权
    int* p = up2.release();               // 放弃控制权
    cout << *p << endl;                   // 输出 20
    delete p;
    return 0;
}
```
**思路**：`up1` 管理 `int(20)`，通过 `move` 转移给 `up2`，`up1` 变为空；`up2.release()` 放弃所有权返回裸指针，`up2` 变空，最后手动 `delete` 释放。  
**解答**：  
程序输出 `20`。  
- `std::move(up1)` 将 `up1` 所管理的对象转移给 `up2`，`up1` 不再持有任何资源。  
- `release()` 释放 `up2` 对对象的所有权，返回底层裸指针，调用者需要负责释放（这里通过 `delete p` 完成）。  
**技巧**：`unique_ptr` 的移动语义确保资源所有权唯一且可传递；`release()` 常用于将所有权交给传统 API 或需要手动控制的场景，注意避免泄漏。

#### 函数模板的定义与编译期多态 【高频】
【定义】
函数模板是一种泛型编程机制，它使用泛型标记（如 `typename T`）定义一个函数的蓝图。编译器在编译期根据实际调用时传入的参数类型，自动生成对应类型的具体函数实例。这种在编译期确定具体调用代码的机制，被称为**静多态**或编译期多态。

原理与推导：
函数模板的核心原理是**编译期代码生成**。编译器遇到一个函数模板的定义时，并不会立即生成可执行代码。只有当该模板被调用（使用）时，编译器才根据调用的参数类型，生成一份针对该类型的特化版本，并进行编译。
```cpp
template<typename T>
T sum(T a, T b) { return a + b; }

int main() {
    int a = sum(1, 2);        // 编译器生成 int sum(int, int)
    double b = sum(1.0, 2.0); // 编译器生成 double sum(double, double)
    return 0;
}
```
**为什么模板的声明和定义通常必须放在头文件中？**
因为模板的实例化发生在编译期。如果模板的定义在 `a.cpp` 中，而调用在 `main.cpp` 中，编译 `a.cpp` 时，编译器看不到任何调用，因此不会生成任何实例代码。当链接器试图链接 `main.cpp` 中对 `sum<int>` 的调用时，就会因为找不到定义而报链接错误。因此，模板的完整定义必须在调用点对编译器可见，通常做法是直接放在头文件中。

【解题步骤】
1.  **识别类型依赖**：确定函数中哪些地方依赖于类型。
2.  **声明模板参数**：在函数前使用 `template<typename T>` 或 `template<class T>` 声明一个类型参数 `T`。
3.  **替换具体类型**：将原函数中所有需要泛化的具体类型（如 `int`, `double`）替换为模板参数 `T`。
4.  **注意约束**：当模板函数用于自定义类型时，必须确保该类型支持模板函数体内使用的所有操作符（如 `operator>`），否则编译报错。

【易错点】

| 易混概念 | 辨析 |
| :--- | :--- |
| **静多态（模板） vs 动多态（虚函数）** | **编译期 vs 运行期**：模板在编译时确定类型，生成代码，无运行时开销，但会导致代码膨胀。虚函数在运行时通过虚函数表动态查找，有间接调用开销，但更灵活。 |
| **模板的编译错误** | 通常会产生大量、难以阅读的错误信息。检查时通常只需要关注**第一个错误**。出现 `no match for ‘operator...’` 等提示，说明模板对类型参数施加了未被满足的操作约束。 |
| **分开编译** | 普通函数可以将声明放在头文件，定义放在源文件。**模板不能这样做**，必须在头文件中提供完整定义，否则会链接失败。 |

**例题**：
**题目**：分析以下代码为何会编译错误，并给出解决方案 [L9-多态与模板-2026.pptx P41]。
```cpp
class MyInt {
public:
    int data;
    MyInt(int val): data(val) {};
};

int main() {
    MyInt arr_c[] = {3, 2, 4, 1, 5};
    sort(arr_c, 5); // 假设 sort 是能对数组排序的函数模板
    output(arr_c, 5);
    return 0;
}
```
**思路**：错误在于函数模板 `sort` 对 `MyInt` 类型的实例化。需要分析 `sort` 模板内部对类型参数 `T` 施加了哪些操作要求，并检查 `MyInt` 类是否支持这些操作。
**解答**：
1.  **分析错误**：编译器报错 `error: no match for ‘operator>’`。这表明在 `sort` 函数模板内部，代码试图使用 `operator>` 来比较两个 `MyInt` 对象的大小，但 `MyInt` 类并未重载该操作符。
2.  **解决方案**：为 `MyInt` 类重载 `operator>`。同时，为了能在 `output` 中输出，通常也需要重载 `operator<<`。
```cpp
class MyInt {
public:
    int data;
    MyInt(int val): data(val) {};
    // 重载 > 操作符，用于比较
    bool operator>(const MyInt& b){
        return data > b.data;
    }
    // 重载 << 操作符，用于输出
    friend std::ostream& operator<<(std::ostream& out, const MyInt& obj){
        out << obj.data;
        return out;
    }
};
```
**技巧**：使用模板编写通用代码时，要清楚模板对类型参数施加了哪些“隐含接口”或“操作约束”。为自定义类型使用模板时，必须确保其满足这些约束。

#### 类模板的定义、声明与实例化 【重点】
【定义】
类模板（Class Template）是将类的成员类型参数化的机制。它允许创建一个类的蓝图，其中的某些类型（如数据成员的类型、成员函数的参数或返回值类型）在定义时不被指定，而是在使用该类创建对象时作为模板参数传入。

原理与推导：
与函数模板类似，类模板的实例化也发生在编译期。当使用 `类模板名<具体类型>` 声明一个对象时，编译器会生成一个真实的、所有类型都已确定的类，然后为这个类创建对象。
```cpp
template <typename T> class A {
    T data; // 成员类型 T 将在实例化时确定
public:
    A(T _data): data(_data) {}
    void print() { cout << data << endl; }
};

int main() {
    A<int> a(1);    // 实例化 A<int> 类，并创建对象 a
    A<double> b(1.2); // 实例化 A<double> 类，并创建对象 b
    return 0;
}
```
类外定义成员函数时，需要加上模板声明和作用域，且成员函数名前的类名必须带模板参数： `A<T>`。
```cpp
template<typename T>
void A<T>::print() { cout << data << endl; }
```

【易错点】
类模板的类名不代表一个具体的类型，`A` 不是类型，`A<int>` 才是类型。

**例题**：（自编）  
**题目**：定义一个模板类 `Stack<T>`，提供 `push(T)` 和 `T pop()` 方法，内部使用 `vector<T>` 存储。在类外定义成员函数，并写 `main` 函数创建 `Stack<int>` 对象，推入 `1,2,3` 并打印弹出元素。  
**思路**：类模板的基本语法、类外定义、作用域 `Stack<T>`。  
**解答**：
```cpp
#include <vector>
#include <iostream>
using namespace std;

template<typename T>
class Stack {
    vector<T> elems;
public:
    void push(const T& elem);
    T pop();
};

template<typename T>
void Stack<T>::push(const T& elem) {
    elems.push_back(elem);
}

template<typename T>
T Stack<T>::pop() {
    if (elems.empty()) throw runtime_error("empty stack");
    T elem = elems.back();
    elems.pop_back();
    return elem;
}

int main() {
    Stack<int> s;
    s.push(1);
    s.push(2);
    s.push(3);
    while (/* not empty */)  // 略
        cout << s.pop() << endl; // 输出 3 2 1
}
```
**技巧**：类外定义成员函数时，必须带上 `template<typename T>`，且类名后跟 `<T>`，两者缺一不可。

#### 非类型模板参数与编译期确定的值 【重点】
【定义】
模板参数不一定是类型，也可以是**编译期确定的值**，这被称为非类型模板参数（Non-type template parameter）。常见的有整数、枚举、指针或引用。非类型参数使得可以基于值对模板进行参数化，如指定数组的固定大小。

原理与推导：
非类型参数的值必须在编译时就能确定。因此可以传入整数常量、`const` 变量或字面量，但不能传入运行时才能确定的普通变量。
```cpp
template<typename T, unsigned size> // size 是非类型参数
class array {
    T elems[size]; // 使用非类型参数定义数组成员大小
public:
    ... 
};

int main(){
    int n = 5;
    // array<char, n> array0; // 错误！n 是变量，编译期不可知
    const int m = 5;
    array<char, m> array1; // 正确，m 是编译时常量
    array<char, 5> array2; // 正确
    return 0;
}
```

【易错点】

| 对比项 | 类型参数 | 非类型参数 |
| :--- | :--- | :--- |
| **作用** | 抽象掉“类型” | 抽象掉“值” |
| **声明** | `typename T` 或 `class T` | 具体类型名，如 `int N` |
| **实参** | 任何类型，如 `int`, `MyClass` | 编译期常量，如 `10`, `const int` |
| **典型用途** | 泛型容器 `std::vector<T>` | 固定大小数组 `std::array<T, N>` |

**例题**：（自编）  
**题目**：定义一个模板类 `FixedArray<T, int N>`，内部包含 `T data[N]`，并提供下标访问。在 `main` 中实例化 `FixedArray<double, 10>` 并演示基本用法。  
**思路**：非类型参数 `N` 必须是常量，编译期确定数组大小。  
**解答**：
```cpp
#include <iostream>
using namespace std;

template<typename T, int N>
class FixedArray {
    T data[N];
public:
    T& operator[](int i) { return data[i]; }
    const T& operator[](int i) const { return data[i]; }
};

int main() {
    FixedArray<double, 5> arr;
    for (int i = 0; i < 5; ++i) arr[i] = i * 1.1;
    // 输出 arr[2] 等
}
```
**技巧**：非类型参数必须是编译期常量，若尝试传入变量则编译失败。C++17 的 `auto` 非类型参数允许更灵活的推导。

#### 模板与多态的对比：静多态与动多态 【必背】
【定义】
模板和虚函数是实现C++多态的两种核心方式。
- **静多态（Static Polymorphism）**：通过模板实现，在**编译期**通过代码生成解决函数调用，体现为函数重载和模板实例化。
- **动多态（Dynamic Polymorphism）**：通过继承和虚函数实现，在**运行期**通过虚函数表查找解决函数调用。

原理与推导：

| 对比维度 | 静多态（模板） | 动多态（虚函数+继承） |
| :--- | :--- | :--- |
| **实现机制** | 编译时生成具体类型代码（实例化） | 运行时通过虚函数表（vtable）动态绑定 |
| **时机** | **编译期** | **运行期** |
| **性能** | **高效**，无间接调用开销 | 存在一次额外的指针间接寻址开销 |
| **类型约束** | **非侵入式**，类型只需满足语法要求（鸭子类型） | **侵入式**，类型必须显式继承自特定基类 |
| **代码体积** | 不同参数实例化会生成多份代码，可能导致**代码膨胀** | 一份函数代码，通过基类指针/引用调用，体积小 |
| **灵活性** | 不同实例生成的类型相互独立，编译后不可变 | 可在运行时通过基类指针灵活替换不同派生类对象 |

【易错点】
模板是利用“看起来一样”的代码接口（鸭子类型）在编译期实现多态，而虚函数是利用“确实一样”的基类接口在运行期实现多态。

**例题**：（自编）  
**题目**：分别使用模板和虚函数实现一个打印容器所有元素的函数，并简述两种方式的优缺点。  
**思路**：模板版本利用 `begin()`/`end()` 的鸭子类型，无需继承；虚函数版本需要定义一个抽象的 `Printable` 基类。  
**解答**：  
模板版本：
```cpp
template <typename Container>
void print(const Container& c) {
    for (const auto& elem : c) cout << elem << ' ';
}
```
虚函数版本：
```cpp
class Printable {
public:
    virtual void print(ostream& os) const = 0;
    virtual ~Printable() {}
};
class IntVec : public Printable {
    vector<int> data;
public:
    void print(ostream& os) const override {
        for (int v : data) os << v << ' ';
    }
};
```
优缺点：模板版本零开销，但类型必须在编译期确定，且每个具体容器类型会生成一份函数，可能导致代码膨胀。虚函数版本可在运行时通过基类指针操作不同派生类对象，但引入了虚函数调用开销和继承体系约束。  
**技巧**：优先使用模板实现静态多态以获得高性能；当需要运行时动态切换行为或将不同类型存入同一容器时，使用 `std::function` 或虚函数。

#### 成员函数模板（自学）
【定义】
成员函数模板是指在普通类或类模板中，将其成员函数定义为模板。这允许该成员函数拥有自己独立的、与类模板参数不同的模板参数，从而提供了更高维度的泛化能力。
```cpp
// 普通类中的成员函数模板
class normal_class {
public:
    int value;
    // 类内定义
    template<typename T> void set(T const& v) {
        value = int(v);
    }
    // 类内声明
    template<typename T> T get();
};
// 类外定义
template<typename T>
T normal_class::get() {
    return T(value);
}
```

原理与推导：
当一个类模板的成员函数自身也是模板时，类的模板参数和函数的模板参数是**两层独立的参数**。在类外定义时，必须分别提供两套 `template<...>` 声明，并且顺序固定：先类模板参数，再成员函数模板参数。
```cpp
template<typename T0> class A { 
    T0 value; 
public:
    template<typename T1> void set(T1 const& v); // 声明
};
// 类外定义
template<typename T0>  // 类模板参数
template<typename T1>  // 成员函数模板参数
void A<T0>::set(T1 const& v) { 
    value = T0(v); 
}
```
绝不能错误地合并成 `template<typename T0, typename T1>`，这会被理解为同时接受两个参数的单一模板。

【解题步骤】
1.  **分析层级**：区分模板参数属于类还是成员函数。
2.  **书写顺序**：在类外定义时，先写类模板的 `template<...>`，再写成员函数模板的 `template<...>`。
3.  **类型推导**：成员函数模板的参数能从调用实参推导，也可以显式指定无法推导的部分。
    ```cpp
    A<int> a;
    a.set(5);          // 自动推导T1为int
    double t = a.get<double>(); // 手动指定T1为double
    ```

【易错点】

| 错误写法 | 正确写法 | 解析 |
| :--- | :--- | :--- |
| `template<typename T0, typename T1>` <br> `T1 A<T0>::get(){...}` | `template<typename T0> template<typename T1>` <br> `T1 A<T0>::get(){...}` | 前者会被编译器理解为一个带有两个参数的单一模板，语法错误。必须分别声明。 |
| 将非类型参数传递为变量 | `int n=5; array<int, n> a;` | `const int n=5;` 或 `array<int, 5> a;` | 非类型模板参数要求编译期常量。 |

### 六、模板特化与重载 （自学）

#### 函数模板特化 【重点】

##### 函数模板的特化与重载辨析

**【定义】**
- **函数模板特化**：为特定类型提供不同于通用模板的特殊实现。
- **非模板重载函数**：一个普通函数，与函数模板同名，但参数列表不同或更具体。
- 三者调用优先级：**非模板重载函数 > 全特化函数模板 > 通用函数模板**。编译器总是优先选择最“具体”的那个版本。

**【原理与推导】**
重载决议时，编译器将候选函数分为两组：普通函数和从模板实例化出的函数。如果存在一个完全匹配的普通函数，它会被优先选择。函数模板特化不参与首次重载决议，只有当函数模板被选中后，编译器才会检查是否有适用于此类型的最佳特化版本。这一复杂规则是【易错点】的根源。

**例题**：（来源 [L10-模板与stl初步-2026.pptx P82]）  
**题目**：阅读以下代码，说明 `f(p)` 调用了哪个函数，并解释原因。  
```cpp
template<class T> void f(T) { cout << "full template" << endl; }
template<class T> void f(T*) { cout << "full template -> overload template" << endl; }
template<> void f(char*) { cout << "overload template -> specialized" << endl; }

int main() {
    char *p;
    f(p);
    return 0;
}
```
**思路**：首先进行重载决议，选出最佳基础模板：`f(T*)` 比 `f(T)` 更特化，因此选中 `f(T*)`。然后检查该基础模板是否拥有针对 `char*` 的特化版本——发现有 `f(char*)`，故最终调用特化版本。  
**解答**：程序输出 `overload template -> specialized`。  
**技巧**：函数模板匹配顺序：第一步选择最匹配的基础模板，第二步在该基础模板上查找最优特化。特化不会跨基础模板生效。

#### 类模板特化

**【定义】**
类模板特化是指为某个类模板的特定类型参数组合，提供一份专门的实现。它可以完全偏离通用模板的定义，拥有完全不同的成员、方法和数据布局，以满足特定类型的需求或优化。

**【原理与推导】**
当编译器实例化一个类模板时，会首先查找是否存在与模板参数完全匹配的全部或部分特化版本。
- **全特化**：当所有模板参数都被具体的类型固定时。
- **偏特化（部分特化）**：当只有部分参数被固定，或对参数施加了额外约束（如类型变为指针、引用）时。

使用特化的典型场景包括：
1. **类型萃取（Type Traits）**：在 STL 中广泛使用，用于在编译期查询或修改类型属性。
2. **为特定类型提供更优实现**：如 `vector<bool>` 使用位压缩存储，显著节省空间。
3. **处理指针类型的特殊逻辑**：对 `T*` 进行偏特化，实现深拷贝等与值类型不同的行为。

**【解题步骤】**
1. **声明通用模板（Primary Template）**：
   ```cpp
   // 主模板
   template <class T>
   struct MyClass {
       static void info() { std::cout << "Generic" << std::endl; }
   };
   ```
2. **全特化**：为 `T = int` 实现一个完全不同的版本。
   ```cpp
   template <>
   struct MyClass<int> {
       static void info() { std::cout << "Specialized for int" << std::endl; }
   };
   ```
3. **偏特化**：为 `T` 是指针类型提供通用实现。
   ```cpp
   template <class T>
   struct MyClass<T*> {
       static void info() { std::cout << "Partial specialized for pointer" << std::endl; }
   };
   ```
4. **使用**：`MyClass<double>::info();` 输出 "Generic"；`MyClass<int>::info();` 输出 "Specialized for int"；`MyClass<double*>::info();` 输出 "Partial specialized for pointer"。

**【易错点】**
- **函数模板不能偏特化**：这是函数模板与类模板的关键区别。函数模板只能**重载**来实现类似偏特化的效果。
- **特化版本的声明与定义**
  | 易混概念 | 错误理解 | 正确做法 |
  | :--- | :--- | :--- |
  | **特化时机** | 在使用特化版本**之后**才声明或定义特化。 | 特化版本必须在所有使用该特化的源代码文件中提前声明（通常是放在头文件中，紧跟主模板之后），否则编译器可能会实例化主模板，导致 ODR 违例或非预期行为。 |
  | **偏特化函数** | `template<typename T> void func<T*>(T* p){}` | 这是**非法**的函数模板偏特化。正确做法是**函数模板重载**：<br>`template<typename T> void func(T* p){}` |

**【例题】**
**题目**：阅读代码，判断输出结果。
  ```cpp
  #include <iostream>
  template <class T> class Container {
  public:
      void info() { std::cout << "Generic" << std::endl; }
  };
  // (1) 偏特化：T是指针
  template <class T> class Container<T*> {
  public:
      void info() { std::cout << "Pointer" << std::endl; }
  };
  // (2) 全特化：T = int*
  template <> class Container<int*> {
  public:
      void info() { std::cout << "int*" << std::endl; }
  };

  int main() {
      Container<char> c1;         c1.info();
      Container<char*> c2;        c2.info();
      Container<int*> c3;         c3.info();
      return 0;
  }
  ```

**思路**：考查类模板特化选择的优先级：全特化 > 偏特化 > 主模板。编译器会为每个实例化找出最匹配的特化版本。

**解答**：
  - `Container<char>` 无法匹配 (1) 和 (2)，选择**主模板**。输出：`Generic`
  - `Container<char*>` 能够匹配偏特化 (1) （T = char），并且没有针对 `char*` 的全特化，选择**偏特化**。输出：`Pointer`
  - `Container<int*>` 能够匹配偏特化 (1) （T = int）和 全特化 (2)，**全特化**是更佳匹配。输出：`int*`

**技巧**：匹配规则可以概括为“普通版本是备胎，偏特化是升级版，全特化是定制版”。存在定制版时，编译器不会考虑升级版或备胎。

### 本章小结
本章知识点紧密围绕“类型安全、资源管理、多态机制”三条主线展开：I/O流和`stringstream`通过继承与模板实现格式化与类型安全转换；函数对象与智能指针是资源管理与可调用对象封装的典范；虚函数、纯虚函数、向上/向下转型构建了运行期多态的完整框架；模板则提供了编译期多态的泛型支持，两者互为补充。

**核心结论**：
1. 流对象不可复制，重载`operator<<`必须返回`ostream&`。
2. 继承体系中，通过基类指针`delete`对象时，基类析构必须为虚函数，否则资源泄漏。
3. `shared_ptr`控制块增减遵循“拷贝加1，析构/重置减1”；`unique_ptr`独占所有权，禁止复制。
4. 构造函数/析构函数中不会触发虚函数动态绑定；纯虚析构必须提供实现。
5. 模板实例化发生在编译期，必须头文件全定义；函数模板无偏特化，靠重载替代。

**必背公式**：
- `ostream& ostream::operator<<(ostream& (*fn)(ostream&));`  // 操纵算子原型
- `SmartPtr& operator=(const SmartPtr& rhs) { ++rhs.rp->count; if (--rp->count == 0) delete rp; rp = rhs.rp; return *this; }`  // 安全赋值
- `class Derived : public Base { void func() override final {} };`  // 最终覆盖
- `template<typename T0> template<typename T1> void A<T0>::func(T1 v) {}`  // 成员函数模板类外定义

【记忆】构建多态与泛型程序的两大支柱：运行时看虚函数表与动态转型，编译时靠模板实例化与类型推导。
