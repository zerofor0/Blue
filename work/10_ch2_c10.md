### 六、核心机制深度回顾

#### 纯虚函数、抽象类与纯虚析构函数 【重点】

##### 1. 纯虚函数与抽象类

###### 【定义】
- **纯虚函数**：在基类中声明但不实现的虚函数，其声明形式为在函数原型后加上 `= 0`（例如：`virtual void func() = 0;`）。纯虚函数可以拥有函数体，但必须在类外定义。
- **抽象类**：包含至少一个纯虚函数的类。抽象类不能被实例化，只能作为基类使用。

###### 原理与推导
- **设计目的**：
    - **接口规范**：强制所有派生类提供统一接口的具体实现。
    - **代码隔离**：基类只定义“做什么”，不定义“怎么做”，实现接口与实现的分离。
    - **多态基础**：通过基类指针或引用调用纯虚函数，在运行时根据实际对象类型动态绑定到派生类的实现。
- **限制与规则**：
    1.  若派生类未覆盖基类中的**所有**纯虚函数（纯虚析构函数除外），则该派生类仍为抽象类，不能被实例化。
    2.  纯虚析构函数比较特殊，即使派生类未显式实现，编译器也会自动合成析构函数，因此不影响派生类的实例化（前提是该派生类覆盖了其他所有纯虚函数）。

###### 【解题步骤】（判断与使用抽象类）
1.  **判断是否是抽象类**：查看类中是否声明了纯虚函数（`= 0`）。
2.  **判断派生类能否实例化**：检查派生类是否覆盖了基类中除析构函数外的所有纯虚函数。
3.  **使用抽象类**：永远通过指针或引用来操作抽象类对象，利用多态特性。

###### 【易错点】

| 易混淆概念 | 分析与区别 |
| :--- | :--- |
| **纯虚函数与空函数体** | `virtual void f() = 0;` 是纯虚函数，类为抽象类。`virtual void f() {}` 是普通虚函数，类可以实例化。 |
| **纯虚函数定义** | 纯虚函数可以有实现（在类外定义），这种定义常为派生类提供默认行为。但即使有定义，类依然是抽象类。 |
| **纯虚析构函数特殊性** | 必须提供函数体；即使派生类未显式实现，也不会导致派生类成为抽象类（编译器会自动合成）。 |

###### 例题
**例题**：
**题目**：以下关于抽象类的代码中，哪些语句会导致编译错误？为什么？
```cpp
class Base {
public:
    virtual void func() = 0;
};
void Base::func() { cout << "Base func" << endl; } // 定义纯虚函数体

class Derive1 : public Base {};

class Derive2 : public Base {
public:
    void func() override { Base::func(); }
};

int main() {
    // 1
    Base b;
    // 2
    Derive1 d1;
    // 3
    Derive2 d2;
    d2.func();
    return 0;
}
```
**思路**：本题考察抽象类与纯虚函数的规则。关键在于检查哪些类是抽象类，以及抽象类能否实例化。
**解答**：
1.  语句 `Base b;` 编译错误。因为 `Base` 类包含纯虚函数，是抽象类，不能实例化对象。
2.  语句 `Derive1 d1;` 编译错误。`Derive1` 继承了 `Base`，但未覆盖纯虚函数 `func()`，因此它仍然是抽象类，不能实例化。
3.  语句 `Derive2 d2; d2.func();` 正确。`Derive2` 覆盖了纯虚函数 `func()`，不再是抽象类，可以实例化并调用函数。程序会输出 `Base func`。
**技巧**：判断一个类能否实例化的唯一标准：它自身或它的所有基类中，是否存在**未被覆盖**的非析构纯虚函数。纯虚析构函数不影响实例化。

##### 2. 纯虚析构函数

###### 【定义】
- **纯虚析构函数**：声明为纯虚的析构函数（`virtual ~ClassName() = 0;`）。其**必须提供函数体**（通常在类外实现）。
- **目的**：仅作为一种技术手段，将基类变为抽象类，防止直接实例化基类对象。

###### 原理与推导
- **为何需要函数体**：派生类析构时，会逐级调用基类的析构函数。如果纯虚析构函数没有函数体，链接阶段会出错。
- **对派生类的影响**：
    - **一般纯虚函数**：若派生类不覆盖，派生类仍是抽象类。
    - **纯虚析构函数**：若派生类不**显式**实现，编译器会自动合成一个析构函数。因此，只要派生类覆盖了基类的其他所有纯虚函数，该派生类即可被实例化。
- **常见设计模式**：当一个类既需要作为抽象基类（定义接口），又没有其他适合设为纯虚的函数时，将析构函数设为纯虚。

###### 【解题步骤】（分析与使用纯虚析构）
1.  **判断意图**：看到纯虚析构函数，应马上意识到其首要目的是让基类变为抽象类。
2.  **检查实现**：确保纯虚析构函数在类外有对应的函数体实现。
3.  **分析派生类**：判断一个继承自此类（其中只有析构是纯虚的，没有其他纯虚函数）的派生类是否可直接实例化。答案是可以，因为编译器会为它合成析构函数。

###### 【易错点】
- **易错1：忘记实现函数体。** 只写 `virtual ~Base() = 0;` 而不在类外写 `Base::~Base() {}`，会导致任何继承它的派生类在实例化或析构时产生链接错误。
- **易错2：混淆对派生类的影响。** 误以为继承自仅有纯虚析构函数基类的派生类也是抽象类。**正确理解**：该派生类是具体类，可以实例化。

###### 例题
**例题**：[L9-多态与模板-2026.pptx P12]
**题目**：阅读下面程序，写出运行结果。
```cpp
#include <iostream>
using namespace std;

class Base{ 
public: 
	virtual ~Base()=0; 
};
Base::~Base() {cout<<"Base destroyed"<<endl;}

class Derive1: public Base {};
class Derive2: public Base {
public: 
	virtual ~Derive2() {cout<<"Derive2 destroyed"<<endl;} 
};

int main()
{
    Base* p1 = new Derive1;
    Base* p2 = new Derive2;
    delete p1;
    cout << “------” << endl;
    delete p2;
    return 0;
}
```
**思路**：本题考察虚析构函数的调用顺序以及纯虚析构函数的行为。当`delete`基类指针时，会根据指针所指对象的实际类型，从最派生的类开始，逐级向上调用析构函数。
**解答**：
1.  `delete p1;` ：`p1`指向`Derive1`对象。`Derive1`无自定义析构，执行编译器合成的析构函数，随后调用基类`Base`的析构函数，输出 `Base destroyed`。
2.  `delete p2;` ：`p2`指向`Derive2`对象。先执行`Derive2`的自定义析构，输出 `Derive2 destroyed`，然后调用基类`Base`的析构函数，输出 `Base destroyed`。
运行结果：
```
Base destroyed
------
Derive2 destroyed
Base destroyed
```
**技巧**：（基类）虚析构函数保证了在多态场景下，析构过程是完整、正确的链式调用。务必为多态基类声明虚析构函数。

#### 向上向下类型转换与虚函数表 【高频】

##### 1. 向上类型转换
###### 【定义】
将派生类对象的指针或引用，转换为基类类型的指针或引用。这是C++中一种安全的、可由编译器隐式完成的转换。

##### 2. 向下类型转换
###### 【定义】
将基类类型的指针或引用，转换为派生类类型的指针或引用。此操作有风险，必须**显式**进行。

###### 原理与推导
- **为何需要**：面向对象编程中，常使用基类指针集合来管理多种派生类对象（向上转型），这保留了共性，但丢失了特性。当需要调用某个派生类特有的接口时，必须将其还原回特定的派生类类型（向下转型）。
- **实现机制**：C++提供 `dynamic_cast` 和 `static_cast` 两种方式进行显式向下转型。

| 特性 | `dynamic_cast` | `static_cast` |
| :--- | :--- | :--- |
| **安全机制** | **运行时类型检查**。通过虚函数表（RTTI）判断实际对象是否可以被安全转换为目标类型。 | **编译时静态检查**。仅检查继承关系，不保证运行时指向对象的实际类型。 |
| **性能** | **较慢**。需要额外的运行时开销。 | **很快**。无运行时开销，与C风格类型转换类似。 |
| **失败行为** | **指针转换**失败返回 `nullptr`。<br>**引用转换**失败抛出 `std::bad_cast` 异常。 | **不安全**。转换“成功”但指针可能指向非法内存，导致未定义行为。 |
| **前置条件** | 源类型 `T1` 必须是**多态类型**（声明或继承了至少一个虚函数），否则编译错误。 | 源类型 `T1` 和目标类型 `T2` 之间必须有继承关系或已定义的转换路径，否则编译错误。 |
| **使用场景** | **安全的向下类型转换**。当你不确定基类指针是否真的指向某个特定派生类对象时。 | **确信安全的向下类型转换**。你已经明确知道基类指针所指对象的实际类型，出于性能考虑使用。 |

###### 【解题步骤】（安全的向下类型转换）
1.  **优先使用 `dynamic_cast`**：在需要向下转换时，首选 `dynamic_cast` 确保安全。
2.  **检查结果**：
    - 对指针形式 `T2* p = dynamic_cast<T2*>(ptr_base);`，必须检查 `p` 是否为 `nullptr`。
    - 对引用形式 `T2& r = dynamic_cast<T2&>(ref_base);`，必须使用 `try-catch` 块捕获异常。
3.  **使用 `static_cast`**：仅在性能敏感且通过逻辑完全能确保转换安全性的代码段使用。使用时应添加注释说明理由。

###### 【易错点】
- **对象切片**：向上或向下类型转换应作用于**指针或引用**，而非对象本身。对对象进行转换会导致“对象切片”，丢失派生类信息。
- **忘记检查 `dynamic_cast` 返回值**：对指针版本直接解引用而不检查 `nullptr`，是导致程序崩溃的常见原因。
- **对非多态类型使用 `dynamic_cast`**：会导致编译错误。一定要确保被转换的源类型有虚函数。

###### 例题
**例题**：[L9-多态与模板-2026.pptx P23]
**题目**：阅读下面程序，写出运行结果。
```cpp
#include <iostream>
using namespace std;

class Pet { public:  virtual ~Pet() {} };
class Dog : public Pet { public: void run() { cout << "dog run" << endl; } };
class Bird : public Pet { public: void fly() { cout << "bird fly" << endl; } };

void action(Pet* p) {
    auto d = dynamic_cast<Dog*>(p);
    auto b = dynamic_cast<Bird*>(p);
    if (d)
        d->run();
    else if(b)
        b->fly();
}
int main() {
    Pet* p[2];
    p[0] = new Dog;
    p[1] = new Bird;
    for (int i = 0; i < 2; ++i) {
        action(p[i]);
    }
    return 0;
}
```
**思路**：此题是`dynamic_cast`的典型应用。基类指针`Pet* p`指向不同的派生类对象。在`action`函数内部，通过`dynamic_cast`尝试将其转换为具体的派生类指针，并根据转换是否成功（是否为空）来执行派生类特有的行为。
**解答**：
1.  `action(p[0])`：`p[0]`实际指向`Dog`对象。`dynamic_cast<Dog*>(p)`成功，返回有效指针，执行 `d->run()`，输出 `dog run`。
2.  `action(p[1])`：`p[1]`实际指向`Bird`对象。`dynamic_cast<Dog*>(p)`失败，返回`nullptr`；接着 `dynamic_cast<Bird*>(p)`成功，返回有效指针，执行 `b->fly()`，输出 `bird fly`。
运行结果：
```
dog run
bird fly
```
**技巧**：利用 `dynamic_cast` 加 `if` 判断的模式，可以根据基类指针所指对象的真实类型实现分派，这是一种安全且清晰的“根据类型执行不同操作”的方法。
