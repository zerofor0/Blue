### 一、输入输出流（iostream）回顾
#### 流类层次结构与基类抽象 【重点】
【定义】  
`iostream` 类库以抽象基类为核心，组织为如图所示的层次结构：

- `ios_base` — 最底层抽象，管理格式状态等通用功能。
- `ios` — 继承自 `ios_base`，提供指向 `streambuf` 的指针。
- `istream` — 输入流抽象基类，重载 `>>`，提供 `get`、`peek`、`ignore` 等输入操作。
- `ostream` — 输出流抽象基类，重载 `<<`，提供 `put`、`flush` 等输出操作。
- `iostream` — 多重继承 `istream` 和 `ostream`，同时支持输入、输出。
- `ifstream` / `ofstream` / `fstream` — 文件流，分别继承 `istream` / `ostream` / `iostream`。
- `istringstream` / `ostringstream` / `stringstream` — 字符串流，同理继承。

`cin` 是 `istream` 的全局对象（标准输入），`cout` / `cerr` / `clog` 是 `ostream` 的全局对象（标准输出/错误/日志）。

原理与推导  
通过面向对象的多态设计，所有流都提供了统一的接口，例如 `operator>>` 和 `operator<<`。  
- `ostream` 内部为基本类型（`int`, `double`, `const char*` 等）重载了一系列 `operator<<`，调用系统函数完成实际 I/O。  
- 用户自定义类型只需定义 `ostream& operator<<(ostream&, const T&)` 全局函数或友元，即可无缝接入流接口。  
- 链式调用依赖 `<<` 左结合性和返回 `ostream&` 的引用，形如：  

```cpp
// 简化的 ostream 内部实现示意
class ostream {
public:
    ostream& operator<<(char c) {
        printf("%c", c);
        return *this;
    }
    ostream& operator<<(const char* str) {
        printf("%s", str);
        return *this;
    }
} cout;
```

执行 `cout << "hello" << ' ' << "world";` 时，各次调用均返回 `cout` 自身引用，实现连续输出。

【解题步骤】  
1. 识别类层次：根据需求（文件/字符串/标准 I/O）选择对应的流类。  
2. 编写输出运算符重载：若 `Test` 类需要 `cout << obj`，定义 `std::ostream& operator<<(std::ostream& out, const Test& src)`。  
3. 确保返回 `ostream&` 以支持链式调用。  
4. 若需同时支持输入，对称定义 `istream& operator>>(istream& in, Test& dst)`。

【易错点】  
- 将 `operator<<` 定义为成员函数会使得左操作数必须为自定义类，破坏链式调用习惯；应定义为友元或全局函数。  
- 返回类型必须是 `ostream&` 而非 `ostream`，否则不能连续 `<<`。  
- 忘记 `const` 修饰自定义类型参数可能导致右值实参无法绑定。  
- 混淆 `istream` 和 `ostream` 的继承关系，误用 `cin` 的类型。

#### 流操纵算子原理与自定义实现 【高频】
【定义】  
流操纵算子（Stream Manipulator）是一类可改变流格式状态的特殊对象或函数，它们通过 `operator<<`（或 `operator>>`）被插入到流中，例如 `endl`、`fixed`、`setprecision`、`setw` 等。

原理与推导  
实现流操纵算子有两种典型模式：

1. **无参数函数式算子（C++标准定义）**：`endl` 实际是函数 `ostream& endl(ostream& os)`，它输出 `\n` 并调用 `os.flush()`。为了支持 `cout << endl`，`ostream` 重载了函数指针版本的 `operator<<`：

```cpp
// endl 函数定义
ostream& endl(ostream& os) {
    os.put('\n');
    os.flush();
    return os;
}

// ostream 中对函数指针的重载
ostream& operator<<(ostream& (*fn)(ostream&)) {
    return (*fn)(*this);
}
```

这样 `cout << endl` 就等价于 `endl(cout)`。

2. **有参数对象式算子（编译器自定义）**：某些操纵算子（如 `setprecision`）需要参数，通常借助辅助类实现。

```cpp
// 辅助类
class setprecision {
    int precision;
public:
    setprecision(int p) : precision(p) {}
    friend class ostream;   // 允许 ostream 访问私有成员
};

// ostream 中对应重载
class ostream {
    int precision;
public:
    ostream& operator<<(const setprecision& m) {
        precision = m.precision;
        return *this;
    }
};
```

使用 `cout << setprecision(2);` 时，`setprecision(2)` 构造临时对象，`operator<<` 提取其参数并修改流的精度状态。

【解题步骤】  
1. 记住常用的预定义操纵算子：`endl`（换行并刷缓冲区）、`fixed` / `scientific` / `defaultfloat`（浮点格式）、`setprecision(n)`、`setw(n)`、`setfill(ch)`、`oct` / `hex` / `dec` 等。  
2. 理解两层重载：`endl` 型算子依赖函数指针重载；`setprecision` 型算子依赖自定义类对象及对应 `operator<<`。  
3. 编写自定义无参操纵算子时，模仿 `endl` 的函数签名和对应 `operator<<`；编写有参算子时，定义辅助类并为其提供 `operator<<` 重载。  
4. 注意 `iomanip` 头文件提供 `setprecision`、`setw` 等，使用时需 `#include <iomanip>`。

【易错点】  
- 混淆 `endl` 和 `'\n'`：`endl` 额外调用 `flush()`，用于需要立即输出的场景（如交互式程序）；仅想换行而不强制刷新时用 `'\n'` 效率更高。  
- 认为所有操纵算子的实现方式都由标准严格定义：实际上 `setprecision` 的实现因编译器而异（辅助类或其它技术），但效果一致。  
- 在 `<<` 链中忘记操纵算子返回流的引用，导致链式调用中断（实现者易错）。  
- 自定义操纵算子时未在正确命名空间（`std`）中声明，可能找不到匹配的重载。

**例题**：  
**题目**：（来源 [L11-iostream&函数对象和智能指针-2026.pptx P10]）  
```cpp
cout << setprecision(2) << 1.05 << endl;
```
请问输出结果是什么？

**思路**：`setprecision(2)` 设置输出精度为2位有效数字（默认浮点格式下）；`1.05` 用默认格式输出，取2位精度应四舍五入为 `1.1`；`endl` 换行并刷新。

**解答**：  
```
1.1
```
（`1.05` 保留两位精度后四舍五入得 `1.1`，小数点后的零不显示。）

**技巧**：`setprecision` 影响的是有效数字位数，若与 `fixed` 结合则变成小数位数，需注意区分。

#### 格式化输出控制 【重点】
【定义】  
利用 `<iomanip>` 中操纵算子（也可直接调用 `ios_base` 的 `precision()` 等成员函数）控制输出格式，包括浮点格式、进制、域宽、填充字符等。

原理与推导  
流内部维护了一系列格式状态标志，`operator<<` 会根据这些标志转换数据。操纵算子通过修改状态标志达到格式化效果。常用操纵算子及其作用：

| 操纵算子 | 作用 |
|--------|------|
| `fixed` / `scientific` / `defaultfloat` | 定点/科学计数法/恢复默认浮点格式 |
| `setprecision(n)` | 设置浮点输出精度（配合 `fixed` 为小数位数，否则为有效位数） |
| `setw(n)` | 设置下一次输出的最小域宽 |
| `setfill(ch)` | 配合 `setw`，指定填充字符（默认空格） |
| `oct` / `hex` / `dec` | 八进制/十六进制/十进制输出整数 |
| `left` / `right` / `internal` | 域内对齐方式 |
| `endl` / `flush` / `ws` | 换行并冲缓冲/只冲缓冲/提取空白符 |

所有操作都通过插入到 `cout` 等流中完成。需注意 `setw` 只对紧随其后的一次输出有效，而 `fixed`、`setprecision` 等持续有效直到被修改。

【解题步骤】  
1. 确定需要控制的格式类型（浮点、进制、对齐）。  
2. 选择对应操纵算子，将其插入到输出链中，位置放在待输出量之前。  
3. 需要域宽时，`setw` 紧挨在目标数据前，且注意其“一次性”特性。  
4. 结合 `fixed` 与 `setprecision` 控制小数位数是常见组合。  
5. 结束特殊格式后常用 `defaultfloat`、`dec` 等恢复默认状态，避免影响后续输出。

【易错点】  
- 误以为 `setprecision` 总是设置小数位数；仅在启用 `fixed` 或 `scientific` 时才是小数位数，默认是有效数字。  
- `setw` 仅影响下一次输出，之后域宽恢复默认（0）；忘写 `setw` 导致域宽不对齐。  
- 进制转换后忘记用 `dec` 恢复十进制，使后续整数都以八进制或十六进制展示，发现 bug 困难。  
- 填充字符需要用 `setfill` 配合 `setw`，单独 `setfill` 不起显示效果。  

**例题**：  
**题目**：（来源 [L11-iostream&函数对象和智能指针-2026.pptx P9]）  
请写出以下代码的输出结果：  
```cpp

cout << scientific << 2018.0 << " " << 0.0001 << endl;
cout << defaultfloat;
cout << setprecision(2) << 3.1415926 << endl;

cout << dec;
cout << setw(3) << setfill('*') << 5 << endl;
```

**思路**：逐行分析格式状态与数据：第一行定点格式，默认精度6位小数；第二行科学计数法；第三行 `defaultfloat` 恢复默认浮点，接着 `setprecision(2)` 设置有效数字位数2；第四行八进制和十六进制输出12；第五行十进制恢复后，`setw(3)` 配合 `setfill('*')` 产生带星号填充的宽度3的整数5。

```
2.018000e+03 1.000000e-04

14 c

```

#### 不能复制的对象语义 【高频】
【定义】  
`ostream`（以及所有流对象）不可复制，只支持移动。其复制构造函数为 `delete`，提供了移动构造函数。

```cpp
ostream(const ostream&) = delete;
ostream(ostream&& x);
```

`cout` 是一个全局唯一的 `ostream` 对象，不能被复制。

原理与推导  
为什么流对象不能复制？  
1. **资源唯一性**：每个流对象关联着一个唯一的底层缓冲区（`streambuf`），如果允许多个流对象共享同一个缓冲区，会造成输出混乱或竞态，复制语义难以管理。  
2. **避免开销**：流的内部状态（格式标志、异常状态、绑定流等）复杂，拷贝成本高且语义不清晰。  
3. **面向对象思想**：一个全局 `cout` 对象封装了标准输出，符合“单一职责”和“全局接入点”的设计。

重载 `operator<<` 时，必须返回 `ostream&` 引用，从而：  
- 避免产生无效的拷贝请求。  
- 支持链式调用，且始终操作同一个流对象。

实际上，如果 `operator<<` 返回值类型不是引用，会触发已删除拷贝构造函数的编译错误。

【解题步骤】  
1. 编写流输出运算符时，第一参数为 `std::ostream&`，返回类型为 `std::ostream&`。  
2. 返回语句通常是 `return out;`，在函数内调用 `out << a << b;` 后返回引用。  
3. 理解 `cout` 单例性质，避免试图创建另一个 `ostream` 对象（除非通过已提供的其他全局对象如 `cerr`）。  
4. 当需要将 `ostream` 作为参数传递时，一律使用引用（或指针）。

【易错点】  
- 误将 `operator<<` 返回值写为 `ostream` 导致编译错误（尝试复制）。  
- 混淆 `cin`、`cout` 是类还是对象：它们是对象，`cin` 是 `istream` 对象，`cout` 是 `ostream` 对象。  
- 移动语义允许通过 `std::move` 改变流的所有权，但一般极少使用，多数情况下只是将 `ifstream` 等作为局部变量传递（如 `std::ifstream ifs("file")`），函数返回时自动关闭。  
- 对 `cout` 使用 `auto x = std::move(cout);` 会使得 `cout` 处于未指定状态，后续不能再安全使用，属于危险操作。

**例题**：

### 二、字符串流（stringstream）回顾
#### 缓冲区操作与状态位管理 【高频】
【定义】  
`std::stringstream`（定义在 `<sstream>`）内部维护一个 `std::string` 缓冲区，同时提供输入（读取）和输出（写入）功能。它继承自 `iostream`，因而同时具备 `istream` 和 `ostream` 的方法。

通过 `operator<<` 向缓冲区写入数据，通过 `operator>>` 从缓冲区读取数据，就像操作标准输入输出一样，但数据全部驻留在内存中。

原理与推导  
写入时数据被追加到内部缓冲区末尾（类似 `operator+=`），读取时按照流位置前进，从缓冲区头开始逐个读取解析。  
- 状态位（`eofbit`, `failbit`, `badbit`）同样适用：当读取超出缓冲区末尾时，将置 `eofbit`；格式解析失败则置 `failbit`。  
- 可以使用 `clear()` 重置状态位，用 `str("")` 清空缓冲区，用 `peek()` 查看下一个字符而不提取。

常用模式：  

```cpp
stringstream ss;
ss << 10 << " " << 20;   // 写入 "10 20"
int a, b;
ss >> a >> b;            // 读取成功, a=10, b=20
```

【解题步骤】  
1. 包含 `<sstream>`，构造 `stringstream` 对象（可传入初始字符串）。  
2. 将数据格式化为字符串：用 `<<` 连续写入，完全如 `cout`。  
3. 从字符串解析数据：用 `>>` 读取，注意类型匹配，失败时处理状态位。  
4. 若需要复用流，先 `clear()` 再 `str("")` 重置。  
5. 多次读取时留意流内位置，必要时使用 `seekg(0)` 回到开头。

【易错点】  
- 读取失败后流状态变为 `fail`，必须调用 `clear()` 才能继续操作，否则后续读写被忽略。  
- 缓冲区和流位置是独立的概念：写入后读取位置在缓冲区头，无需手动定位；但若先部分读取再追加写入，可能需要 `clear()` + `seekp(0, ios::end)` 保证一致性。  
- `operator>>` 以空白字符为分隔，读取字符串数据时不保留内部的空格。  
- 忘记包含 `<sstream>` 头文件导致编译错误。

#### 类型转换与模板函数结合 【必背】
【定义】  
`stringstream` 经常作为类型转换的工具：字符串到数值（`"42"` → `int`），数值到字符串等。结合模板函数可写出通用的转换代码。

原理与推导  
输入输出流对基本类型已经重载了 `>>` 和 `<<`，因此：

```cpp
// 数值 -> 字符串
template<typename T>
std::string toString(const T& val) {
    std::ostringstream oss;
    oss << val;
    return oss.str();
}

// 字符串 -> 数值
template<typename T>
T fromString(const std::string& str) {
    T val;
    std::istringstream iss(str);
    iss >> val;
    return val;
}
```

利用 `ostringstream`（只输出）和 `istringstream`（只输入）分别实现转换，避免了 `sprintf`/`scanf` 的类型不安全和格式指定问题。

【解题步骤】  
1. 数值→字符串：`ostringstream oss; oss << value; return oss.str();`。  
2. 字符串→数值：`istringstream iss(str); T val; iss >> val;`，注意检查 `iss.fail()` 来处理格式错误。  
3. 封装成模板函数，提高代码复用性。  
4. 结合 `std::stoi`、`std::stod` 等现代 C++ 函数也可以完成转换，但 `stringstream` 更通用且适合自定义类型。

【易错点】  
- 使用 `stringstream` 同时读写时，若未将流位置重置容易读取到错误数据。最好分别用 `ostringstream` 和 `istringstream` 各司其职。  
- 转换失败时不检查状态位，导致未初始化的 `val` 被当做有效值。记得添加 `if(iss.fail())` 处理。  
- 若字符串含有空格，`>>` 只会读取第一个单词，需要用 `std::getline(iss, val)` 读取完整一行。  
- 与 `std::to_string` / `std::stoi` 混淆：前者直接转换数值到字符串（便捷），后者将字符串转为数值并可能抛出异常；`stringstream` 方式更灵活（可处理自定义类型，适用于流格式化）。

**例题**：  
**题目**：（来源 [L11-iostream&函数对象和智能指针-2026.pptx P23]）  
```cpp
#include <sstream>
using namespace std;

int main() {
    stringstream ss;
    ss << "10";
    ss << "0 200";

    int a, b;
    ss >> a >> b;   // a = ?, b = ?
    return 0;
}
```
问：变量 `a` 和 `b` 的值分别是多少？

**思路**：`ss` 初始为空，写入 `"10"` 后缓冲区为 `"10"`，再写入 `"0 200"`，缓冲区变为 `"100 200"`。随后提取两个 `int`，第一个读到 `100`，第二个读到 `200`。

**解答**：  
`a` = 100，`b` = 200。

**技巧**：`stringstream` 的写入是连续追加的，多个 `<<` 操作直接拼接字符串，无自动分隔。本题中 `"10"` 和 `"0"` 直接邻接变成 `"100"`，因此解析时作为一个完整数字。使用时应留意这种拼接效应。

#### 头文件与继承关系 【重点】
【定义】  
字符串流相关的头文件为 `<sstream>`，其中定义了三个类模板（并用 `char` 实例化出我们常用的类型）：

| 类 | 继承关系 | 用途 |
|----|----------|------|
| `std::istringstream` | 继承 `std::istream` | 输入字符串流，只读 |
| `std::ostringstream` | 继承 `std::ostream` | 输出字符串流，只写 |
| `std::stringstream` | 继承 `std::iostream` | 输入输出字符串流，可读写 |

`iostream` 又由 `istream` 和 `ostream` 多重继承而来，因此 `stringstream` 接口涵盖全部输入输出操作。

原理与推导  
继承结构使得 `stringstream` 对象可以作为 `istream&` 或 `ostream&` 参数传递给任何接受这些基类的函数。例如，`getline(ifs, str)` 同样可写为 `getline(std::cin, str)` 或 `getline(ss, str)`，因为函数接受 `istream&`。

与文件流类似，字符串流通过重写基类虚函数，操作内部 `stringbuf`（派生自 `streambuf`）实现内存中的 I/O。

【解题步骤】  
1. 只在需要内存字符串格式化时引入 `<sstream>`。  
2. 根据只需要读、写、还是都要选用 `istringstream` / `ostringstream` / `stringstream`。  
3. 将字符串流对象当作 `istream` 或 `ostream` 使用即可，无需关注底层。  
4. 若要将自定义类型也支持 `<<`/`>>` 与字符串流配合，只需保证该自定义类型已有对应的重载运算符。

【易错点】  
- 只包含 `<iostream>` 无法使用 `stringstream`，必须包含 `<sstream>`。  
- 混淆文件流头文件 `<fstream>` 与字符串流头文件 `<sstream>`。  
- 用 `stringstream` 却忘记其继承自 `iostream`，导致误认为不能使用某些 `istream`/`ostream` 的成员函数（实际全部可用）。

【例题：待补充】

### 三、文件流（fstream）回顾
#### 文件打开模式与操作生命周期 【重点】
【定义】  
文件流操作声明于 `<fstream>`，主要类及其继承关系：

- `std::ifstream` — 输入文件流，继承 `istream`
- `std::ofstream` — 输出文件流，继承 `ostream`
- `std::fstream` — 输入输出文件流，继承 `iostream`

打开模式用标志位组合（`std::ios_base::openmode`），常用模式：

| 模式 | 含义 |
|------|------|
| `in` | 读模式（`ifstream` 默认） |
| `out` | 写模式（`ofstream` 默认） |
| `binary` | 二进制模式 |
| `ate` | 打开时定位到文件末尾 |
| `app` | 追加模式（每次写入都定位到文件末尾） |
| `trunc` | 打开前清空文件（`out` 默认） |

构造时可直接指定文件名和模式，如：  
```cpp
ifstream ifs("input.txt");                     // 默认文本读
ifstream ifs("data.bin", ifstream::binary);   // 二进制读
ofstream ofs("output.txt", ofstream::out | ofstream::app); // 追加写
```

原理与推导  
文件流对象在构造时打开文件，内部调用操作系统的文件打开函数，建立文件缓冲区。  
- 对象析构时自动关闭文件（RAII）。  
- 也可显式调用 `open()` / `close()`；`close()` 后可以再次打开另一个文件。  
- 判断文件是否成功打开：可以检查流对象本身（它重载了 `operator bool`），如 `if(!ifs) { /* 处理错误 */ }`。

【解题步骤】  
1. 包含 `<fstream>`，声明文件流对象并传递文件名/模式。  
2. 使用前检查流对象是否有效（`if(ifs)` 或 `ifs.is_open()`）。  
3. 通过 `<<` / `>>` 或 `getline` 进行读写，操作方式与 `cin` / `cout` 一致。  
4. 操作完毕，自动析构或显式 `close()` 关闭文件，保证数据写回磁盘。  
5. 需要改变关联文件时，先 `close()` 再 `open("other.txt")`。

【易错点】  
- 忘记检查文件打开状态，导致读取数据为空或写文件无输出。  
- 写入文件后未关闭或刷新缓冲区，程序异常终止时文件内容可能丢失（除非显式 `flush()` 或 `close()`）。  
- 如果在文本模式下读二进制文件，可能因 `0x0D` `0x0A` 转换导致数据错误，务必使用 `binary` 模式。  
- 对象被复制会使多个对象关联同一文件句柄？流对象禁止复制，但移动会造成悬空状态。通常只需作为局部变量即可。

【例题：待补充】

#### 输入读取策略与流状态检查 【高频】
【定义】  
从文件流读取数据的常用手段：`>>`（格式化提取）、`getline`（读一行）、`get` / `peek` / `ignore`（细粒度控制）。  
流状态标志（`good()`, `eof()`, `fail()`, `bad()`）用于判断读取是否成功。

原理与推导  
`ifstream` 继承了 `istream` 的全部输入成员。重要成员函数：

- `operator bool()` — 对流是否处于可读状态进行隐式判断，通常用作 `while(ifs)`
- `ifs >> ws` — 提取前导空白，但不读取有效字符（`ws` 是操纵算子）
- `ifs.peek()` — 返回下一个字符而不提取，`EOF` 表示结束
- `ifs.get()` — 读取一个字符，失败返回 `EOF`
- `ifs.ignore(n, delim)` — 跳过最多 `n` 个字符或直到遇见 `delim`

典型读取策略示例（P18 演示）：

```cpp
ifstream ifs("input.txt");
while (ifs) {                // 只要流状态为 good
    ifs >> ws;               // 跳过空白
    int c = ifs.peek();      // 偷看下一个字符
    if (c == EOF) break;
    if (isdigit(c)) {        // 是数字？
        int n;
        ifs >> n;
        cout << "Read a number: " << n << endl;
    } else {
        string str;
        ifs >> str;
        cout << "Read a word: " << str << endl;
    }
}
```

这段代码展示了安全读取的方式：先去除空白，预览，再根据字符类型选择不同读取策略。

【解题步骤】  
1. 打开文件后进行循环读取：`while (ifs)` 或 `while (getline(ifs, line))`。  
2. 利用 `peek()` 预判数据类型，避免用错读取方式。  
3. 每次格式化输入 (`>>`) 后检查 `ifs.fail()`，若类型不匹配会导致 `failbit`，需中断或跳过。  
4. 使用 `ifs.clear()` 重置错误状态，`ifs.ignore(n, '\n')` 跳过错误行，继续处理。  
5. 读取二进制文件时用 `read(char*, streamsize)`，并结合 `gcount()` 获取实际读取字节数。

【易错点】  
- 习惯 `while (!ifs.eof())` 控制循环：文件末尾标志只有在尝试读取越过末尾后才设置，会导致最后一次读取被处理两次或产生错误。正确方式是 `while (ifs >> var)` 或 `while (getline(ifs, line))`。  
- `peek()` 返回 `int` 类型，不可直接赋给 `char`，否则 `EOF` 值可能无法区分（比较前先转为 `int`）。  
- 混合使用 `getline` 和 `>>` 时，`>>` 会遗留换行符在缓冲区，导致后续 `getline` 读取空行；需要用 `ifs.ignore()` 消耗掉残留分隔符。  
- 忘记 `ifs >> ws` 去除空白导致 `peek()` 看到空格而误判。

【例题：待补充】

- 【定义】`stringstream` 是一种同时具备输入和输出功能的流，其内部维护一个字符缓冲区（buffer）用于暂存数据。通过 `<<` 将数据写入缓冲区，通过 `>>` 从缓冲区读取数据，读写均基于同一缓冲区。
- 原理与推导：`stringstream` 继承自 `iostream`，因此支持双向操作。缓冲区内容不会因读取而消失，读取只是移动内部的“读指针”。  
  重要操作：  
  • `ss.str()` 返回整个缓冲区字符串，无论已读部分还是未读部分。  
  • `ss.str("")` 将缓冲区内容清空。  
  • `ss.clear()` 清除流的状态标志（eofbit、failbit、badbit 等），但不清空缓冲区。  
  状态位：`goodbit`（无错误）、`eofbit`（到达尾部）、`failbit`（格式错误，如类型不匹配）、`badbit`（严重错误）。一旦进入 `failbit` 或 `eofbit`，后续读写可能被忽略，必须先用 `clear()` 恢复。
- 【解题步骤】
  1. 写入多个数据：`ss<<a<<b<<c;`，数据间自动以空格分隔。
  2. 按写入顺序读取：`ss>>v1>>v2;`，类型必须匹配，否则设置 failbit。
  3. 若需要复用 `stringstream` 对象（如循环内），必须先 `ss.str("")` 清空缓冲区，再 `ss.clear()` 复位状态。
  4. 想要整体查看缓冲区：`cout << ss.str();`。
- 【易错点】
  - `ss.clear()` 和 `ss.str("")` 容易混淆：`clear()` 只复位状态，不清内容；`str("")` 只清内容，不自动清状态。两者往往需要同时调用。
  - 读取后缓冲区并未缩短，`str()` 依旧返回全部写入的内容，这可能导致“读空了为什么还能输出”的困惑。
  - 多次 `>>` 提取时，若某次提取失败（如格式不对），`failbit` 被置位，后续所有提取都将跳过，需 `clear()` 唤醒。
  - 文件流（fstream）也有类似状态位机制，行为一致。

| 易混对象 | 功能 | 是否清空缓冲区 | 是否清除状态位 |
|----------|------|----------------|----------------|
| `ss.str("")` | 将缓冲区内容设为空 | 是 | 否 |
| `ss.clear()` | 清除流状态标志 | 否 | 是 |

**例题**：
**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P24] 分析程序输出。
```cpp
stringstream ss;
ss << "100 200";
cout << ss.str() << endl;       // ①
int a, b;
ss >> a;
cout << ss.str() << endl;       // ②
ss >> b;
// a 和 b 的值？
```
**思路**：`str()` 返回缓冲区完整内容，不受读指针影响。
**解答**：
- ① 输出 `"100 200"`；
- ② 输出 `"100 200"`；
- `a = 100`，`b = 200`。
**技巧**：`stringstream` 的 `str()` 始终返回全部缓冲区内容，只有重新写入才会改变。

- 【定义】利用 `stringstream` 可编写通用的类型转换模板函数 `convert<OutType, InType>`，实现任意两种支持流插入/提取操作的类型相互转换。
- 原理与推导：  
  ```cpp
  template<class OutType, class InType>
  OutType convert(InType val) {
      static stringstream ss;
      ss.str("");     // 清空缓冲区
      ss.clear();     // 清除状态位
      ss << val;      // 将输入写入缓冲区
      OutType res;
      ss >> res;      // 从缓冲区提取输出类型
      return res;
  }
  ```
  使用 `static stringstream` 避免反复构造；每次调用前先清空缓冲区并清除状态位，防止上次失败的状态影响本次转换。  
  该函数不仅能替代 `std::to_string` 和 `std::stoi`，还可用于自定义类型（需实现 `operator<<` 和 `operator>>`）。
- 【解题步骤】
  1. 包含 `<sstream>`。
  2. 定义双类型模板，内部静态 `stringstream`。
  3. 每次调用前顺序调用 `ss.str("")` 和 `ss.clear()`。
  4. `ss << val;` 写入，`ss >> res;` 读取并返回。
- 【易错点】
  - 若不清除状态位，上次失败留下的 `failbit` 会导致本次 `>>` 直接失败，`res` 可能为未定义值。
  - 静态对象在多线程环境下不安全，单线程作业或学习场景可接受。
  - 自定义类型转换时必须提供对应的 `operator<<` 与 `operator>>`，否则编译失败。

| 对比项 | `std::to_string` / `std::stoi` | `convert` 模板 |
|--------|-------------------------------|----------------|
| 支持类型 | 仅限内建算术类型 | 任何重载了 `<<` / `>>` 的类型 |
| 灵活性 | 若干重载函数 | 一个函数模板，自动推导 |

**例题**：
**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P25] 实现一个通用转换函数，使 `convert<string>(123)` 返回 `"123"`，`convert<int>(“456”)` 返回 `456`。
**思路**：模板 + stringstream 居中转换。
**解答**：
```cpp
template<class OutType, class InType>
OutType convert(InType val) {
    static stringstream ss;
    ss.str("");
    ss.clear();
    ss << val;
    OutType res;
    ss >> res;
    return res;
}
```
**技巧**：`str("")` 和 `clear()` 配对使用是复用流对象的经典模式，任何流（文件、字符串）均适用。

- 【定义】`stringstream` 定义在 `<sstream>` 中，**继承自 `iostream`**，因此同时拥有输入能力（`>>`）和输出能力（`<<`）。这与 `ifstream`（只读，继承 `istream`）和 `ofstream`（只写，继承 `ostream`）形成对比。
- 原理与推导：流类层次简化关系为：  
  `ios_base` → `ios` → `istream` / `ostream` → `iostream` → `stringstream`（以及 `fstream`）。  
  `ifstream` 继承自 `istream`，不能写入；`ofstream` 继承自 `ostream`，不能读取。  
  因此，`stringstream` 既能格式化写入字符串，也能从字符串中解析数据；`fstream` 若想同时读写文件，需要指定 `ios::in | ios::out` 模式。
- 【解题步骤】
  1. 需要读写字符串缓冲区：`#include <sstream>`，使用 `stringstream`。
  2. 仅读文件：`#include <fstream>`，使用 `ifstream`。
  3. 仅写文件：`ofstream`。
  4. 辨别错误说法：常见“`ifstream` 是 `iostream` 子类，所以能读写”为典型错误。
- 【易错点】
  - 头文件与类名混淆：`stringstream` 在 `<sstream>`，`ifstream` / `ofstream` / `fstream` 在 `<fstream>`。
  - 继承关系：`iostream` 不是 `istream` 的直接子类，而是同时继承 `istream` 和 `ostream`（多重继承）。`stringstream` 和 `fstream` 继承自 `iostream`，具有双重能力；`ifstream` 仅继承 `istream`，无法输出。
  - 考题中常出现“`ifstream` 是 `iostream` 子类”，属于陷阱。

| 流类 | 头文件 | 继承自 | 输入 | 输出 |
|------|--------|--------|------|------|
| `stringstream` | `<sstream>` | `iostream` | ✓ | ✓ |
| `ifstream` | `<fstream>` | `istream` | ✓ | ✗ |
| `ofstream` | `<fstream>` | `ostream` | ✗ | ✓ |
| `fstream` | `<fstream>` | `iostream` | ✓ | ✓ |

**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P27] 下列说法错误的是：  
A. `scanf` 有可能写入非法内存  
B. `stringstream` 是继承自 `iostream` 的子类  
C. `stringstream` 既可以作为输入流，也可以作为输出流  
D. `ifstream` 是 `iostream` 的子类，能同时从文件中读入数据和写出数据
**思路**：D 项中 `ifstream` 继承自 `istream`，不能写入；其余说法正确。

**技巧**：抓住“只有 `iostream` 的派生类才同时拥有输入输出能力”这一原则即可快速排除干扰项。

### 四、函数对象（Function Object）回顾

#### 函数指针与函数对象的概念统一 【高频】
- 【定义】C++ 中，函数调用既可以通过函数指针实现，也可以通过**函数对象**（重载了 `operator()` 的对象）实现。STL 算法（如 `sort`）通常将其比较准则的类型参数化为模板类型 `Compare`，可以接受任何可调用实体，包括函数指针和函数对象。这种统一依托于模板的“鸭子类型”：只要 `comp(a,b)` 语法有效，即可编译通过。
- 原理与推导：  
  函数指针示例：`bool (*comp)(int,int) = myCompare;`，然后 `comp(x,y)` 调用。  
  函数对象示例：`greater<int> compObj;`，然后 `compObj(x,y)`。  
  `sort` 声明为：
  ```cpp
  template<class Iterator, class Compare>
  void sort(Iterator first, Iterator last, Compare comp);
  ```
  `Compare` 既可以是 `bool(*)(int,int)`，也可以是 `greater<int>` 类类型。编译器在实例化模板时检查 `comp(*i,*j)` 是否合法，而不关心具体类型。  
  函数对象可以携带内部状态（如调用计数），且其调用操作符允许内联优化，性能通常优于函数指针。
- 【解题步骤】
  1. 如果只需指定一种固定比较逻辑且无内部状态，使用普通函数 + 函数指针即可。
  2. 若需要更强的内联性、或要携带额外数据（如阈值），定义函数对象类并重载 `operator()`。
  3. 当需要将多种可调用类型传递给同一个泛型函数时，使用模板参数（如 `Compare`）自动适配。
- 【易错点】
  - 函数指针声明时可读性差，善用 `auto` 或 `using` 别名：`auto f = myFunc;` 或 `using Cmp = bool(*)(int,int);`。
  - 函数对象需要先实例化：`sort(v.begin(), v.end(), greater<int>())`，括号不可省略；若写成 `greater<int>` 则传递的是类型而非对象，导致编译错误。
  - 函数对象作为比较器时，应声明 `operator()` 为 `const`，因为标准算法可能要求函数调用不修改比较器本身。

| 对比维度 | 函数指针 | 函数对象 |
|----------|----------|----------|
| 类型 | `RetType (*) (Args...)` | 含有 `operator()` 的类类型 |
| 是否可带状态 | 否（静态函数） | 可以，成员变量保存状态 |
| 内联优化 | 难以内联 | 可内联，调用开销更小 |
| 典型用法 | `sort(v.begin(),v.end(),cmpFunc);` | `sort(v.begin(),v.end(),greater<int>());` |

**例题**：
**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P41] 实现一个 `mySort` 函数，使其既能接受普通比较函数，又能接受 `greater<int>()` 对象。
**思路**：模板化比较器类型。
**解答**：
```cpp
template<class Iterator, class Compare>
void mySort(Iterator first, Iterator last, Compare comp) {
    for (auto i = first; i != last; ++i)
        for (auto j = i; j != last; ++j)
            if (!comp(*i, *j))
                swap(*i, *j);
}
// 调用：
mySort(arr, arr+5, comp);           // comp 为函数指针
mySort(arr, arr+5, greater<int>()); // greater<int>() 为函数对象
```
**技巧**：STL 算法通过模板将比较准则提升为“可调用对象”概念，这使得 lambda 表达式也可直接传入（本质是匿名函数对象）。

#### operator()重载与鸭子类型 【必背】
- 【定义】函数对象的核心是重载 `operator()`。C++ 没有定义函数对象的统一接口，而是依赖“**鸭子类型**”（Duck Typing）：如果一个对象支持 `()` 调用语法，它就可以当作函数使用，无需继承特定基类。
- 原理与推导：  
  一个典型的比较函数对象如下：
  ```cpp
  template<class T>
  class greater {
  public:
      bool operator()(const T& a, const T& b) const {
          return a > b;
      }
  };
  ```
  当使用 `greater<int>()` 创建匿名对象，并将其传递给 `sort`，编译器看到 `comp(*i,*j)` 调用的是 `greater<int>::operator()`，只要该成员存在且可访问，编译就能通过。这种静态检查机制即为“鸭子类型”。  
  如果希望对象在被拷贝后仍保持行为一致，通常将 `operator()` 声明为 `const`，表示调用不会改变对象内部状态。
- 【解题步骤】
  1. 定义一个类，将行为逻辑写入 `public` 成员函数 `RetType operator() (参数列表)`。
  2. 若该对象仅作为比较器且自身无需修改，将成员函数声明为 `const`，参数尽量用 `const T&`。
  3. 创建对象：`ClassName()` 或 `ClassName obj;`，然后即可使用 `obj(...)`。
  4. 理解该机制后，可顺理成章地使用 lambda 表达式（编译器自动生成匿名函数对象类）。
- 【易错点】
  - `operator()` 必须设为 `public`，否则外部不可见。
  - 常犯错误：将 `operator()` 声明为静态 (`static`) 或虚函数，前者失去对象特性，后者破坏内联和模板调用。
  - 混淆“对象类型”和“实例”：`sort(v.begin(), v.end(), greater<int>)` 缺少括号，传递了类型而非对象，编译报错；正确为 `greater<int>()`。
  - 缺失 `const` 限定可能导致与标准库配合时编译出错，尤其当算法期望比较器为 const 可调用时。

**例题**：
**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P38] 手动实现一个 `greater` 模板类，使 `greater<int>()(2,1)` 返回 `true`，`(1,1)` 返回 `false`。
**思路**：重载 `operator()`，参数 `const T&`，返回 `bool`。
**解答**：
```cpp
template<class T>
class greater {
public:
    bool operator()(const T& a, const T& b) const {
        return a > b;
    }
};
```
**技巧**：注意三个 `const` —— 两个参数 `const` 防止误改元素，成员函数 `const` 保证比较器对象状态不变。该写法与标准库 `std::greater<T>` 一致。

#### 比较器在STL中的深度应用 【高频】
- 【定义】在 `std::sort`、`std::priority_queue`、`std::set`/`std::map` 等容器与算法中，比较器决定了元素的排序或优先级。比较器可以是函数指针或函数对象（包括预定义的 `less<T>`、`greater<T>`）。自定义类型排序时，有两个典型方案：1）重载类型自身的 `operator<`；2）提供独立的比较器类。
- 原理与推导：  
  `sort` 默认使用 `operator<` 进行升序排列。若要降序，可传入 `greater<Type>()`。  
  对于自定义类 `People`，重载 `operator<` 后，默认排序生效；但如果我们想在不同场景下按不同字段排序（年龄、体重），重载单一 `operator<` 会冲突。此时可定义比较器对象：
  ```cpp
  struct ByAge {
      bool operator()(const People& a, const People& b) const {
          return a.age < b.age;
      }
  };
  sort(v.begin(), v.end(), ByAge());
  ```
  STL 关联容器也以比较器维持有序性：`set<People, ByAge>`。
- 【解题步骤】
  1. 若只有一种全局默认排序方式，可在类内重载 `bool operator<(const T&) const`。
  2. 若需要多种排序，每种定义各自的结构体（函数对象），重载 `operator()`。
  3. 注意比较器语义必须满足严格弱序（strict weak ordering）：不能同时 `comp(a,b)` 与 `comp(b,a)` 为真；等价关系需传递。
  4. 使用标准预定义比较器：`less<T>`、`greater<T>` 位于 `<functional>`。
- 【易错点】
  - 重载 `operator<` 时，必须加 `const` 且参数使用 `const T&`，否则 STL 算法编译报错。
  - 自定义比较器若返回 `true` 表示 a 应排在 b 之前；降序时返回 `a > b`。
  - `set` 的模板参数顺序：`set<Key, Compare>`，默认为 `less<Key>`，若要传入自定义比较器，写法为 `set<People, ByAge>`，不需要加括号（因为类型即可，实例由容器内部构造）。
  - `priority_queue` 的比较器含义相反：默认大顶堆 `less` 实际上产生降序；自定义需额外注意。
  - 比较器对象一般无状态，适合短期匿名对象传递；注意生命周期（通常全程有效）。

| 方法 | 实现 | 适用场景 |
|------|------|----------|
| 重载 `operator<` | `bool operator<(const T& b) const { ... }` | 该类型天然只有一种排序规则 |
| 自定义函数对象 | `struct Cmp { bool operator()(const T&, const T&) const; };` | 多种排序规则并存，或排序标准库类型 |
| 使用标准库对象 | `greater<T>()` | 内建类型逆序 |

**题目**：[L11-iostream&函数对象和智能指针-2026.pptx P42-43] `People` 类含 `age` 和 `weight`，实现按年龄升序排序的两种方式，并说明优劣。

**解答**：

**【定义】**
函数对象（Function Object，也称仿函数 Functor）是重载了 `operator()` 的类对象，使其可以像函数一样被调用。函数指针是指向函数的指针变量，同样可以用于调用目标函数。二者从调用方的角度来看，行为统一：都可以使用 `func(args)` 的形式进行调用。

**【原理与推导】**
在 C++ 中，任何可以以 `f(args)` 形式调用的实体，都可以看作一种可调用对象（Callable）。函数指针天然支持这种调用语法；函数对象则通过自定义的 `operator()` 成员函数实现相同的语法。当我们将可调用对象作为参数传递给算法（如 `sort`）时，不需要关心它到底是函数指针还是对象实例——只要它能通过 `()` 调用并返回期望类型即可。这种“只关心行为，不关心具体类型”的思想被称为鸭子类型（Duck Typing）。

**【解题步骤】**
- 识别场景：需要将一段行为（如比较规则、处理步骤）作为参数传递给模板函数或 STL 算法。
- 决定形式：
  1. 若逻辑简单且无状态，优先使用普通函数，传函数指针。
  2. 若需要维护状态（如计数器、阈值），定义函数对象类，重载 `operator()`。
  3. 若需要将多种不同形式的可调用对象统一存储或传递，使用 `std::function` 进行类型擦除。
- 使用：传递给模板参数（直接推导类型）或 `std::function` 对象。

**【易错点】**

| 易混概念 | 函数指针 | 函数对象 |
|---------|---------|----------|
| 本质 | 指向代码段的指针 | 类实例，可拥有数据成员 |
| 状态 | 无状态（除非指向函数内静态变量） | 可以有状态（通过成员变量） |
| 类型 | 编译期确定，函数签名必须匹配 | 编译期确定类型，但通过模板参数可泛化 |
| 赋值给 `std::function` | 支持 | 支持 |
| 存储于数组中 | 需要数组元素类型一致（函数指针类型严格） | 不同类型函数对象不能直接放入同一数组，除非使用类型擦除 |

- 常见误解：认为 `std::function` 能够直接存储“所有可调用对象”的类型而不发生转换——实际上 `std::function` 使用了类型擦除，运行时可能产生性能开销，不能完全替代模板。
- 错误用法：试图将一个有状态的函数对象赋值给普通函数指针（无法实现）。

**例题**：

**【定义】**
鸭子类型（Duck Typing）是一种动态类型风格：如果一个对象能像函数一样调用（即支持 `obj(args)`），那么它就可以被当作函数使用，而不必关心它到底是函数指针还是某个类的实例。在 C++ 中，这一特性通过重载 `operator()` 并结合模板实现。

**【原理与推导】**
C++ 模板在编译期间会对传入的参数进行实例化，要求参数支持某种操作（如 `a(b)`）。如果传入的对象拥有 `operator()` 成员函数，模板代码就能通过编译。编译器并不会在模板定义处检查类型的具体实现，而是等到实例化时进行匹配。这恰好实现了鸭子类型：只要看起来像函数、能调用，就能作为函数使用。

函数对象重载 `operator()` 的基本形式：
```cpp
class AgeComp {
public:
    bool operator()(const People &a, const People &b) const {
        return a.age < b.age;
    }
};
```
使用时：
```cpp
AgeComp comp;
bool res = comp(p1, p2);        // 直接调用
sort(vec.begin(), vec.end(), AgeComp()); // 传递给 STL 算法
```

**【解题步骤】**
1. 设计一个类，其内部可能包含需要持久化的状态（如比较的基准值）。
2. 在类中声明 `返回类型 operator()(参数列表) const;`（常成员函数以避免副作用）。
3. 将该类的临时对象或实例作为参数传递给接受可调用对象的模板函数。

**【易错点】**

| 易混概念 | 函数调用 | 普通成员函数调用 |
|---------|---------|-----------------|
| 语法 | `obj(args)` | `obj.func(args)` |
| 本质 | 调用 `obj.operator()(args)` | 调用具名成员函数 |
| 鸭子类型适用 | 支持 | 不支持（除非模板要求具名成员） |

- 常见错误：在传递函数对象时忘记加括号，例如 `sort(... , AgeComp)` 传入的是类型名而非实例，应写成 `AgeComp()`。
- 注意 `operator()` 的 const 修饰：如果缺失，可能无法绑定到某些只接受常引用可调用对象的算法。

**例题**：
**题目**：定义一个函数对象 `Threshold`，其构造函数接受一个阈值 `int t`，内部重载 `operator()(int x)` 返回 `x > t`。配合 `count_if` 统计一个 `vector<int>` 中大于 5 的元素个数。 [来源 L11-iostream&函数对象和智能指针-2026.pptx P45 改编]

**【定义】**
STL 中的许多算法（如 `sort`、`lower_bound`、`priority_queue`）以及关联容器（`map`、`set`）都需要一种“比较关系”来确定元素顺序。比较器可以是函数指针、函数对象或 lambda 表达式，只要满足严格弱序要求。

**【原理与推导】**
以 `sort` 为例，默认使用 `operator<` 进行升序排序。如果要自定义排序规则，可提供第三个参数 `comp`，要求 `comp(a, b)` 在 `a` 应排在 `b` 之前时返回 `true`。STL 通过模板接收任意类型的比较器，在算法内部调用 `comp(*it1, *it2)`。实现比较器的方式有三种：
1. **重载 `operator<`**：对自定义类型的对象，直接定义默认的 `<` 比较。
2. **定义比较函数**：编写全局函数，传入函数指针。
3. **定义函数对象**：重载 `operator()` 的类，可带状态。
4. （C++11）**lambda 表达式**：直接内联写匿名函数。

**【解题步骤】**
- 确定排序或查找的规则。
- 选择合适的方式实现比较器：
  - 若规则固定且无额外状态，优先使用 lambda 或普通函数。
  - 若需要复用比较器并携带参数，用函数对象。
  - 若只需要改变某一次排序，使用 lambda。
- 传递给 STL 算法或容器（如 `map<Key, Value, Comp>`）。

**【易错点】**
- 严格弱序：比较函数必须满足非自反性、非对称性和传递性。例如 `comp(a,a)` 必须返回 `false`。常见错误：`return a <= b` 会导致 `a <= a` 返回 `true`，违反严格弱序。
- 函数对象作为模板参数：对于 `map` 等容器，需要在模板参数中传入比较器的类型，构造函数中传入实例。例如 `map<People, int, AgeComp> m;`，如果 `AgeComp` 是类名，默认构造即可。
- 忘记 `const`：`operator()` 必须声明为 `const` 成员，否则无法被只接受常引用比较器的算法使用。

#### std::function类型擦除机制 【必背】

**【定义】**
`std::function` 是一个通用多态函数包装器，定义在 `<functional>` 头文件中。它可以存储、复制、调用任何满足指定签名的可调用目标：函数、函数对象、lambda 表达式、成员函数指针等。其实现依赖类型擦除（Type Erasure）技术。

**【原理与推导】**
类型擦除的核心思想是将具体类型的信息隐藏在一个统一的接口后面。`std::function<Ret(Args...)>` 内部通常会持有：
- 一个基类（如 `callable_base`），定义了虚函数 `invoke` 用于调用目标；
- 针对每种具体可调用对象类型，派生出一个模板子类，实现 `invoke`；
- 通过一个基类指针（或类似机制）指向实际的派生对象，在调用时进行动态分发。

这样，`std::function` 只需要知道返回值类型和参数列表，不需要知道被包装对象的静态类型，从而可以在运行时统一处理不同类型的可调用实体。

使用示例：
```cpp
#include <functional>
std::function<string()> f;
f = readFromScreen;        // 函数指针
f = ReadFromFile();        // 函数对象
string result = f();       // 调用
```

**【解题步骤】**
1. 确定可调用实体的签名：返回类型 `Ret` 和参数类型 `Args...`。
2. 声明 `std::function<Ret(Args...)>` 对象。
3. 通过赋值或构造函数将可调用对象存入其中。
4. 通过 `operator()` 触发调用。

**【易错点】**

| 易混概念 | 模板参数 | std::function |
|---------|---------|---------------|
| 确定时机 | 编译期 | 运行期（通过类型擦除） |
| 运行开销 | 无额外开销（编译期内联优化） | 有虚函数调用或间接调用开销 |
| 可统一种类的类型 | 同一模板实例化只能处理确定类型，但可通过模板参数泛化 | 可以接受任何匹配签名的可调用对象 |

- 错误：试图用 `std::function` 完全替代模板以获得灵活性，忽略了性能损失。
- 内存占用：`std::function` 可能使用堆分配来存储较大的可调用对象（若大于内部缓冲区，通常一个指针大小以上）。

**【例题】**

**例题**：
**题目**：解释为什么 `string(*readFunc2)(); readFunc2 = ReadFromFile();` 编译失败，而 `std::function<string()> readFunc; readFunc = ReadFromFile();` 成功。 [来源 L11-iostream&函数对象和智能指针-2026.pptx P53]
**思路**：函数指针类型必须严格匹配函数的静态类型，而 `ReadFromFile()` 是一个类的实例，不是函数。`std::function` 通过类型擦除抹去了具体类型差异。
**解答**：
函数指针 `string(*)(void)` 只能指向具有完全匹配签名的函数。`ReadFromFile` 是一个类类型，其对象不能隐式转换为函数指针。因此赋值非法。
`std::function<string()>` 实现了类型擦除，内部会构造一个适配器，使得 `ReadFromFile` 的 `operator()` 调用与 `string()` 接口匹配，因此赋值合法。
**技巧**：如需将不同的可调用对象统一保存（如存于容器、作为回调），使用 `std::function` 消除类型差异。

#### 模板、虚函数与std::function实现多态的对比 【必背】

**【定义】**
在 C++ 中实现多态有三种典型方式：
1. **虚函数多态**：通过基类指针或引用调用派生类重写的虚函数。
2. **模板（编译期多态）**：通过模板参数传递不同类型，在编译期间生成对应版本的函数。
3. **std::function 多态**：利用类型擦除，在运行期统一包装不同类型可调用对象。

**【原理与推导】**
- **虚函数**：基于继承和虚函数表（vtable），运行时通过查表决定调用哪个函数。需要预先定义基类接口，所有派生类必须继承自该基类。
- **模板**：在编译时根据传入参数的实际类型实例化函数或类，不同实例可调用对应类型的成员函数，无需共同基类。实现的是静态多态（鸭子类型）。
- **std::function**：通过类型擦除将调用逻辑封装在内部，向外暴露统一的函数签名接口。在运行时通过间接调用实现多态，无需使用者关心具体类型。

三者对比如下：

| 特性 | 虚函数 | 模板 | std::function |
|------|--------|------|---------------|
| 多态时机 | 运行期 | 编译期 | 运行期（类型擦除） |
| 是否需要公共基类 | 是 | 否（鸭子类型） | 否（只需签名匹配） |
| 运行时开销 | 一次虚函数查表 | 无（内联优化） | 间接调用，可能涉及堆分配 |
| 灵活性 | 受限于继承体系 | 非常高，但类型需在编译期确定 | 运行时灵活组合不同类型的可调用实体 |
| 代码膨胀 | 小 | 易致代码膨胀（大量实例化） | 较小 |

**【解题步骤】**
在需要回调或多态行为的场景，根据需求选择：
- 如果多态类型的数量在编译期就已固定，且对性能要求极高，使用模板。
- 如果多态类型需要运行时动态改变（如通过配置文件动态加载处理模块），且可以承担一定开销，使用 `std::function` 或虚函数。
- 如果已经有完善的类层次结构，且需要扩展性强，使用虚函数。

**【易错点】**
- 误以为 `std::function` 是模板，运行时零开销。实际上它通常会引入动态内存分配和间接调用。
- 模板在调用不同函数对象时，如果函数对象类型不同，会实例化出多份代码，可能导致可执行文件体积增大。
- 用 `std::function` 作为函数参数可能导致不必要的类型擦除开销，当不需要运行时多态时，应优先使用模板参数。

**【例题】**

**例题**：
**题目**：对比以下三种实现 `process` 函数的方式，说明各自的优缺点和使用场景。 [来源 L11-iostream&函数对象和智能指针-2026.pptx P47-55]
```cpp
// 方式一：虚函数
class CalculatorBase {
public:
    virtual string read() = 0;
    virtual string calculate(string) = 0;
    virtual void write(string) = 0;
    void process() {
        string data = read();
        string output = calculate(data);
        write(output);
    }
};

// 方式二：模板
template<class ReadFunc, class CalFunc, class WriteFunc>
void process(ReadFunc r, CalFunc c, WriteFunc w) {
    string data = r();
    string output = c(data);
    w(output);
}

// 方式三：std::function
void process(function<string()> r,
             function<string(string)> c,
             function<void(string)> w) {
    string data = r();
    string output = c(data);
    w(output);
}
```
**思路**：从多态时机、代码重用、性能、灵活性角度分析。
**解答**：
- **虚函数版**：需要继承体系，运行时多态。适合有一族具有相同接口的类，且运行时需要动态绑定。扩展需新增子类，不修改 `process`。
- **模板版**：编译期多态，无需继承，灵活支持任何具有对应调用能力的类型（函数指针、函数对象等）。零运行时开销，但会造成模板实例化膨胀，且不同类型不能放入统一容器。
- **std::function 版**：运行时类型擦除，兼具灵活性，可统一存储不同可调用对象，但引入了间接调用和内存分配开销。适合需要动态变换回调，但对性能不极敏感的场景。
**技巧**：先考虑模板实现高性能，若需动态存储多种回调则引入 `std::function`，若已有完善的类体系则用虚函数。
### 五、智能指针与引用计数回顾

#### shared_ptr 引用计数原理与循环引用 【高频】

【定义】  
`std::shared_ptr` 是 C++11 引入的智能指针，位于 `<memory>` 头文件。它通过**引用计数**实现共享所有权：多个 `shared_ptr` 可以指向同一个对象，内部维护一个控制块记录引用计数，当计数归零时自动 `delete` 所指对象。

原理与推导  
每个 `shared_ptr` 内部存储一个指向所管理对象的裸指针和一个指向控制块的指针。控制块包含引用计数、弱引用计数及删除器等信息。  
- 构造函数 `shared_ptr<T>(new T(...))` 创建新控制块，引用计数初始化为 1。  
- 拷贝构造/拷贝赋值时，两个 `shared_ptr` 共享同一个控制块，引用计数加 1。  
- 移动构造/移动赋值时，所有权转移，引用计数不变。  
- 析构或 `reset()` 时引用计数减 1；若计数变为 0，则调用 `delete` 销毁对象并释放控制块。

**循环引用问题**：若两个对象互相持有对方的 `shared_ptr`，则引用计数永远不为 0，导致内存泄漏。  
例如：类 `Parent` 持有 `shared_ptr<Child>`，类 `Child` 持有 `shared_ptr<Parent>`，形成环状结构，离开作用域后两者均不会析构。

【解题步骤】  
分析引用计数的考题通常按以下步骤：  
1. 标明每个 `shared_ptr` 变量的生命周期（作用域）。  
2. 追踪构造、拷贝、赋值、移动、`reset()` 等操作对引用计数的影响。  
3. 判断对象销毁的时机（引用计数何时变为 0）。  
4. 若存在双向引用，识别循环引用并判断是否泄漏，应用 `weak_ptr` 解决。

【易错点】  
❌ 使用同一裸指针初始化多个 `shared_ptr`  
```cpp
int* p = new int(5);
shared_ptr<int> sp1(p);
shared_ptr<int> sp2(p);   // 错误！产生两个独立控制块，导致重复 delete
```
✅ 应直接从 `new` 表达式初始化或使用 `make_shared`。  

❌ 使用 `shared_ptr` 管理动态数组  
```cpp
shared_ptr<int> sp(new int[10]); // 错误！默认删除器使用 delete，应使用 delete[]
```
✅ 建议用 `std::vector` 或自定义删除器。  

❌ 循环引用未使用 `weak_ptr` 导致泄漏。

【例题】  
**题目**：阅读以下代码，写出程序的输出。 [L11-iostream&函数对象和智能指针-2026.pptx P61]  
```cpp
#include <memory>
#include <iostream>
using namespace std;

int main() {
    shared_ptr<int> p1(new int(4));
    cout << p1.use_count() << ' ';
    {
        shared_ptr<int> p2 = p1;
        cout << p1.use_count() << ' ';
        cout << p2.use_count() << ' ';
    }
    cout << p1.use_count() << ' ';
}
```
**思路**：根据作用域分析引用计数的变化：`p2` 在内部块内拷贝自 `p1`，离开块后销毁，计数恢复。  
**解答**：  
1. `p1` 创建，引用计数 = 1，输出 `1 `。  
2. 进入内部块，`p2` 拷贝自 `p1`，计数变为 2，`p1.use_count()` 和 `p2.use_count()` 均输出 `2 `。  
3. 离开内部块，`p2` 析构，计数减为 1，输出 `1`。  
程序最终输出：`1 2 2 1`。  
**技巧**：`shared_ptr` 的拷贝构造和赋值均会使引用计数加 1；`use_count()` 返回当前控制块的共享指针数量。

#### 手写引用计数（SmartPtr）的拷贝赋值语义 【必背】

【定义】  
通过自定义模板类 `SmartPtr` 和辅助类 `U_Ptr` 实现引用计数智能指针，理解 `shared_ptr` 底层拷贝、赋值、析构的计数管理机制。

原理与推导  
- 辅助类 `U_Ptr<T>` 存放实际数据指针 `T* p` 和引用计数 `int count`，构造函数将 `count` 初始化为 1。`SmartPtr` 持有 `U_Ptr<T>* rp`。  
- **拷贝构造** `SmartPtr(const SmartPtr& sp)`：将 `rp` 指向与源对象相同的 `U_Ptr`，并使 `rp->count` 增加 1。  
- **拷贝赋值运算符** `operator=(const SmartPtr& rhs)`：  
  ++`rhs.rp->count`；  
  if `--rp->count == 0`，则 `delete rp`；  
  `rp = rhs.rp`。  
  该顺序保证自赋值安全：先增加右侧计数，再减少自身计数，即使 `this == &rhs` 也能正确工作。  
- **析构函数**：`--rp->count` 递减，若为 0 则 `delete rp`（`U_Ptr` 析构函数会 `delete p` 释放实际对象）。  
- **解引用运算符** `*` 和成员访问运算符 `->`：分别返回 `*(rp->p)` 和 `rp->p`。  
- **友元关系**：`SmartPtr` 需要访问 `U_Ptr` 的私有成员，故将 `SmartPtr<T>` 声明为 `U_Ptr` 的友元类。

【解题步骤】  
分析自己实现的智能指针时：  
1. 确定 `U_Ptr` 对象的创建时机（智能指针构造时）。  
2. 跟踪拷贝/赋值时引用计数的增减顺序（特别注意赋值运算符的先加后减）。  
3. 判断何时删除 `U_Ptr` 指针（即删除控制块）以及何时删除实际对象（`U_Ptr` 析构时）。  
4. 验证自赋值安全性和异常安全性。

【易错点】  
- 赋值运算符若先减后加会导致自赋值时对象被提前删除。  
- 忘记友元声明会导致 `SmartPtr` 无法访问 `U_Ptr` 的私有成员 `count` 和 `p`。  
- `operator*` 返回引用，`operator->` 返回指针。  
- 在实际对象析构后继续访问（如 `use_count()` 仅表示共享指针数量，不代表对象依然存活）。

【例题】  
**题目**：给定以下自定义智能指针代码，运行 `main` 函数，写出输出结果，并解释赋值 `ptr3 = ptr2` 时引用计数的变化。 [L11-iostream&函数对象和智能指针-2026.pptx P66-68]  
```cpp
template <typename T> class SmartPtr;
template <typename T>
class U_Ptr {
    friend class SmartPtr<T>;
    U_Ptr(T *ptr) : p(ptr), count(1) {}
    ~U_Ptr() { delete p; }
    int count;
    T *p;
};

template <typename T>
class SmartPtr {
    U_Ptr<T> *rp;
public:
    SmartPtr(T *ptr) : rp(new U_Ptr<T>(ptr)) {}
    SmartPtr(const SmartPtr<T> &sp) : rp(sp.rp) { ++rp->count; }
    SmartPtr& operator=(const SmartPtr<T>& rhs) {
        ++rhs.rp->count;
        if (--rp->count == 0) delete rp;
        rp = rhs.rp;
        return *this;
    }
    ~SmartPtr() { if (--rp->count == 0) delete rp; }
    T& operator*() { return *(rp->p); }
    T* operator->() { return rp->p; }
};

int main() {
    int *pi = new int(2);
    SmartPtr<int> ptr1(pi);
    SmartPtr<int> ptr2(ptr1);
    SmartPtr<int> ptr3(new int(3));
    ptr3 = ptr2;
    cout << *ptr1 << endl;
    *ptr1 = 20;
    cout << *ptr2 << endl;
    return 0;
}
```
**思路**：  
- `ptr1` 绑定 `int(2)`，计数 1。  
- `ptr2` 拷贝自 `ptr1`，两者指向同一 `U_Ptr`，计数变为 2。  
- `ptr3` 新建 `int(3)`，计数 1。  
- `ptr3 = ptr2`：先增加 `ptr2` 的计数（变为 3），再减少 `ptr3` 原先计数（变 0），销毁 `int(3)` 的控制块；然后 `ptr3.rp` 指向 `ptr2` 的控制块。最终三个 `SmartPtr` 共享 `int(2)`，计数 3。  
- 输出 `*ptr1` 为 2；`*ptr1 = 20` 修改共享对象；`*ptr2` 也输出 20。  
**解答**：程序输出：  
```
2
20
```
赋值 `ptr3 = ptr2` 时：`rhs`（即 `ptr2`）引用计数先由 2 增为 3；`ptr3` 原引用计数由 1 减为 0，触发 `delete rp`（从而 `delete int(3)`）；最后 `ptr3.rp` 指向 `ptr2` 的控制块。  
**技巧**：赋值运算符中先增 `rhs` 计数再减自身计数，是处理自赋值的经典写法，同时保证异常安全。

#### weak_ptr 与 shared_ptr 协作打破循环 【高频】

【定义】  
`std::weak_ptr` 是一种不控制对象生命周期的智能指针，指向 `shared_ptr` 管理的对象，但不增加引用计数。主要用于监视 `shared_ptr` 对象是否存在，以及解决 `shared_ptr` 循环引用造成的内存泄漏。

原理与推导  
- `weak_ptr` 必须从 `shared_ptr` 或另一个 `weak_ptr` 构造，指向由 `shared_ptr` 管理的内存。  
- 它不参与引用计数，因此不影响对象的销毁。  
- 使用 `lock()` 返回一个临时的 `shared_ptr`，若对象还存在则返回非空指针，否则返回 `nullptr`。  
- 成员函数 `use_count()` 返回对应 `shared_ptr` 的引用计数；`expired()` 返回对象是否已被销毁（即引用计数为 0）。  
- 典型应用场景：树结构中的父指针、观察者模式、缓存等需要反向引用又不希望阻止销毁的场景。

**打破循环引用**：当两个类互相持有对方的 `shared_ptr` 时，会发生循环引用。将其中一个改为 `weak_ptr` 后，该指针不增加引用计数，打破环状依赖，保证析构链顺利进行。

【解题步骤】  
处理循环引用类时：  
1. 画出对象间持有关系，确定哪些指针必须是 `shared_ptr`，哪些可用 `weak_ptr`。  
2. 修改循环中的一方为 `weak_ptr`，并通过 `lock()` 临时获取 `shared_ptr` 来访问对象。  
3. 确保在 `lock()` 返回的 `shared_ptr` 存活期间对象不被意外销毁。  
4. 必要时使用 `expired()` 检查对象是否有效，避免空悬访问。

【易错点】  
❌ 误用 `weak_ptr` 直接解引用：`weak_ptr` 不能直接使用 `*` 或 `->`，必须先 `lock()` 转换为 `shared_ptr`。  
❌ 循环引用判断错误：只有形成双向 `shared_ptr` 持有才会泄漏，单向持有不存在问题。  
❌ 访问 `weak_ptr` 时忘记检查 `expired()` 或 `lock()` 返回值是否为 `nullptr`。

【例题】  
**题目**：以下代码在 `test()` 中创建了两个相互引用的对象，请问输出什么？为什么？如何修改才能让对象正常析构？ [L11-iostream&函数对象和智能指针-2026.pptx P70-74]  
```cpp
#include <memory>
#include <iostream>
using namespace std;

class Child;
class Parent {
    shared_ptr<Child> child;
public:
    Parent()  { cout << "parent constructing" << endl; }
    ~Parent() { cout << "parent destructing" << endl; }
    void setChild(shared_ptr<Child> c) { child = c; }
};

class Child {
    shared_ptr<Parent> parent;   // 造成循环引用
public:
    Child()  { cout << "child constructing" << endl; }
    ~Child() { cout << "child destructing" << endl; }
    void setParent(shared_ptr<Parent> p) { parent = p; }
};

void test() {
    shared_ptr<Parent> p(new Parent());
    shared_ptr<Child> c(new Child());
    p->setChild(c);
    c->setParent(p);
}

int main() {
    test();
    return 0;
}
```
**思路**：`test` 中 `p` 和 `c` 均为 `shared_ptr`，互相持有对方后引用计数均为 2，离开 `test` 时各自 `reset` 使计数降为 1，均不为 0，导致两个对象的内存泄漏。  
**解答**：  
- 原程序输出：`parent constructing` 和 `child constructing`，没有析构输出。  
- 修改方法：将 `Child` 类中的 `shared_ptr<Parent>` 改为 `weak_ptr<Parent>`，并调整 `setParent` 保持传入 `shared_ptr` 转换为 `weak_ptr`，访问 `Parent` 时通过 `lock()` 获得临时 `shared_ptr`。  
修改后程序输出：  
```
parent constructing
child constructing
parent destructing
child destructing
```
**技巧**：设计双向关联时，选择“拥有权”较弱的方使用 `weak_ptr`；被 `weak_ptr` 指向的对象不会阻止其析构。

#### unique_ptr 独占所有权与移动语义 【重点】

【定义】  
`std::unique_ptr` 是一种独占所有权的智能指针，同一时刻只能有一个 `unique_ptr` 指向特定对象。它无法拷贝，只能移动，体现资源独占语义。离开作用域时自动 `delete` 所管理的对象。

原理与推导  
- 构造函数 `unique_ptr<T>(new T(...))` 获得所有权；C++14 起推荐使用 `std::make_unique<T>(args...)` 避免裸指针暴露。  
- `unique_ptr` 禁用了拷贝构造和拷贝赋值，仅允许移动构造和移动赋值。移动操作将内部指针的所有权转移给目标 `unique_ptr`，源指针变为空（`nullptr`）。  
- `release()` 放弃所有权，返回裸指针，并且 `unique_ptr` 自身变为空。调用者需自行 `delete` 返回的指针（或将其移交给另一个智能指针）。  
- `reset()` 销毁当前管理的对象并接管新指针（或置空）。  
- 由于没有引用计数开销，性能优于 `shared_ptr`，且语义更清晰。

【解题步骤】  
使用 `unique_ptr` 时需注意：  
1. 确定资源的所有权模型——是否只能有一个拥有者。  
2. 用 `std::move` 显式转移所有权。  
3. 若需要将所有权交给外部，使用 `release()` 并确保接收方负责释放。  
4. 不应直接使用`unique_ptr`的地址进行多次释放，也不应把`release()`返回的裸指针交付给多个 `unique_ptr`。  

【易错点】  
❌ 尝试拷贝 `unique_ptr`，导致编译错误。  
```cpp
unique_ptr<int> up1 = make_unique<int>(10);
unique_ptr<int> up2 = up1;   // 错误！拷贝构造被删除
```
❌ 移动后继续使用源 `unique_ptr`（源指针已为空）。  
❌ `release()` 后忘记释放裸指针，造成内存泄漏。  
```cpp
int* raw = up.release();   // 必须负责 delete raw
```
❌ 使用 `unique_ptr` 管理数组时，默认删除器调用 `delete` 而非 `delete[]`，应使用特化 `unique_ptr<T[]>`。

【例题】  
**题目**：写出以下程序的执行结果，并说明 `release()` 和 `move` 的作用。 [L11-iostream&函数对象和智能指针-2026.pptx P80]  
```cpp
#include <memory>
#include <iostream>
using namespace std;

int main() {
    auto up1 = make_unique<int>(20);
    // unique_ptr<int> up2 = up1; // 编译错误，不能复制
    unique_ptr<int> up2 = std::move(up1);  // 移动所有权
    int* p = up2.release();               // 放弃控制权
    cout << *p << endl;                   // 输出 20
    delete p;
    return 0;
}
```
**思路**：`up1` 管理 `int(20)`，通过 `move` 转移给 `up2`，`up1` 变为空；`up2.release()` 放弃所有权返回裸指针，`up2` 变空，最后手动 `delete` 释放。  
**解答**：  
程序输出 `20`。  
- `std::move(up1)` 将 `up1` 所管理的对象转移给 `up2`，`up1` 不再持有任何资源。  
- `release()` 释放 `up2` 对对象的所有权，返回底层裸指针，调用者需要负责释放（这里通过 `delete p` 完成）。  
**技巧**：`unique_ptr` 的移动语义确保资源所有权唯一且可传递；`release()` 常用于将所有权交给传统 API 或需要手动控制的场景，注意避免泄漏。
### 七、正则表达式与C++实现

#### 正则表达式语法基础 【高频】

##### 【定义】正则表达式
正则表达式是由字母和符号组成的特殊文本，用于定义搜索文本时的一种规则模式。它可以极大地简化字符串匹配、查找和替换的复杂度。

##### 原理与推导
传统字符串校验方式（如 `check` 函数示例）需要对每个字符逐一判断，代码冗长且难以维护。正则表达式提供了一种声明式的模式匹配语言，通过特定的元字符组合来描述字符串的构成规则，将复杂的逻辑判断转化为模式字符串。

##### 【解题步骤】
1. 分析目标字符串的构成规则。
2. 将规则拆解为字符要求和重复次数。
3. 使用字符簇和重复模式组合成完整的正则表达式。

##### 字符簇与特殊字符
| 语法 | 含义 | 示例 |
|------|------|------|
| `[a-z]` | 匹配单个小写字母 | `[a-z]` 匹配 `a` 到 `z` 中的任意一个字符 |
| `[0-9]` | 匹配单个数字 | `[0-9]` 匹配 `0` 到 `9` 中的任意一个字符 |
| `[aeiou]` | 匹配指定集合中的任意一个字符 | 匹配任意一个元音字母 |
| `[^a-z]` | **取反**：匹配不在范围内的任意单个字符 | `[^c]ar` 匹配 `par`、`gar`，但不匹配 `car` |
| `\d` | 等价于 `[0-9]`，匹配单个数字 | `\d\d` 匹配 `12` |
| `\D` | 等价于 `[^0-9]`，匹配单个非数字 | `\D` 匹配 `a`、`B`、`_` |
| `\w` | 等价于 `[a-zA-Z0-9_]`，匹配字母、数字、下划线 | `\w+` 匹配 `hello_123` |
| `\W` | 等价于 `[^a-zA-Z0-9_]`，匹配非单词字符 | `\W` 匹配 `@`、` ` |
| `\s` | 匹配所有空白字符（空格、`\t`、`\n` 等） | `a\sb` 匹配 `a b` |
| `\S` | 匹配所有非空白字符 | `\S+` 匹配 `hello` |
| `.` | 匹配除换行符 `\n` 以外的任意单个字符 | `.ar` 匹配 `car`、`gar` |
| `\.` | 转义：匹配字面意义上的句号字符 `.` | `ge\.` 匹配 `ge.` |
| `^` | 匹配字符串的**开头** | `^The` 只匹配以 `The` 开头的字符串 |
| `$` | 匹配字符串的**结尾** | `bucket$` 只匹配以 `bucket` 结尾的字符串 |

##### 重复模式
| 语法 | 含义 | 示例 |
|------|------|------|
| `{n}` | 前面的内容恰好出现 `n` 次 | `a{4}` 匹配 `aaaa` |
| `{n,m}` | 前面的内容出现 `n` 到 `m` 次 | `a{2,4}` 匹配 `aa`、`aaa`、`aaaa` |
| `{n,}` | 前面的内容至少出现 `n` 次 | `a{2,}` 匹配 `aa`、`aaa`、... |
| `+` | 前面的内容至少出现 1 次（等价于 `{1,}`） | `a\w+` 匹配 `ar`、`arked` |
| `*` | 前面的内容出现 0 次或多次（等价于 `{0,}`） | `a\w*` 匹配 `a`、`ar`、`arked` |

##### 【易错点】
| 易混概念 | 区别与说明 |
|----------|------------|
| `[a-z]` vs `[a-z]+` | `[...]` 本身只匹配**单个**字符；必须配合 `+`、`*` 或 `{n,m}` 等才能匹配多个字符 |
| 通配符 `.` vs 正则 `.` | Shell 通配符中 `*` 代表任意字符串；正则中用 `.*` 表达类似含义 |
| `\` 转义冲突 | C++ 字符串中 `\` 也是转义符，写正则时必须用 `\\d` 或使用原生字符串 `R"(\d)"` |
| 大小写敏感 | `[a-z]` 不匹配大写字母，如果需要匹配建议使用 `[a-zA-Z]` 或设置忽略大小写标志 |
| 取反 `[^...]` | `[^0-9]` 匹配任意一个非数字字符，不是匹配"没有数字"，需要配合 `+` 使用 |

##### 【例题】
**例题**：用户名注册场景（课件 P84-85）
**题目**：设计一个正则表达式，校验用户名长度为3~15个字符，且只能包含小写字母、数字、下划线。合法：`john_123`；非法：`John_123` / `jo` / `@john`。
**思路**：拆解规则为：字符集 `[a-z0-9_]` 和长度限制 `{3,15}`，再加上字符串起止锚定符确保完全匹配。
**解答**：
```
^[a-z0-9_]{3,15}$
```
- `^` 和 `$` 确保整个字符串完全匹配，而非仅匹配子串。
- `[a-z0-9_]` 定义允许的单个字符范围。
- `{3,15}` 限制前面的字符集重复次数。
**技巧**：校验类场景必须加 `^` 和 `$`，否则只要存在合法子串就会误判通过。

#### C++正则表达式库 `<regex>` 【重点】

##### 正则表达式的三种模式
| 模式 | 含义 | 示例场景 |
|------|------|----------|
| **匹配 (Match)** | 判断**整个字符串**是否完全满足正则表达式 | 用户名格式校验 |
| **搜索 (Search)** | 在字符串中**查找**满足正则表达式的子串 | 提取文本中的数字串 |
| **替换 (Replace)** | 将匹配的子串**替换**为指定文本 | 敏感词过滤、格式化 |

##### 【定义】原生字符串
C++ 原生字符串语法 `R"(...)"` 取消转义，保留字符串的字面值。书写正则表达式时强烈推荐使用，避免双重转义。

```
"\\d+"  =  R"(\d+)"    // 都表示正则表达式 \d+
```

##### 关键组件与用法
| 功能 | 库组件 | 说明 |
|------|--------|------|
| 正则表达式对象 | `std::regex` | 保存编译后的正则表达式 |
| 匹配结果 | `std::smatch` | 存储匹配结果及捕获分组信息 |
| 完全匹配 | `std::regex_match(s, re)` | 字符串 `s` 是否**完全**匹配 `re` |
| 完全匹配（含捕获） | `std::regex_match(s, result, re)` | 匹配并存储结果到 `smatch result` |
| 搜索 | `std::regex_search(s, result, re)` | 搜索**第一个**匹配的子串 |
| 替换 | `std::regex_replace(s, re, fmt)` | 替换**所有**匹配子串为 `fmt`，返回新字符串 |

#### 捕获与分组 【必背】

分组使用 `()` 将正则表达式的一部分括起来。匹配成功后，每个括号对应的子匹配内容会被"捕获"并存储在 `smatch` 对象中，便于提取结构化的信息片段。

1. 在正则表达式中用 `()` 包裹需要提取的部分。
2. 调用 `regex_match` 或 `regex_search`，传入 `smatch` 对象。
3. 通过 `sm[i]` 访问第 `i` 个分组内容。
   - `sm[0]`：始终是完整匹配的字符串。
   - `sm[1]`：第一个括号分组的内容。
   - `sm[2]`：第二个括号分组的内容，依此类推。

| 错误写法/理解 | 正确写法/理解 |
|---------------|----------------|
| 认为 `sm[0]` 是第一个括号的内容 | `sm[0]` 是完整匹配，`sm[1]` 才是第一个括号分组 |
| `regex_match` 和 `regex_search` 混用 | `match` 要求**全串**匹配；`search` 只要求**存在子串**匹配 |
| 连续搜索时忘记更新搜索范围 | 使用 `sm.suffix().str()` 获取剩余未搜索的部分作为下一次搜索的字符串 |

##### 替换中的分组引用
在 `regex_replace` 的替换文本中，可使用特殊符号引用捕获分组：
- `$&`：代表整个匹配的子串。
- `$1`, `$2`, ...：代表第 1、2 个捕获分组的内容。

**示例**：`regex_replace(s, regex(R"((sub)([\S]*))"), "[$&]")` 将每个匹配的单词用方括号括起来。

#### 综合应用示例：学生信息整理 【高频】

##### 背景与分析
从一段自我介绍文本中分别提取：姓名、出生日期、电话号码、邮箱。各信息特征如下：
- **姓名**：以 "I am xxx." 或 "My name is xxx." 开头。
- **出生日期**：`yyyy-mm-dd` 或 `yyyy.mm.dd` 格式。
- **电话**：11位数字，不以0开头。
- **邮箱**：由数字、字母、`@`、`.` 组成。

1. 分别为每个信息项编写对应的正则表达式。
2. 对同一输入字符串分别执行 `regex_search` 提取各项。
3. 对于日期，利用捕获分组分别获取年、月、日，并进行格式化（如补零）。
4. 按要求格式输出所有提取到的信息。

##### 关键正则表达式设计
| 信息项 | 正则表达式 | 说明 |
|--------|-----------|------|
| 姓名 | `(My name is \|I am )(\w+)\.` | 捕获第二个分组 `sm[2]` 得到姓名 |
| 生日 | `(\d{4})[\.-](\d{1,2})[\.-](\d{1,2})` | 捕获三个分组：年、月、日 |
| 电话 | `[1-9]\d{10}` | 首位1-9，后跟10位数字，共11位 |
| 邮箱 | `[\w]+@(\w+\.)+\w+` | 匹配 `xxx@yyy.zzz` 格式，分组1捕获主域名部分 |

##### 例题：【课件 P103-105】
**例题**：学生信息提取
**题目**：
输入字符串：
```
I am zhangshuaishuai. I was born on 2000.10.2. My phone number is 18866667777 and you can also reach me by my email: zhangss@tsinghua.edu.cn
```
提取姓名、生日（格式 `yyyy.mm.dd`，月日不足两位补零）、电话号码、邮箱并输出。

**思路**：
1. 分别定义姓名、日期、电话、邮箱的正则表达式对象。
2. 使用 `regex_search` 逐个提取信息，利用 `smatch` 获取捕获分组。
3. 日期部分使用 `stoi` 将捕获的年月日字符串转为整数，再用标准输出控制格式（本例中课件代码直接输出 `sm[i]` 未补零，实际生产环境中可按需格式化）。

**解答**：
```cpp
void extract(string input) {
    smatch sm;
    regex get_name(R"((My name is |I am )(\w+)\.)");
    regex get_date(R"((\d{4})[\.-](\d{1,2})[\.-](\d{1,2}))");
    regex get_mobile(R"([1-9]\d{10})");
    regex get_email(R"([\w]+@(\w+\.)+\w+)");

    if (std::regex_search(input, sm, get_name))
        cout << sm[2] << endl;
    
    int date[3] = {0};
    if (regex_search(input, sm, get_date)){
        for (int i = 1; i <= 3; i++)
            date[i - 1] = stoi(sm[i]);
        cout << date[0] << "." << date[1] << "." << date[2] << endl;
    }
    
    if (regex_search(input, sm, get_mobile))
        cout << sm[0] << endl;
    
    if (regex_search(input, sm, get_email))
        cout << sm[0] << endl;
}
```
**技巧**：姓名正则 `(\w+)\.` 依赖句子结尾的句号作为分隔，实际应用需注意文本格式稳定性；电话正则已通过首位 `[1-9]` 排除了以 0 开头的号码。

##### 1. 【定义】
`std::function` 是C++标准库提供的通用多态函数包装器，可以存储、复制和调用任何可调用目标（函数、函数指针、函数对象、lambda表达式等）。它通过 **类型擦除（Type Erasure）** 机制，屏蔽不同可调用类型的差异，提供统一的接口。

- **声明形式**：`std::function<返回类型(参数类型列表)>`
- 例：`std::function<int(int, int&)>` 表示存储一个返回`int`，接受`int`和`int&`两个参数的可调用对象。

##### 2. 原理与推导：类型擦除与兼容性准则

`std::function` 内部通常使用一个模板化的派生类继承自一个非模板的基类，将具体类型信息“擦除”在基类接口之后。关键在于其 **绑定兼容性准则**，该准则源于调用安全性：

> `function` 参数必须比实际参数 **更严格**，`function` 返回值必须比实际返回值 **更宽松**。

- **逻辑推导**：
  1. **参数准则**：所有能被 `std::function` 接受的参数，都必须能传递给实际的可调用对象。因此，`function` 上声明的参数类型必须隐含地向实际参数类型转换是安全的。通常，这意味着 `function` 的参数类型（接受端）不能比实际参数（要求端）更“宽松”。
      - **更严格** 的含义：例如，接受 `int`、`const int&` 比接受 `int&` 更严格，因为 `int` 和 `const int&` 能接收右值或常量，而 `int&` 只能绑定左值。
  2. **返回值准则**：所有实际可调用对象可能返回的值，都必须能被 `std::function` 的返回类型所包容。因此，`function` 的返回类型不能比实际返回类型更“严格”。
      - **更宽松** 的含义：例如，实际返回 `int&`，`function` 返回 `int` 是安全的（可以拷贝引用），反之则不安全（可能绑定到临时对象）。

- **绑定优先级（严格程度）**：
根据PPT第122页，对于参数绑定而言，其严格程度从高到低（从松到严的理解顺序相反）遵循：
  - **左值**（`T&`） < **常量左值**（`const T&`） < **常量右值**（`const T&&`） < **右值**（`T&&`）
  - 不带引用/const的普通值类型（如`int`）在参数传递中，其行为最宽松，可包容左值、右值等多种情况。在准则中，它比带引用的类型更“严格”（此处“严格”指对传入`function`的实参限制更严，更安全）。

##### 3. 【解题步骤】`std::function` 兼容性判断
用于判断 `function<Fx(Args...)>` 能否绑定可调用对象 `Callable`。

1.  **提取类型**：明确 `Callable` 的实际参数类型 `P` 和返回值类型 `R`，以及 `std::function` 声明的参数类型 `FArgs` 和返回值类型 `FRet`。
2.  **检查参数**：对每一对 `FArg`（来自 `FArgs`） 和 `PArg`（来自 `P`），必须满足：`FArg` 能安全地传递给 `PArg`。根据上述原则，`FArg` 应比 `PArg` 更严格。可查询绑定优先级表。
3.  **检查返回值**：必须满足：`R` 能安全地赋值/转换为 `FRet`。根据原则，`FRet` 应比 `R` 更宽松。
4.  **特别例外**：当实际参数类型为 `T&&`（右值引用）时，`std::function` 的参数可以为 `T`。因为 `std::function` 内部会拷贝实参生成一个左值副本，然后以右值形式传给实际函数。这是唯一允许 `function` 参数比实际参数宽松的情况。
5.  **结论**：参数与返回值均满足上述要求，则可编译通过。

##### 4. 【易错点】

| 易混概念 | 辨析 | 正例/反例 |
| :--- | :--- | :--- |
| **函数声明 vs `std::function` 包装** | 对于函数 `int func(const int& x, int& b)`，调用时 `func(int_val, int_ref)` 是合法的。但在 `std::function` 包装时，必须严格遵循兼容性准则。 | 允许：`function<int(int, int&)>`<br/>不允许：`function<int(int, int)>` （第二参数 `int` 比 `int&` 宽松） |
| **`function` 参数 `T` vs `T&`** | `function` 用 `T&` 比 `T` 更宽松（限制少），因为 `T&` 只能接左值，而 `T` 能接左值和右值。因此，在准则中 `T` 比 `T&` 更严格。 | 实际参数为 `int&`，则 `function` 参数可为 `int&` 或 `int`，但若实际参数为 `int`，`function` 参数为 `int&` 则可能失败。 |
| **`function` 返回值 `T` vs `T&`** | `function` 用 `T` 比 `T&` 更宽松（包容性强），因为返回 `T` 时，无论实际返回 `T` 还是 `T&` 都可以拷贝。返回 `T&` 更严格。 | 实际返回 `int&`，`function` 返回 `int` （正确） 或 `int&` （正确）。实际返回 `int`，`function` 返回 `int&` 则错误。 |
| **右值参数（`T&&`）的例外** | 当实际参数为 `T&&` 时，准则存在例外，允许 `std::function` 参数为 `T`。其内部通过拷贝参数来安全地生成右值。 | `int& operator()(int&&)` 可被 `function<int&(int)>` 包装。 |

##### 5. 例题

**例题**：来自课堂拓展自学题 [L11-iostream&函数对象和智能指针-2026.pptx P112-P118]

**题目**：给定函数 `int func1(const int &x, int &b) {...}`，对以下 `std::function` 声明进行判断，哪些能成功编译并绑定 `func1`？
A. `function<int(int, int)> pf1 = func1;`
B. `function<int(int, int&)> pf1 = func1;`
C. `function<int(int&, int&)> pf1 = func1;`
D. `function<int(const int&, int&)> pf1 = func1;`
E. `function<int(const int, int)> pf1 = func1;`
F. `function<int&(int, int&)> pf1 = func1;`

**思路**：应用 `std::function` 兼容性准则。`func1` 实际参数为 `(const int&, int&)`，返回值为 `int`。逐项检查每个选项的参数对和返回值对是否满足“function参数更严格，function返回值更宽松”。

**解答**：
1.  **A.** 第二参数：`function` 的 `int` 比实际的 `int&` 更宽松，不满足参数准则。 ×
2.  **B.** 第一参数：`int` 和 `const int&` 范围一致（都可靠接右值/左值）。第二参数：`int&` 和 `int&` 一致。返回值：`int` 和 `int` 一致。所有条件满足。 **√**
3.  **C.** 第一参数：`function` 的 `int&` 比实际的 `const int&` 更严格（`int&` 只能绑左值），满足准则。第二参数：匹配。返回值匹配。 **√**
4.  **D.** 所有参数类型与返回值类型完全一致。 **√**
5.  **E.** 第二参数：`function` 的 `int` 比实际的 `int&` 更宽松，不满足参数准则。 ×
6.  **F.** 返回值：`function` 的 `int&` 比实际的 `int` 更严格（`function` 返回引用，但实际可能返回临时量），不满足返回值准则。 ×

**技巧**：判断 `std::function` 绑定的口诀：`function`参数的接口要“吝啬”（限制严），`function`返回值的接口要“慷慨”（包容性强）。遇到右值引用参数，立即想到内部拷贝例外。

【定义】
在C++的继承体系中，当通过值传递（pass-by-value）或赋值操作将派生类对象交给基类对象时，会发生**对象切片**。基类对象仅能持有派生类对象中属于基类的那部分数据成员，派生类新增的数据成员和行为（成员函数）会被“切掉”丢失。

原理与推导：
对象切片发生的根本原因在于C++的**值语义**。基类对象在栈或堆上有固定的大小（`sizeof(基类)`），它没有额外的内存空间来存储派生类新增的成员。因此，无论是函数传参（拷贝构造）还是对象赋值（拷贝赋值运算符），都只能复制基类部分，形成一个新的、完整的基类对象，而非指向派生类对象的引用。

这一机制也解释了**早绑定**的行为。当对象被切片后，其静态类型和动态类型都是基类，编译器在编译期就直接将函数调用与基类的实现绑定，因此即使派生类重写了同名函数，通过切片后的基类对象调用的也永远是基类版本。

【解题步骤】
1. 分析对象传递或赋值方式：是值传递/赋值，还是引用/指针传递。
2. 如果是值传递/赋值，则会产生活动对象切片，派生类特有部分丢失。
3. 切片后的行为：调用成员函数时发生早绑定，执行基类版本；只能访问基类成员。

【易错点】
| 概念 | 值传递/赋值（对象切片） | 引用/指针传递（向上转型） |
| :--- | :--- | :--- |
| **是否创建新对象** | 是（创建一个新的基类对象） | 否（仅引用或指向原派生类对象） |
| **派生类数据** | 丢失（只保留基类部分） | 保留（但通过基类指针/引用无法直接访问） |
| **函数绑定方式** | **早绑定**（调用基类函数） | 非虚函数**早绑定**（调用基类函数）<br>虚函数**晚绑定**（根据实际类型调用派生类函数） |
| **本质原因** | 内存限制，基类对象无额外空间 | 没有创建新对象，保持了对象的完整性 |

**例题**：
**题目**：阅读以下代码，写出其运行结果。 [L8-虚函数-2026.pptx P10]
```cpp
#include <iostream>
using namespace std;

class Pet { 
public:  
  void name(){ cout << "Pet::name()" << endl; }
};
class Dog: public Pet { 
public:  
  void name(){ cout << "Dog::name()" << endl; }
};
void getName(Pet p){
    p.name();
}
int main() {
  Dog g;
  g.name();	
  getName(g); 	
  Pet p = g;
  p.name();   
  return 0;
}
```
**思路**：分析三处 `name()` 调用的对象及其绑定方式。`g.name()` 直接调用 `Dog` 对象，输出其版本。`getName(g)` 通过**值传递**，发生对象切片，`p`是全新的 `Pet` 对象。`p = g` 是**值赋值**，同样发生对象切片。因此后两处调用的都是 `Pet` 对象的 `name()` 函数，发生早绑定。
**解答**：
1. `g.name()`：`g` 是 `Dog` 类型，调用 `Dog::name()`。
2. `getName(g)`：传参时发生切片，局部参数 `p` 是 `Pet` 类型，调用 `Pet::name()`。
3. `p.name()`：`p` 是 `Pet` 类型对象，已通过赋值切片，调用 `Pet::name()`。
最终输出：
```
Dog::name()
Pet::name()
Pet::name()
```
**技巧**：判断是否为对象切片，直接看函数参数或赋值目标是否为**基类对象本身**（非引用、非指针）。只要不是指针/引用，切片就必然发生。

【定义】
在使用 `fstream` 读取文件时，有多种策略可以遍历文件内容，包括逐词读取、逐行读取、按分隔符读取等。同时，流的状态位（`goodbit`, `eofbit`, `failbit`, `badbit`) 管理着读取操作的生命周期和正确性。

原理与推导：
文件流的操作基于 `ios_base` 中定义的状态位。
- `good()`：当 `goodbit` 被设置且无任何错误位被设置时为 `true`。
- `eof()`：当读取操作到达文件末尾（EOF）时，`eofbit` 和 `failbit` 被设置。
- `fail()`：当输入操作失败（如类型不匹配，或遇到 EOF 时尝试读取），但流缓冲区未损坏时，`failbit` 被设置。
- `bad()`：当流缓冲区发生严重 I/O 错误（如物理磁盘损坏）时，`badbit` 被设置。

常见的读取策略是将流对象本身作为布尔条件进行判断，这依赖于 `operator bool()`，它通常在 `!fail()` 时返回 `true`。因此，循环 `while (stream >> data)` 在成功读取时继续，遇到 EOF 或类型不匹配失败时退出，这比直接检查 `while (!stream.eof())` 更安全。

【解题步骤】
1. 选择合适的读取函数 (`>>`， `getline()`， `read()`)。
2. 设计读取循环。推荐将读取操作直接作为循环条件，以便在每次读取后立即检查状态。
3. 处理循环结束后可能的状态（是正常EOF，还是格式错误），并据此执行后续逻辑。

【易错点】
| 错误/陷阱 | 正确做法与解释 |
| :--- | :--- |
| **错误写法**：`while (!file.eof()) { file >> data; ... }` | **正确写法**：`while (file >> data) { ... }` <br> **原因**：`eof()` 只有在**尝试读取但越过文件末尾**后才为 `true`。在错误写法中，读取到文件最后一个数据后，`eof()` 仍然为 `false`，循环会再次进入，执行一次无效的读取，导致最后一个数据被处理两次。而将读取操作放在循环条件中，可以在每次读取后立即检查其有效性。 |
| 混用 `>>` 和 `getline()` 导致读取遗漏 | 使用 `>>` 读取后，换行符会留在输入缓冲区。紧接着的 `getline()` 会立即读取到这个空行。应在混合使用前，通过 `cin.ignore()` 清除缓冲区中的换行符。 |

**例题**：
**题目**：请说明使用 `while(!file.eof())` 循环进行文件读取可能存在的问题，并给出更健壮的写法。 [结合 L8-虚函数-2026.pptx 知识，自编]
**思路**：阐述 `eofbit` 的设置时机。说明在最后一次成功读取后，文件指针指向末尾，`eof()` 并未立即触发，导致多执行一次循环体。这对应了值语义中的“状态滞后”问题。
**解答**：
`eofbit` 不是在读到最后一个字节时设置，而是在**读不到内容时**才设置。当读取完最后一个有效数据后，流状态仍是 `good`，`eof()` 返回 `false`。循环继续执行，下一次读取失败，此时 `eof()` 才变为 `true`，但上一次读取的数据可能被再次处理。
更健壮的写法是将读取操作作为循环条件：
```cpp
int data;
while (file >> data) { // 读取成功才进入循环体
    // 处理data
}
```
此写法在每次读取后立即检查流状态，避免了最后一条数据的重复处理。
**技巧**：在文件读取循环中，始终将“读取操作”作为循环的守卫条件，而不是先检查状态再读取。

### 六、核心机制深度回顾

#### 重载（Overload）、重写覆盖（Override）与重写隐藏（Redefining）辨析 【必背】

【定义】
| 概念 | 英文 | 定义 | 发生条件 |
| :--- | :--- | :--- | :--- |
| **重载** | Overload | 在**同一作用域**内，函数名相同但**参数列表不同**（类型、个数或顺序）的多个函数。 | 同一个类内部，或同一个命名空间。 |
| **重写覆盖** | Override | 在派生类中，重写基类的**虚函数**。函数签名（函数名、参数列表、const属性）必须完全一致。 | 跨作用域（基类与派生类），基类函数必须有 `virtual` 关键字。 |
| **重写隐藏** | Redefining | 在派生类中，定义了一个与基类**非虚函数**同名的成员函数。基类的那个同名函数在派生类中被“隐藏”。 | 跨作用域（基类与派生类），基类函数**没有** `virtual` 关键字。 |

原理与推导：
- **重载**是编译时多态，编译器通过参数类型在编译期决定调用哪个函数。
- **重写覆盖**是运行时多态，通过虚函数表机制实现晚绑定，根据对象的实际类型调用函数。
- **重写隐藏**发生在没有 `virtual` 时，派生类的同名函数会将基类的所有同名重载版本都隐藏起来。通过派生类对象或指针调用该函数时，只会考虑派生类作用域的函数，导致基类版本不可见（除非使用 `using Base::func;` 或显式指定作用域 `Base::func()`）。

【解题步骤】
1. 确认函数所处的作用域：是否跨基类/派生类。
2. 确认函数签名是否一致（函数名、参数、`const`）。
3. 确认基类函数是否声明为 `virtual`。
   - 若为同一作用域且参数不同 -> **重载**。
   - 若跨作用域、签名一致、基类为 `virtual` -> **重写覆盖**。
   - 若跨作用域、基类非 `virtual` -> **重写隐藏**。

【易错点】
| 易混点 | 说明 |
| :--- | :--- |
| 隐藏会隐藏所有重载版本 | 若派生类定义了 `void func(int)`，即使基类有 `void func()` 和 `void func(double)`，它们在派生类中也会被全部隐藏，无法直接通过派生类对象调用。 |
| `const` 影响签名 | `void func()` 和 `void func() const` 是不同的函数签名。如果基类是 `virtual void func() const`，而派生类写成了 `void func()`，则这不是重写覆盖，而是重写隐藏。应使用 `override` 关键字来让编译器检查。 |
| 派生类可加或不加 `virtual` | 一旦基类声明为 `virtual`，派生类中对该函数的重写即使不写 `virtual`，它依然是虚函数。但建议始终加上 `override` 关键字以提高可读性和安全性。 |
| 指针 vs. 对象的绑定差异 | 重写覆盖（晚绑定）仅对**指针和引用**有效。对于对象本身（值语义），即使函数是虚函数，也会因为对象切片而发生早绑定，调用基类版本。 |

**例题**：
**题目**：阅读以下代码，说明三个 `play()` 调用的结果，并解释原因。 [L8-虚函数-2026.pptx P17, P18]
```cpp
class Instrument {
public:
  virtual void play() { cout << "Instrument::play" << endl; }
};
class Wind : public Instrument {
public:
  void play() { cout << "Wind::play" << endl; }
};

void tune(Instrument& ins) { ins.play(); }  // 引用传递
void tuneByVal(Instrument ins) { ins.play(); } // 值传递

int main() {
  Wind flute;
  tune(flute);       // 调用1
  tuneByVal(flute);  // 调用2
  Instrument inst = flute;
  inst.play();       // 调用3
}
```
**思路**：辨别三种情况下的绑定机制。调用1是引用传递，触发虚函数晚绑定。调用2和3都发生了对象切片，早绑定到基类的 `play()`。
**解答**：
- **调用1 (`tune(flute)`)** ：参数是 `Instrument&`，传递引用，未发生切片。`play()` 是虚函数，编译器进行**晚绑定**，根据 `ins` 引用的实际对象类型（`Wind`）调用 `Wind::play()`，输出 `Wind::play`。
- **调用2 (`tuneByVal(flute)`)** ：参数是 `Instrument` 对象，传值时发生**对象切片**，`ins` 是一个全新的 `Instrument` 对象。此时无论 `play()` 是否为虚函数，都进行**早绑定**，调用 `Instrument::play()`，输出 `Instrument::play`。
- **调用3 (`inst.play()`)** ：`inst` 是 `Instrument` 对象，通过赋值发生切片。同样是**早绑定**，调用 `Instrument::play()`，输出 `Instrument::play`。
**技巧**：多态的“钥匙”是指针和引用，它们避免了对象切片，从而保留了对象的完整类型信息（vptr），使得晚绑定成为可能。

#### 构造函数与析构函数的虚函数行为 【高频】

【定义】
- **构造函数不能是虚函数**。
- **析构函数**在处理多态继承时，**基类的析构函数通常应该声明为虚函数**，以保证正确析构。
- 在构造和析构期间，虚函数机制的行为是“单向”的：只能调用当前正在构建或析构的那个类及其基类的版本。

原理与推导：
1.  **为何构造函数不能是虚函数**：
    - 虚函数机制依赖**虚函数指针（vptr）**，而 vptr 是在构造函数的执行过程中逐步初始化的。
    - 在调用构造函数之前，对象的内存已经分配，但 vptr 尚未指向正确的虚函数表（通常会先指向基类的虚函数表）。
    - 如果构造函数是虚函数，那么为了调用它，就需要通过 vptr 去查找，此时 vptr 又没初始化好，形成“先有鸡还是先有蛋”的悖论。因此语法上禁止。

2.  **为何析构函数推荐为虚函数**：
    - 若通过 `Base* ptr = new Derived();` 管理对象，最终执行 `delete ptr;`。
    - 如果基类析构函数不是虚函数，则 `delete` 只会调用基类的析构函数，派生类中动态分配的资源将无法释放，造成**内存泄漏**。
    - 将基类析构函数声明为虚函数后，`delete` 会通过 vptr 找到派生类的虚函数表，先调用派生类的析构函数，再逐级向上调用基类析构函数，实现完全析构。

3.  **构造/析构期间的虚函数调用**：
    - 当基类构造函数正在执行时，派生类部分尚未初始化。如果此时调用虚函数并期望它能多态地调用派生类版本，就会访问到未初始化的成员，导致未定义行为。因此，C++标准规定，在此阶段会强制将虚函数“降级”为当前类的版本进行静态调用。
    - 同理，在析构函数中，派生类部分已被销毁，虚函数调用也会退化为当前类的版本。

【易错点】
- **内存泄漏**：当设计一个可能被继承的类时，如果忘记将析构函数声明为虚函数，通过基类指针 `delete` 派生类对象会导致部分资源泄露。**经验法则**：只要一个类包含虚函数（表明它被设计为多态基类），其析构函数就应该声明为虚函数。
- **纯虚析构函数**也必须提供定义（函数体），因为派生类析构时会最终调用它。

**例题**：
**题目**：请指出以下代码存在的问题，并说明后果。 [结合 L8-虚函数-2026.pptx P11 知识，自编]
```cpp
class Base {
public:
    Base() { foo(); } // 构造函数中调用虚函数
    virtual void foo() { cout << "Base::foo" << endl; }
    ~Base() {}        // 非虚析构函数！
};

class Derived : public Base {
    int* data;
public:
    Derived() : data(new int[100]) {}
    virtual void foo() override { cout << "Derived::foo" << endl; }
    ~Derived() { delete[] data; }
};

int main() {
    Base* ptr = new Derived();
    delete ptr; 
    return 0;
}
```
**思路**：识别两个关键错误：构造函数内的虚函数调用行为，和非虚析构函数导致的内存泄漏。
**解答**：
1. **构造函数中的虚函数调用**：`Base` 构造函数中调用了 `foo()`。虽然 `foo` 是虚函数，但在构造 `Derived` 对象时，`Base` 的构造函数先执行。此时 `Derived` 部分尚未初始化，因此虚函数调用不会分派到 `Derived::foo`，而是调用 `Base::foo`。输出 `Base::foo`，而不是期望的 `Derived::foo`。
2. **非虚析构函数导致内存泄漏**：`delete ptr` 时，`ptr` 的静态类型是 `Base*`，而 `~Base()` 不是虚函数。因此，编译器进行静态绑定，只调用了 `Base` 的析构函数。`Derived` 的析构函数未被调用，其构造函数中分配的 `data` 数组（`new int[100]`）无法被释放，发生**内存泄漏**。
**技巧**：只要一个类被设计为多态基类（拥有虚函数），就**必须**将其析构函数声明为虚函数。

#### 纯虚函数、抽象类与纯虚析构函数 【重点】

【定义】
- **纯虚函数**：在基类中声明但只给出接口而不提供实现（或可选择提供默认实现）的虚函数。声明格式为在函数签名后加上 `= 0`。
  ```cpp
  virtual ReturnType FuncName(ArgList) = 0;
  ```
- **抽象类**：包含至少一个纯虚函数的类。抽象类**不能实例化**对象，只能作为基类为派生类提供接口规范。
- **纯虚析构函数**：声明为 `virtual ~ClassName() = 0;` 的析构函数。它使得类成为抽象类，但**必须**在类外提供定义（函数体），因为派生类析构时会最终调用基类的析构函数。

原理与推导：
抽象类定义了一套派生类必须遵守的“合同”或“接口”。任何继承自抽象类的派生类，如果没有重写覆盖（实现）所有的纯虚函数，那么它自身也是一个抽象类，仍然无法实例化。这强制派生类实现特定的行为，是实现运行时多态的重要基石。

纯虚析构函数是一种特殊的设计模式：当一个类没有其他适合作为纯虚函数的成员函数，但设计者又希望这个类是抽象类时，可以将析构函数声明为纯虚函数。但必须牢记，析构链的最末端需要实际执行基类的析构逻辑，因此语法强制要求为其提供函数体。

【解题步骤】
1. 检查类中是否有 `= 0` 声明的函数。
2. 若有，则该类是抽象类，禁止创建对象。
3. 检查其派生类：是否实现了所有的纯虚函数？若否，派生类仍是抽象类。
4. 如果声明了纯虚析构函数，检查是否在类外提供了定义体（通常为空 `{}`）。

【易错点】
| 易错点 | 说明 |
| :--- | :--- |
| **纯虚函数可以有实现** | `= 0` 语法仅仅表示该类是抽象的，并不意味着该函数不能有函数体。可以在类外为纯虚函数提供定义，派生类可以通过显式作用域调用 `Base::func()` 来使用这个默认实现。 |
| **纯虚析构必须定义** | 声明 `virtual ~Base() = 0;` 后，**必须**在类外写 `Base::~Base() {}`。否则在链接阶段会报错，因为派生类析构时找不到基类析构函数的具体代码。 |
| **派生类不实现全部纯虚函数会导致自身也抽象** | 忘记重写覆盖任何一个继承来的纯虚函数，都会导致派生类仍然是抽象类，使其无法被实例化。C++11 的 `override` 关键字可以帮助检查签名是否匹配，避免因签名错误导致未正确覆盖。 |

**例题**：
**题目**：以下关于纯虚函数和抽象类的代码存在哪些问题？请指出并修正。 [结合 L8-虚函数-2026.pptx 知识点，自编]
```cpp
class AbstractBase {
public:
    virtual void Interface() = 0;
    virtual ~AbstractBase() = 0; // 纯虚析构函数
};

// 错误1：忘记为纯虚析构函数提供定义

class Concrete : public AbstractBase {
public:
    void Interface() { /* ... */ }
};

int main() {
    AbstractBase obj; // 错误2：试图实例化抽象类
    Concrete con;     // 错误3：因第一个错误导致链接失败
    return 0;
}
```
**思路**：逐行分析代码中的语法/语义违规。纯虚析构函数缺少定义。抽象类被实例化。`Concrete` 本身定义正确，但因基类析构未定义而无法成功链接。
**解答**：
存在两个主要问题：
1. **纯虚析构函数未定义**：`AbstractBase` 声明了 `virtual ~AbstractBase() = 0;`，但没有在任何地方提供实现。必须在类外添加定义：
   ```cpp
   AbstractBase::~AbstractBase() {} // 提供空实现
   ```
2. **抽象类被实例化**：`AbstractBase obj;` 试图创建一个抽象类的对象，这在 C++ 中是被禁止的，编译器会直接报错。应删除此行或改为指针/引用。
修正后，`Concrete con;` 的构造才能成功链接，因为其析构时会正确调用已定义的 `AbstractBase::~AbstractBase()`。
**技巧**：当你需要将一个类声明为抽象类，但又没有合适的成员函数可以作为纯虚函数时，一个常用技巧就是将析构函数声明为纯虚函数。但**切记**：这是“定义”的第一步，第二步必须是**在类外提供该析构函数的函数体**。

##### 1. 三个概念的定义

**【定义】重载（Overload）**
在**同一作用域**（同一个类内，或同为全局函数），函数名相同，但**参数列表（类型或个数）必须不同**，返回值可以相同或不同。重载与虚函数无关，编译器通过参数类型在编译期决议。

**【定义】重写覆盖（Override）**
派生类重新定义基类中的**虚函数**，函数名、参数列表、返回值（一般情况下）**必须完全相同**。派生类虚函数表中原本指向基类虚函数的位置会被**覆盖**为派生类版本，从而通过基类指针/引用调用时实现多态。

**【定义】重写隐藏（Redefining）**
派生类定义了与基类**同名的函数**，但满足下列条件之一：
- 基类同名函数**不是虚函数**；
- 即使基类函数是虚函数，但**参数列表不同**。
此时派生类的函数把基类中**所有同名函数**（无论参数是否相同）都隐藏起来，在派生类对象上无法直接调用基类的同名函数，只能通过 `基类::函数名` 显式调用。

##### 2. 对比表

| 特性 | 重载 (Overload) | 重写覆盖 (Override) | 重写隐藏 (Redefining) |
|------|----------------|---------------------|----------------------|
| 作用域 | 同一作用域（同一个类/全局） | 基类-派生类之间 | 基类-派生类之间 |
| 基类函数是否为虚 | 无要求 | 必须是虚函数 | 不是虚函数，或参数不同 |
| 函数名 | 必须相同 | 必须相同 | 必须相同 |
| 参数列表 | 必须不同 | 必须相同（协变返回值除外） | 必须不同，或基类非虚 |
| 虚函数表影响 | 无关 | 派生类虚表中对应条目被覆盖 | 不覆盖，基类虚表项保留 |
| 派生类对象能否直接调用基类版本 | 视作用域而定 | 被隐藏，需 `Base::foo()` | 被隐藏，需 `Base::foo()` |
| 决议方式 | 编译期（静态绑定） | 运行期（动态绑定） | 编译期（静态绑定） |

##### 3. 原理与推导
- **重写覆盖改变虚函数表**：当派生类重写基类虚函数时，编译器将派生类虚函数表（vtable）中相应的条目替换为派生类函数地址；因此通过基类指针调用时，会查表找到派生类的版本。
- **重写隐藏不改变虚函数表**：隐藏发生时，派生类同名函数与基类的虚函数表无关，它只是在派生类作用域中“屏蔽”了基类的名字。基类的虚函数表中原有条目仍然指向基类版本；但如果用基类指针调用，依然可能调用到基类版本（除非派生类的同名非虚函数参数完全不同，且未覆盖虚表）。
- **隐蔽规则额外提醒**：一旦派生类中出现与基类同名的任何成员函数，基类中**所有同名函数**（无论是否为虚、无论参数）均在派生类作用域被隐藏，除非使用 `using` 声明把它们引入派生类作用域。

##### 4. 【解题步骤】如何识别重载、覆盖、隐藏
1. 检查作用域：如果在**同一个类**或同为自由函数，且函数名相同、参数不同 → 重载。
2. 如果在派生类和基类之间，检查基类同名函数是否为 `virtual`：
   - 若**不是**虚拟函数 → 一定为重写隐藏（即使参数相同）。
   - 若是虚拟函数，检查参数列表是否完全相同：
     - 参数**完全相同** → **重写覆盖**。
     - 参数**不同** → **重写隐藏**（即使基类函数是虚函数）。
3. 分析代码时，注意“隐藏”会导致派生类对象无法直接调用基类同名函数，而要通过 `Base::func()` 或基类指针/引用。

##### 5. 【易错点】
- **误将隐藏当作覆盖**：基类有虚函数，但派生类写错了参数类型（例如 `foo(float)` 覆写 `foo(int)`），这实际是重写隐藏，不会产生多态。使用 `override` 关键字可以让编译器主动报错。
- **隐藏的“全部屏蔽”效应**：在派生类中定义了与基类同名的函数后，基类所有同名重载版本都被隐藏。如果希望保留部分重载，需要用 `using Base::foo;` 引入。
- **返回值协变是覆盖的特例**：如果基类虚函数返回 `Base*`，派生类重写返回 `Derived*`，仍属于重写覆盖（协变返回类型），但覆盖条件仍要求参数相同。

##### 6. 例题

**例题**：辨析覆盖与隐藏，并测试 `override` 关键字效果  
**题目**：[L8-虚函数-2026.pptx P34-P37]
```cpp
class Base {
public:
    virtual void foo() { cout << "Base::foo()" << endl; }
    virtual void foo(int) { cout << "Base::foo(int)" << endl; }
    void bar() {}
};

class Derived1 : public Base {
public:
    void foo(int) { cout << "Derived1::foo(int)" << endl; }   // ①
};

class Derived2 : public Base {
public:
    void foo(float) { cout << "Derived2::foo(float)" << endl; } // ②
};

class Derived3 : public Base {
public:
    void foo(int) override { cout << "Derived3::foo(int)" << endl; } // ③
    // void foo(float) override {}  // 编译错误：参数不同不是重写
};
```
以及 main 中的调用：
```cpp
Derived1 d1; Derived2 d2;
Base* p1 = &d1; Base* p2 = &d2;
p1->foo();          // ④
p2->foo();          // ⑤
d1.foo(3);          // ⑥
d2.foo(3.0);        // ⑦
p1->foo(3);         // ⑧
p2->foo(3.0);       // ⑨
```
请判断 ①–③ 各属于重载、覆盖、隐藏中的哪一种？写出 ④–⑨ 的输出结果。

**思路**：先根据基类函数是否虚、参数是否相同判定①、②、③的性质，再结合虚函数表与隐藏规则分析多态调用和直接调用的行为。注意隐藏会导致派生类对象无法直接调用基类同名的无参版本 `foo()`。

**解答**：
- ① 基类 `foo(int)` 是虚函数，派生类参数完全相同 → **重写覆盖**。
- ② 基类 `foo(int)` 是虚函数，但派生类参数为 `float`，不同 → **重写隐藏**。且派生类没有定义其它 `foo`，所以隐藏了基类的所有 `foo`。
- ③ 使用 `override` 显式声明，与①相同 → 重写覆盖。

运行时：
- ④ `p1->foo()`：由于 `Derived1` 重写覆盖了 `foo(int)` 而未定义 `foo()`，虚函数表里 `foo()` 仍然是继承自 `Base` 的版本，所以输出 `Base::foo()`。
- ⑤ `p2->foo()`：`Derived2` 隐藏了基类所有 `foo`，但其对象内的虚函数表仍然保存了基类 `foo()` 的地址，通过基类指针调用 `foo()` 会查表得到 `Base::foo()`，输出 `Base::foo()`。
- ⑥ `d1.foo(3)`：直接调用派生类自己的 `foo(int)`，输出 `Derived1::foo(int)`。
- ⑦ `d2.foo(3.0)`：调用派生类的 `foo(float)`，输出 `Derived2::foo(float)`。
- ⑧ `p1->foo(3)`：基类指针调用虚函数，由于 `Derived1` 重写覆盖了 `foo(int)`，虚表中指向 `Derived1::foo(int)`，输出 `Derived1::foo(int)`。
- ⑨ `p2->foo(3.0)`：实参 `3.0` 是 `double`，但被隐式转换为 `int` 以匹配 `Base::foo(int)`；`Derived2` 并未覆盖虚表中 `foo(int)` 的条目，故调用 `Base::foo(int)`，输出 `Base::foo(int)`。

**技巧**：`override` 仅用于检查，不改变运行结果；判断隐藏/覆盖关键看“参数+virtual”是否完全匹配。基类指针访问虚函数只看虚表，不受派生类作用域隐藏影响。

##### 1. 虚函数表（VPTR）与对象内存布局（预备）

**【定义】** 每个含有虚函数的类或派生类，编译器为其生成一个**虚函数表（vtable）**，其中存放该类所有虚函数的地址。每个对象内部隐含一个指向该表的指针 **VPTR**（通常占一个机器字长）。
**【必有】** 对象大小 = 非静态数据成员大小（加上内存对齐） + 一个 VPTR 的大小（若有虚函数）。多个虚函数只增加虚表条目，不增加对象大小，因为对象只存一个指针。

**示例**：[L8-虚函数-2026.pptx P21-P22]
```cpp
class NoVirtual  { int a; };                    // sizeof = 4（仅int）
class OneVirtual { int a; virtual void f1(); }; // sizeof = 12（int 4 + VPTR 8)
class TwoVirtual { int a; virtual void f1(); virtual int f2(); }; // sizeof = 12
```
在 64 位系统 `void*` 为 8 字节，加上对齐，出现 12 的结果。

##### 2. 构造函数与虚函数

**【定义】构造函数不能是虚函数，也不必是虚函数。**
- **不能**：创建对象时需要知道对象的完整类型以设置 VPTR，而 VPTR 的初始化发生在构造函数执行前（由编译器插入），若允许构造函数为虚，就需要在 VPTR 设置之前通过虚机制调度，这形成逻辑悖论。
- **不必**：构造函数的作用是初始化该具体类的数据成员，调用时类的类型是明确的，不需要多态。

**【核心规则】在构造函数（和析构函数）中调用虚函数时，虚机制不工作，实际调用的是当前类的“本地版本”。**
- **原理**：C++ 中基类子对象先于派生类部分初始化。执行基类构造函数时，派生类的数据成员尚未初始化，对象的动态类型尚不完整。为保证安全，编译器将在构造/析构期间将对象的动态类型“锁定”为正在构造/析构的类，因此虚函数调用被静态绑定到当前类的版本。
- **例子**：[L8-虚函数-2026.pptx P24]
```cpp
class Base {
public:
    virtual void foo() { cout << "Base::foo" << endl; }
    Base() { foo(); }   // 构造函数内调用虚函数
};
class Derived : public Base {
    int _num;
public:
    void foo() override { cout << "Derived::foo" << _num << endl; }
    Derived(int j) : _num(j) {}
};
```
执行 `Derived d(0);` 时，先执行 `Base` 构造，此时 `Derived::_num` 尚未初始化。`Base()` 中的 `foo()` 调用的是 `Base::foo`，而不是派生类版本——避免访问未初始化的 `_num`。

##### 3. 析构函数与虚函数

**【定义】析构函数可以是虚函数，且常常应该是虚函数。虚析构函数仍必须提供函数体。**
- **为什么要是虚函数**：当通过基类指针 `delete` 一个派生类对象时，如果基类析构函数不是虚函数，则只会调用基类的析构函数，派生类部分无法正确析构，造成资源泄漏。
- **示例**：[L8-虚函数-2026.pptx P27-P28]
```cpp
class Base1 { public: ~Base1() { cout << "~Base1()\n"; } };          // 非虚析构
class Derived1 : public Base1 { public: ~Derived1() { cout << "~Derived1()\n"; } };

class Base2 { public: virtual ~Base2() { cout << "~Base2()\n"; } };   // 虚析构
class Derived2 : public Base2 { public: ~Derived2() { cout << "~Derived2()\n"; } };
```
测试：
```cpp
Base1* bp = new Derived1; delete bp;   // 只输出 ~Base1()，Derived1 析构未调用
Base2* b2p = new Derived2; delete b2p; // 输出 ~Derived2() 再 ~Base2()
```
- **在析构函数中调用虚函数**：与构造函数类似，虚机制不工作。析构时派生类部分已先析构，对象的动态类型变为当前析构的类，故虚调用被静态绑定到当前类的版本，防止访问已销毁的派生类成员。

**【重要原则】总是将基类的析构函数声明为虚析构函数**，除非确定该类不会被用作多态基类。

##### 4. 【解题步骤】分析构造/析构中虚函数调用
1. 判断构造或析构执行时对象的“当前类”。
2. 记住虚机制在构造/析构中失效，调用的版本就是当前类中定义的函数（如果有），若无则按普通编译查找。
3. 析构顺序与构造相反：先析构派生类自身部分，然后成员对象，最后基类部分。在派生类析构函数体执行后，基类析构中不能再依赖派生类成员。
4. 对于 `delete` 基类指针，检查基类析构是否为虚，决定是否完整清理派生类资源。

- **构造函数误用为虚函数**：编译错误或概念错误。
- **忘记将基类析构函数设为虚函数**：导致 `delete` 基类指针时派生类析构未执行 → 内存泄漏（常见于多态容器）。
- **认为构造/析构中虚函数会多态**：实际上它们调用的是“当前类”版本，可能产生非预期结果，尤其当派生类重写了虚函数并依赖派生类成员时，会出现未初始化或已销毁成员的使用风险。
- **多重继承时构造顺序与虚表切换**：更复杂，但基本原则不变——每个基类子对象构造期间对象的动态类型为该基类。

**例题**：虚析构的必要性辨析  
**题目**：[L8-虚函数-2026.pptx P29 选择题改编]
下列关于虚函数的说法不正确的是（ ）。  
A. 构造函数不可以是虚函数  
B. 析构函数可以是虚函数  
C. 类中的虚函数可能并不显式地带有 virtual 声明  
D. 一个类中的虚函数越多，其对象的大小就越大

**思路**：逐一分析选项，结合 VPTR 原理和虚函数表知识。C 项因为派生类重写虚函数时 virtual 可省略但仍为虚；D 项多个虚函数只增加虚表条目，对象大小不变。

**解答**：不正确的是 **D**。  
- A 正确，构造函数不能是虚。  
- B 正确，析构函数可以是虚，且应该是虚。  
- C 正确，派生类重写基类虚函数时可不写 `virtual` 关键字，但函数仍是虚函数（继承虚属性）。  
- D 错误，带虚函数的对象只多出一个 VPTR 的大小，与虚函数个数无关；多个虚函数对象大小与只有一个虚函数时相同（见 OneVirtual 和 TwoVirtual 的示例）。

**技巧**：对象大小只由非静态成员和 `#pragma pack` 对齐决定，外加一个可能的 VPTR；虚函数数量不影响 `sizeof`。

**例题**：构造/析构中虚函数调用陷阱  
**题目**：[L8-虚函数-2026.pptx P24 代码运行行为]
已知类定义：
```cpp
class Base {
public:
    virtual void foo() { cout << "Base::foo" << endl; }
    Base() { foo(); }
    void bar() { foo(); }
};
class Derived : public Base {
    int _num;
public:
    void foo() override { cout << "Derived::foo" << _num << endl; }
    Derived(int j) : _num(j) {}
};
```
写出如下代码的输出，并解释。
```cpp
Derived d(0);
Base &b = d;
b.bar();
b.foo();
```

**思路**：`Derived d(0)` 构造时先执行 `Base` 构造，调用 `foo()` 时虚机制不工作，调用 `Base::foo`。之后通过引用调用 `bar()`（非虚函数，但内部 `foo()` 是虚调用，此时对象完整，虚机制工作），以及直接调用虚函数 `b.foo()`。

**解答**：
- 构造 `d(0)` → `Base()` 内 `foo()` 输出 `Base::foo`。
- `b.bar()` → `bar()` 内 `this->foo()` 是虚调用，此时动态类型为 `Derived`，输出 `Derived::foo0`（构造时 `_num` 已由初始化列表初始化为 0）。
- `b.foo()` → 直接虚调用，输出 `Derived::foo0`。

最终输出：
```
Base::foo
Derived::foo0
Derived::foo0
```

**技巧**：构造函数体内的虚调用不“多态”；但构造结束后，对象的动态类型恢复正常。`_num` 由初始化列表在 `Base` 构造之前就完成初始化，所以 `Derived::foo` 输出时 `_num` 为合法值 0。

##### 1. `override` 关键字 【必背】

【定义】
`override` 是 C++11 引入的关键字，用于在派生类中显式声明一个成员函数是对基类虚函数的重写覆盖。它向编译器明确表达程序员的意图，让编译器帮助检查函数签名是否与基类虚函数完全匹配。

原理与推导
在大型项目中，基类的虚函数签名可能发生变动，而派生类中原本意图重写覆盖的函数可能在无意间变成重写隐藏。`override` 通过编译期检查，强制要求被修饰的函数必须成功重写基类中的某个虚函数，否则产生编译错误，从而将运行时难以发现的逻辑错误提前暴露。

【解题步骤】
在派生类中声明重写覆盖函数时，在函数参数列表后、函数体前（或`const`/`final`等限定符之后）添加 `override` 关键字：
1. 确认基类中有对应签名的 `virtual` 函数。
2. 在派生类的同名函数声明末尾加上 `override`。
3. 若编译报错，检查：参数类型、`const` 限定符、返回类型是否与基类虚函数严格一致。

【易错点】

| 错误写法 | 正确写法 | 错误原因 |
| :--- | :--- | :--- |
| `void foo(float) override {}` | `void foo(int ) override {}` | 基类虚函数为 `foo(int)`，参数类型不同，`override` 要求严格匹配。 |
| `void bar() override {}` | // 不应使用 override | 基类中 `bar()` 不是虚函数，`override` 只能修饰重写虚函数的函数。 |

例题
**例题**：课件 [L8-虚函数-2026.pptx P37] 中 Derived3 的定义。
**题目**：
```cpp
class Base {
public:
    virtual void foo(int) { cout << "Base::foo(int)" << endl; }
    void bar() { cout << "Base::bar" << endl; }
};

class Derived3 : public Base {
public:
    // 请判断以下两个成员函数声明是否正确
    void foo(int ) override { cout << "Derived3::foo(int)" << endl; } // (A)
    // void foo(float ) override {}; // (B)
    // void bar() override {};      // (C)
};
```
**思路**：`override` 关键字要求该函数必须是对基类虚函数的一个合法重写覆盖。判断标准：(1) 基类必须有同名同参数的虚函数；(2) 参数类型、返回类型、`const`限定必须完全匹配。
**解答**：
- (A) 正确。`Base::foo(int)` 是虚函数，`Derived3::foo(int)` 签名完全一致，是对其的重写覆盖。
- (B) 错误。参数类型为 `float`，与基类 `foo(int)` 不匹配，且基类中无 `foo(float)` 虚函数，此时 `override` 要求失败，编译器报错。
- (C) 错误。`Base::bar()` 并非虚函数，`override` 只能修饰对虚函数的重写，编译器报错。
**技巧**：始终为派生类中意图重写虚函数的函数添加 `override` 关键字，它是免费的编译期保险。

##### 2. `final` 关键字 【重点】

【定义】
`final` 是 C++11 引入的关键字，有两个应用场景：
1. 修饰虚函数：表示该虚函数是最终版本，禁止后续派生类再次重写覆盖它。
2. 修饰类：表示该类是最终类，禁止被其他类继承。

原理与推导
`final` 提供了在继承链上“封堵”的能力。某些类的实现需要保证其行为不被修改（如安全相关、底层基础功能），使用 `final` 可以防止子类通过重写虚函数或进一步继承来改变既定行为，增强代码的稳定性和安全性。

【解题步骤】
1. **禁止函数被重写**：在虚函数声明末尾（`override` 之后）加上 `final`。
```cpp
class A : public Base {
    void foo() override final {} // 禁止后续派生类重写foo
};
```
2. **禁止类被继承**：在类名后、继承列表前加 `final`。
```cpp
class B final : public A {
    // ...
};
```

【易错点】

| 错误用法 | 正确用法 | 说明 |
| :--- | :--- | :--- |
| `void bar() final {}` | `virtual void bar() final {}` | `final` 确保函数为虚且不可被重写，修饰非虚函数会导致编译错误。 |
| `class Base { void foo() final; };` | `class Base { virtual void foo() final; };` | 同上，`foo` 必须首先是虚函数。 |

例题
**例题**：课件 [L8-虚函数-2026.pptx P40] 中的选择题。
**题目**：
下列说法中正确的一项是（ ）
A. 发生重写隐藏时，函数的参数必须相同。
B. 发生重写覆盖时，函数的参数必须相同。
C. 使用 `override` 修饰的函数可以不是虚函数。
D. 使用 `final` 修饰的函数可以不是虚函数。
**思路**：回顾三个核心概念的定义与限制。
**解答**：
- A 错误。重写隐藏发生的条件之一是参数不同，或基类函数非虚。
- B 正确。重写覆盖要求函数签名（包括函数名和参数类型）完全相同，返回类型协变。
- C 错误。`override` 强制该函数是对基类虚函数的重写，所以它本身就是虚函数。
- D 错误。`final` 明确要求被修饰的函数必须是虚函数。
正确答案为 B。
**技巧**：辨析题紧抓概念本质——覆盖看签名一致，隐藏因签名或虚性缺失，`override`/`final` 必饰虚函数。

##### 3. 重写覆盖的失效条件与协变返回类型 【高频】

【定义】
- **`const` 导致的覆盖失效**：成员函数的 `const` 限定符是函数签名的一部分。若基类虚函数为非 `const`，而派生类同名函数添加了 `const`，签名不同，无法构成重写覆盖，而是形成重写隐藏。
- **协变返回类型**：派生类重写虚函数时，若返回类型为基类虚函数返回类型的指针或引用，且两者为继承关系，也视为合法的重写覆盖。

原理与推导
C++ 将 `const` 等限定符视为函数签名组成，是保证语义一致性的需要。异步修改对象的函数与承诺不修改对象的函数本质不同。协变返回类型允许重写覆盖更精确地返回派生类类型，提升类型安全性，而不破坏多态机制。

【解题步骤】
1. 排查 `const` 覆盖陷阱：
   - 检查派生类意图重写函数的 `const` 限定是否与基类一致。
   - 若不一致，要么移除派生类的 `const`，要么用 `using Base::func;` 恢复被隐藏的函数。
2. 使用协变返回类型：
   - 派生类重写函数的返回类型必须是基类返回类型所在类的派生类的指针或引用。
   - 两者必须是同一类别（都是指针，或都是左值引用，或都是右值引用），不能是多级指针。

【易错点】
- 非 `const` 和 `const` 成员函数是完全不同的函数，很容易因疏忽造成“我以为我重写了”但实际是“隐藏”的错误。
- 协变仅适用于**指针**和**引用**返回类型。直接返回对象类型不是协变的，会导致编译错误。

例题
**例题**：课件 [L8-虚函数-2026.pptx P45] 中 `const` 影响覆盖的例子。
**题目**：
```cpp
class Base1 {
public: virtual void f() { cout << "Base1::f" << endl; }
};
class Derive1 : public Base1 {
public:
    void f() const { cout << "Derive1::f" << endl; } // 意图重写覆盖
    using Base1::f;
};
int main() {
    Derive1 a; const Derive1 b;
    a.f(); b.f();
    return 0;
}
```
**思路**：`Derive1::f() const` 与 `Base1::f()` 的函数签名因 `const` 限定不同而不同，构成重写隐藏。`using Base1::f;` 恢复基类函数可见性。`const` 对象与非 `const` 对象会匹配不同的重载版本。
**解答**：
- `a.f()`：`a` 是非常量对象，有 `Base1::f()` 和 `Derive1::f() const` 两个候选。非常量成员函数优先匹配非常量对象，因此调用 `Base1::f()`。
- `b.f()`：`b` 是常量对象，无法调用非 `const` 的 `Base1::f()`，只能调用 `Derive1::f() const`。
输出：
```
Base1::f
Derive1::f
```
**技巧**：检查虚函数重写时，必须核对函数名、参数列表、`const`/`volatile`/引用限定符等全套签名信息。

##### 1. 纯虚函数与抽象类

###### 【定义】
- **纯虚函数**：在基类中声明但不实现的虚函数，其声明形式为在函数原型后加上 `= 0`（例如：`virtual void func() = 0;`）。纯虚函数可以拥有函数体，但必须在类外定义。
- **抽象类**：包含至少一个纯虚函数的类。抽象类不能被实例化，只能作为基类使用。

###### 原理与推导
- **设计目的**：
    - **接口规范**：强制所有派生类提供统一接口的具体实现。
    - **代码隔离**：基类只定义“做什么”，不定义“怎么做”，实现接口与实现的分离。
    - **多态基础**：通过基类指针或引用调用纯虚函数，在运行时根据实际对象类型动态绑定到派生类的实现。
- **限制与规则**：
    1.  若派生类未覆盖基类中的**所有**纯虚函数（纯虚析构函数除外），则该派生类仍为抽象类，不能被实例化。
    2.  纯虚析构函数比较特殊，即使派生类未显式实现，编译器也会自动合成析构函数，因此不影响派生类的实例化（前提是该派生类覆盖了其他所有纯虚函数）。

###### 【解题步骤】（判断与使用抽象类）
1.  **判断是否是抽象类**：查看类中是否声明了纯虚函数（`= 0`）。
2.  **判断派生类能否实例化**：检查派生类是否覆盖了基类中除析构函数外的所有纯虚函数。
3.  **使用抽象类**：永远通过指针或引用来操作抽象类对象，利用多态特性。

###### 【易错点】

| 易混淆概念 | 分析与区别 |
| :--- | :--- |
| **纯虚函数与空函数体** | `virtual void f() = 0;` 是纯虚函数，类为抽象类。`virtual void f() {}` 是普通虚函数，类可以实例化。 |
| **纯虚函数定义** | 纯虚函数可以有实现（在类外定义），这种定义常为派生类提供默认行为。但即使有定义，类依然是抽象类。 |
| **纯虚析构函数特殊性** | 必须提供函数体；即使派生类未显式实现，也不会导致派生类成为抽象类（编译器会自动合成）。 |

###### 例题
**例题**：
**题目**：以下关于抽象类的代码中，哪些语句会导致编译错误？为什么？
```cpp
class Base {
public:
    virtual void func() = 0;
};
void Base::func() { cout << "Base func" << endl; } // 定义纯虚函数体

class Derive1 : public Base {};

class Derive2 : public Base {
public:
    void func() override { Base::func(); }
};

int main() {
    // 1
    Base b;
    // 2
    Derive1 d1;
    // 3
    Derive2 d2;
    d2.func();
    return 0;
}
```
**思路**：本题考察抽象类与纯虚函数的规则。关键在于检查哪些类是抽象类，以及抽象类能否实例化。
**解答**：
1.  语句 `Base b;` 编译错误。因为 `Base` 类包含纯虚函数，是抽象类，不能实例化对象。
2.  语句 `Derive1 d1;` 编译错误。`Derive1` 继承了 `Base`，但未覆盖纯虚函数 `func()`，因此它仍然是抽象类，不能实例化。
3.  语句 `Derive2 d2; d2.func();` 正确。`Derive2` 覆盖了纯虚函数 `func()`，不再是抽象类，可以实例化并调用函数。程序会输出 `Base func`。
**技巧**：判断一个类能否实例化的唯一标准：它自身或它的所有基类中，是否存在**未被覆盖**的非析构纯虚函数。纯虚析构函数不影响实例化。

##### 2. 纯虚析构函数

- **纯虚析构函数**：声明为纯虚的析构函数（`virtual ~ClassName() = 0;`）。其**必须提供函数体**（通常在类外实现）。
- **目的**：仅作为一种技术手段，将基类变为抽象类，防止直接实例化基类对象。

- **为何需要函数体**：派生类析构时，会逐级调用基类的析构函数。如果纯虚析构函数没有函数体，链接阶段会出错。
- **对派生类的影响**：
    - **一般纯虚函数**：若派生类不覆盖，派生类仍是抽象类。
    - **纯虚析构函数**：若派生类不**显式**实现，编译器会自动合成一个析构函数。因此，只要派生类覆盖了基类的其他所有纯虚函数，该派生类即可被实例化。
- **常见设计模式**：当一个类既需要作为抽象基类（定义接口），又没有其他适合设为纯虚的函数时，将析构函数设为纯虚。

###### 【解题步骤】（分析与使用纯虚析构）
1.  **判断意图**：看到纯虚析构函数，应马上意识到其首要目的是让基类变为抽象类。
2.  **检查实现**：确保纯虚析构函数在类外有对应的函数体实现。
3.  **分析派生类**：判断一个继承自此类（其中只有析构是纯虚的，没有其他纯虚函数）的派生类是否可直接实例化。答案是可以，因为编译器会为它合成析构函数。

- **易错1：忘记实现函数体。** 只写 `virtual ~Base() = 0;` 而不在类外写 `Base::~Base() {}`，会导致任何继承它的派生类在实例化或析构时产生链接错误。
- **易错2：混淆对派生类的影响。** 误以为继承自仅有纯虚析构函数基类的派生类也是抽象类。**正确理解**：该派生类是具体类，可以实例化。

**例题**：[L9-多态与模板-2026.pptx P12]
**题目**：阅读下面程序，写出运行结果。
```cpp

public: 
	virtual ~Base()=0; 
};
Base::~Base() {cout<<"Base destroyed"<<endl;}

class Derive1: public Base {};
class Derive2: public Base {
public: 
	virtual ~Derive2() {cout<<"Derive2 destroyed"<<endl;} 
};

int main()
```

**思路**：本题考察虚析构函数的调用顺序以及纯虚析构函数的行为。当`delete`基类指针时，会根据指针所指对象的实际类型，从最派生的类开始，逐级向上调用析构函数。
**解答**：

2.  `delete p2;` ：`p2`指向`Derive2`对象。先执行`Derive2`的自定义析构，输出 `Derive2 destroyed`，然后调用基类`Base`的析构函数，输出 `Base destroyed`。

#### 向上向下类型转换与虚函数表 【高频】

##### 1. 向上类型转换
将派生类对象的指针或引用，转换为基类类型的指针或引用。这是C++中一种安全的、可由编译器隐式完成的转换。

##### 2. 向下类型转换
将基类类型的指针或引用，转换为派生类类型的指针或引用。此操作有风险，必须**显式**进行。

- **为何需要**：面向对象编程中，常使用基类指针集合来管理多种派生类对象（向上转型），这保留了共性，但丢失了特性。当需要调用某个派生类特有的接口时，必须将其还原回特定的派生类类型（向下转型）。
- **实现机制**：C++提供 `dynamic_cast` 和 `static_cast` 两种方式进行显式向下转型。

| 特性 | `dynamic_cast` | `static_cast` |
| :--- | :--- | :--- |
| **安全机制** | **运行时类型检查**。通过虚函数表（RTTI）判断实际对象是否可以被安全转换为目标类型。 | **编译时静态检查**。仅检查继承关系，不保证运行时指向对象的实际类型。 |
| **性能** | **较慢**。需要额外的运行时开销。 | **很快**。无运行时开销，与C风格类型转换类似。 |
| **失败行为** | **指针转换**失败返回 `nullptr`。<br>**引用转换**失败抛出 `std::bad_cast` 异常。 | **不安全**。转换“成功”但指针可能指向非法内存，导致未定义行为。 |
| **前置条件** | 源类型 `T1` 必须是**多态类型**（声明或继承了至少一个虚函数），否则编译错误。 | 源类型 `T1` 和目标类型 `T2` 之间必须有继承关系或已定义的转换路径，否则编译错误。 |
| **使用场景** | **安全的向下类型转换**。当你不确定基类指针是否真的指向某个特定派生类对象时。 | **确信安全的向下类型转换**。你已经明确知道基类指针所指对象的实际类型，出于性能考虑使用。 |

###### 【解题步骤】（安全的向下类型转换）
1.  **优先使用 `dynamic_cast`**：在需要向下转换时，首选 `dynamic_cast` 确保安全。
2.  **检查结果**：
    - 对指针形式 `T2* p = dynamic_cast<T2*>(ptr_base);`，必须检查 `p` 是否为 `nullptr`。
    - 对引用形式 `T2& r = dynamic_cast<T2&>(ref_base);`，必须使用 `try-catch` 块捕获异常。
3.  **使用 `static_cast`**：仅在性能敏感且通过逻辑完全能确保转换安全性的代码段使用。使用时应添加注释说明理由。

- **对象切片**：向上或向下类型转换应作用于**指针或引用**，而非对象本身。对对象进行转换会导致“对象切片”，丢失派生类信息。
- **忘记检查 `dynamic_cast` 返回值**：对指针版本直接解引用而不检查 `nullptr`，是导致程序崩溃的常见原因。
- **对非多态类型使用 `dynamic_cast`**：会导致编译错误。一定要确保被转换的源类型有虚函数。

**例题**：[L9-多态与模板-2026.pptx P23]
**题目**：阅读下面程序，写出运行结果。
```cpp

    auto d = dynamic_cast<Dog*>(p);
    auto b = dynamic_cast<Bird*>(p);
    if (d)
        d->run();
    else if(b)
        b->fly();
}
int main() {
    Pet* p[2];
    p[0] = new Dog;
    p[1] = new Bird;
    for (int i = 0; i < 2; ++i) {
        action(p[i]);
    }
    return 0;
}
```
**思路**：此题是`dynamic_cast`的典型应用。基类指针`Pet* p`指向不同的派生类对象。在`action`函数内部，通过`dynamic_cast`尝试将其转换为具体的派生类指针，并根据转换是否成功（是否为空）来执行派生类特有的行为。
**解答**：
1.  `action(p[0])`：`p[0]`实际指向`Dog`对象。`dynamic_cast<Dog*>(p)`成功，返回有效指针，执行 `d->run()`，输出 `dog run`。
2.  `action(p[1])`：`p[1]`实际指向`Bird`对象。`dynamic_cast<Dog*>(p)`失败，返回`nullptr`；接着 `dynamic_cast<Bird*>(p)`成功，返回有效指针，执行 `b->fly()`，输出 `bird fly`。
运行结果：
```
dog run
bird fly
```
**技巧**：利用 `dynamic_cast` 加 `if` 判断的模式，可以根据基类指针所指对象的真实类型实现分派，这是一种安全且清晰的“根据类型执行不同操作”的方法。

**【定义】**
- **重载**：同一作用域内，函数名相同但参数列表不同（类型、个数或顺序）的多个函数，构成重载。编译器根据调用时的实参类型在编译期决定调用哪个版本（静态多态）。
- **重写覆盖**：派生类中定义了与基类虚函数完全相同的函数（函数名、参数列表、返回值类型均相同），在运行时通过基类指针或引用调用时，根据对象真实类型决定执行哪个版本（动态多态）。
- **重写隐藏**：派生类中定义了与基类同名的函数（函数名相同，但参数列表或返回值类型可能不同），基类中所有同名函数（无论是否为虚函数、无论参数列表如何）在派生类中被隐藏。通过派生类对象直接调用时，只能访问派生类的版本。

**原理与推导**
C++ 的名字查找规则决定了这一行为：在编译期，编译器从当前作用域开始向外层作用域查找名字。一旦在当前作用域（派生类）找到了同名函数，就不会继续到外层作用域（基类）查找其他重载版本。因此，基类的所有同名函数被“隐藏”。而重写覆盖是动态多态的基石，依赖虚函数表在运行时实现。

**【易错点】**
三者的核心区别在于作用域、参数列表和是否为虚函数。

| 特性 | 重载 | 重写覆盖 | 重写隐藏 |
| :--- | :--- | :--- | :--- |
| **作用域** | 同一作用域内（如一个类内） | 基类与派生类之间 | 基类与派生类之间 |
| **函数名** | 相同 | 相同 | 相同 |
| **参数列表** | **必须不同** | **必须相同** | 可相同也可不同 |
| **基类函数** | 非虚或虚均可 | **必须是虚函数** | 虚或非虚均可 |
| **调用决定期** | 编译期（静态绑定） | 运行期（动态绑定） | 编译期（静态绑定） |
| **经典场景** | 构造函数；同功能但参数不同 | 通过基类指针实现多态 | 派生类定义了同名函数“不小心”遮蔽了基类版本 |

**【解题步骤】**
**辨析题三步法**：
1. **看作用域**：定义在同一个类里？→ **重载**。定义在基类和派生类之间？→ 下一步。
2. **看基类关键字**：基类函数有 `virtual` 吗？没有 → **重写隐藏**（派生类函数无论长什么样都是隐藏）。
3. **看函数签名**：基类函数是 `virtual`，派生类函数名称、参数、返回值都完全相同吗？是 → **重写覆盖**；否 → **重写隐藏**。

**例题**
**题目**：（来源课件 L9-多态与模板-2026.pptx P27 考点）
```cpp
class Base {
public:
    void func(int) { cout << "Base::func(int)" << endl; }
    virtual void bar() { cout << "Base::bar" << endl; }
};
class Derived : public Base {
public:
    void func(double) { cout << "Derived::func(double)" << endl; } // 隐藏了 Base::func(int)
    void bar() { cout << "Derived::bar" << endl; }                // 覆盖了 Base::bar
};

int main() {
    Derived d;
    d.func(1);     // 调用哪个？
    d.func(1.0);   // 调用哪个？
    Base* p = &d;
    p->func(1);    // 调用哪个？
    p->bar();      // 调用哪个？
}
```
**思路**：按“三步法”分析。`Base::func(int)` 非虚，`Derived::func(double)` 构成了重写隐藏。`Base::bar()` 是虚函数，`Derived::bar()` 同名同参，构成重写覆盖。调用行为按静态/动态绑定规则判断。
**解答**：
1. `d.func(1)`：首先在 `Derived` 作用域找到 `func(double)`，基类 `func` 被隐藏且不可见。参数 `1` 是 `int`，可隐式转换为 `double`，调用 `Derived::func(double)`。
2. `d.func(1.0)`：调用 `Derived::func(double)`。
3. `p->func(1)`：`p` 是 `Base*` 类型，`Base` 的 `func` 是非虚函数，编译期绑定。调用 `Base::func(int)`。
4. `p->bar()`：`p` 是 `Base*` 类型，`bar` 是虚函数，运行时因 `p` 指向 `Derived` 对象而调用 `Derived::bar()`。
**技巧**：一句话口诀：“非虚即隐藏，签名一致才覆盖。直接对象看当前，基类指针查基类（除非是虚函数）。”

---

**原理与推导**
- **构造函数**：在构造一个派生类对象时，构造顺序是从基类到派生类。当执行基类构造函数体时，派生类部分尚未构造完成。此时若调用虚函数，若允许动态绑定到派生类版本，将可能访问到未初始化的派生类成员，导致未定义行为。因此 C++ 规定：**在构造和析构期间，虚函数机制被禁用，调用的虚函数版本是当前正在构造或析构的类自身定义的版本。**
- **析构函数**：析构顺序是从派生类到基类，与构造完全相反。当执行基类析构函数时，派生类部分已经析构完成。此时同样为防止访问已销毁的派生类成员，虚函数调用也被绑定到当前类的版本。
- **必须为虚析构**：当通过基类指针 `delete` 一个派生类对象时，如果基类析构函数不是虚函数，将仅调用基类析构函数，派生类部分无法被正确析构，造成资源泄漏。因此，**带有多态性质的基类（即含有虚函数的类）必须声明虚析构函数。**

**【易错点】**
1. **构造/析构中调虚函数的行为**：初学者常误认为会调用最终覆盖版本，实际是调用当前类版本。
2. **虚析构的必要性**：只有在作为基类且有资源需要管理时才必须。如果一个类不打算作为基类，或基类无任何虚函数且不通过基类指针删除对象，则无需虚析构。
3. **纯虚析构必须有定义**：若声明 `virtual ~Base() = 0;`，它仍是虚析构，但必须提供函数体 `Base::~Base() {}`，否则链接器会因派生类析构时会调用它而报错。

**例题**
**题目**：以下代码输出什么？（来源课件 L9-多态与模板-2026.pptx 考点衍生）
```cpp
#include <iostream>
using namespace std;
class Base {
public:
    Base() { build(); }
    virtual void build() { cout << "Base::build" << endl; }
    virtual ~Base() {}
};
class Derived : public Base {
public:
    Derived() : data_(10) {}
    void build() override { cout << "Derived::build, data = " << data_ << endl; }
    ~Derived() { cout << "Derived dtor" << endl; }
private:
    int data_;
};
int main() {
    Base* p = new Derived();
    delete p;
    return 0;
}
```
**思路**：执行 `new Derived()` 时，先构造基类 `Base` 部分。在 `Base` 的构造函数中调用虚函数 `build()`。根据“构造函数中虚函数不生效”原则，此时调用的不是 `Derived::build()`，而是 `Base::build()`。随后构造派生类部分。`delete p` 正确调用虚析构，执行 `~Derived()` 再执行 `~Base()`。
**解答**：
程序输出为：
```
Base::build
Derived dtor
```
**技巧**：永远不要在构造函数或析构函数中调用虚函数，因为这违背了直觉上的多态，并且很可能引入未初始化的数据访问。

---

**【定义】**
- **纯虚函数**：在基类中声明但不提供实现的虚函数，其形式为在声明尾部添加 `= 0`。例如：`virtual void draw() = 0;`。
- **抽象类**：包含至少一个纯虚函数的类。抽象类不能实例化对象，只能作为基类，为其派生类提供公共接口。
- **纯虚析构函数**：用 `= 0` 声明的虚析构函数。其特殊之处在于：即使声明为纯虚，也必须为它提供函数体，否则派生类对象析构时会因找不到基类析构函数实现而导致链接错误。

**原理与推导**
抽象类定义了接口规范，强制派生类必须实现所有纯虚函数，否则派生类自身也成为抽象类。这是面向对象设计中“接口与实现分离”的基石。设计原则：若要让基类只作为接口而不产生对象，就将其设为抽象类（通常至少给一个纯虚析构或其他纯虚函数）。

**【易错点】**
- **抽象类不能有实例**：只要类中有一个 `= 0` 的纯虚函数，就不能用 `new` 创建实例。
- **纯虚析构的双重身份**：`virtual ~Base() = 0;` 既使类成为抽象类，又声明了虚析构。但**必须**在类体外给出实现：`Base::~Base() {}`。
- **`override` 关键字**：在派生类重写虚函数时，推荐使用 `override` 关键字，如 `void draw() override {}`。编译器会检查是否真的覆盖了基类虚函数，可有效避免因函数签名不匹配导致的隐藏错误。

**例题**
**题目**：（来源课件 L9-多态与模板-2026.pptx P25, P26 考点）
```cpp
class Vehicle {
public:
    virtual void startEngine() = 0; // 纯虚函数，接口定义
    virtual ~Vehicle() = 0;        // 纯虚析构，使类变为抽象类
};
Vehicle::~Vehicle() {} // 必须提供定义

// 正确用法：派生类实现所有纯虚函数
class Car : public Vehicle {
public:
    void startEngine() override { /* ... */ }
};

int main() {
    // Vehicle v; // 错误！抽象类不能被实例化
    Vehicle* p = new Car(); // 正确，非抽象派生类实例可赋给基类指针
    p->startEngine();       // 多态调用
    delete p;               // 正确析构（通过虚析构）
}
```
**思路**：这道题考察抽象类的定义与使用规则。`Vehicle` 含有纯虚函数 `startEngine()` 且定义了纯虚析构，因此是抽象类。抽象类唯一的用途就是作为基类指针/引用的类型。删除基类指针时，虚析构保证了完整调用链。
**解答**：代码已做正确示范。关键点有三：1. 纯虚函数使类成为抽象类。2. 纯虚析构必须有实现。3. 派生类重写了纯虚函数后，才能被实例化。
**技巧**：设计一个“仅作接口规范”的基类时，记住两个标配：一个纯虚函数建立抽象性，一个纯虚析构并立即在类外写空实现。

##### 函数模板的定义与编译期多态 【高频】
【定义】
函数模板是一种泛型编程机制，它使用泛型标记（如 `typename T`）定义一个函数的蓝图。编译器在编译期根据实际调用时传入的参数类型，自动生成对应类型的具体函数实例。这种在编译期确定具体调用代码的机制，被称为**静多态**或编译期多态。

原理与推导：
函数模板的核心原理是**编译期代码生成**。编译器遇到一个函数模板的定义时，并不会立即生成可执行代码。只有当该模板被调用（使用）时，编译器才根据调用的参数类型，生成一份针对该类型的特化版本，并进行编译。
```cpp
template<typename T>
T sum(T a, T b) { return a + b; }

int main() {
    int a = sum(1, 2);        // 编译器生成 int sum(int, int)
    double b = sum(1.0, 2.0); // 编译器生成 double sum(double, double)
    return 0;
}
```
**为什么模板的声明和定义通常必须放在头文件中？**
因为模板的实例化发生在编译期。如果模板的定义在 `a.cpp` 中，而调用在 `main.cpp` 中，编译 `a.cpp` 时，编译器看不到任何调用，因此不会生成任何实例代码。当链接器试图链接 `main.cpp` 中对 `sum<int>` 的调用时，就会因为找不到定义而报链接错误。因此，模板的完整定义必须在调用点对编译器可见，通常做法是直接放在头文件中。

【解题步骤】
1.  **识别类型依赖**：确定函数中哪些地方依赖于类型。
2.  **声明模板参数**：在函数前使用 `template<typename T>` 或 `template<class T>` 声明一个类型参数 `T`。
3.  **替换具体类型**：将原函数中所有需要泛化的具体类型（如 `int`, `double`）替换为模板参数 `T`。
4.  **注意约束**：当模板函数用于自定义类型时，必须确保该类型支持模板函数体内使用的所有操作符（如 `operator>`），否则编译报错。

【易错点】

| 易混概念 | 辨析 |
| :--- | :--- |
| **静多态（模板） vs 动多态（虚函数）** | **编译期 vs 运行期**：模板在编译时确定类型，生成代码，无运行时开销，但会导致代码膨胀。虚函数在运行时通过虚函数表动态查找，有间接调用开销，但更灵活。 |
| **模板的编译错误** | 通常会产生大量、难以阅读的错误信息。检查时通常只需要关注**第一个错误**。出现 `no match for ‘operator...’` 等提示，说明模板对类型参数施加了未被满足的操作约束。 |
| **分开编译** | 普通函数可以将声明放在头文件，定义放在源文件。**模板不能这样做**，必须在头文件中提供完整定义，否则会链接失败。 |

**例题**：
**题目**：分析以下代码为何会编译错误，并给出解决方案 [L9-多态与模板-2026.pptx P41]。
```cpp
class MyInt {
public:
    int data;
    MyInt(int val): data(val) {};
};

int main() {
    MyInt arr_c[] = {3, 2, 4, 1, 5};
    sort(arr_c, 5); // 假设 sort 是能对数组排序的函数模板
    output(arr_c, 5);
    return 0;
}
```
**思路**：错误在于函数模板 `sort` 对 `MyInt` 类型的实例化。需要分析 `sort` 模板内部对类型参数 `T` 施加了哪些操作要求，并检查 `MyInt` 类是否支持这些操作。
**解答**：
1.  **分析错误**：编译器报错 `error: no match for ‘operator>’`。这表明在 `sort` 函数模板内部，代码试图使用 `operator>` 来比较两个 `MyInt` 对象的大小，但 `MyInt` 类并未重载该操作符。
2.  **解决方案**：为 `MyInt` 类重载 `operator>`。同时，为了能在 `output` 中输出，通常也需要重载 `operator<<`。
```cpp
class MyInt {
public:
    int data;
    MyInt(int val): data(val) {};
    // 重载 > 操作符，用于比较
    bool operator>(const MyInt& b){
        return data > b.data;
    }
    // 重载 << 操作符，用于输出
    friend std::ostream& operator<<(std::ostream& out, const MyInt& obj){
        out << obj.data;
        return out;
    }
};
```
**技巧**：使用模板编写通用代码时，要清楚模板对类型参数施加了哪些“隐含接口”或“操作约束”。为自定义类型使用模板时，必须确保其满足这些约束。

#### 类模板的定义、声明与实例化 【重点】
【定义】
类模板（Class Template）是将类的成员类型参数化的机制。它允许创建一个类的蓝图，其中的某些类型（如数据成员的类型、成员函数的参数或返回值类型）在定义时不被指定，而是在使用该类创建对象时作为模板参数传入。

原理与推导：
与函数模板类似，类模板的实例化也发生在编译期。当使用 `类模板名<具体类型>` 声明一个对象时，编译器会生成一个真实的、所有类型都已确定的类，然后为这个类创建对象。
```cpp
template <typename T> class A {
    T data; // 成员类型 T 将在实例化时确定
public:
    A(T _data): data(_data) {}
    void print() { cout << data << endl; }
};

int main() {
    A<int> a(1);    // 实例化 A<int> 类，并创建对象 a
    A<double> b(1.2); // 实例化 A<double> 类，并创建对象 b
    return 0;
}
```
类外定义成员函数时，需要加上模板声明和作用域，且成员函数名前的类名必须带模板参数： `A<T>`。
```cpp
template<typename T>
void A<T>::print() { cout << data << endl; }
```

【易错点】
类模板的类名不代表一个具体的类型，`A` 不是类型，`A<int>` 才是类型。

#### 非类型模板参数与编译期确定的值 【重点】
【定义】
模板参数不一定是类型，也可以是**编译期确定的值**，这被称为非类型模板参数（Non-type template parameter）。常见的有整数、枚举、指针或引用。非类型参数使得可以基于值对模板进行参数化，如指定数组的固定大小。

原理与推导：
非类型参数的值必须在编译时就能确定。因此可以传入整数常量、`const` 变量或字面量，但不能传入运行时才能确定的普通变量。
```cpp
template<typename T, unsigned size> // size 是非类型参数
class array {
    T elems[size]; // 使用非类型参数定义数组成员大小
public:
    ... 
};

int main(){
    int n = 5;
    // array<char, n> array0; // 错误！n 是变量，编译期不可知
    const int m = 5;
    array<char, m> array1; // 正确，m 是编译时常量
    array<char, 5> array2; // 正确
    return 0;
}
```

【易错点】

| 对比项 | 类型参数 | 非类型参数 |
| :--- | :--- | :--- |
| **作用** | 抽象掉“类型” | 抽象掉“值” |
| **声明** | `typename T` 或 `class T` | 具体类型名，如 `int N` |
| **实参** | 任何类型，如 `int`, `MyClass` | 编译期常量，如 `10`, `const int` |
| **典型用途** | 泛型容器 `std::vector<T>` | 固定大小数组 `std::array<T, N>` |

**例题**：【例题：待补充】

#### 模板与多态的对比：静多态与动多态 【必背】
【定义】
模板和虚函数是实现C++多态的两种核心方式。
- **静多态（Static Polymorphism）**：通过模板实现，在**编译期**通过代码生成解决函数调用，体现为函数重载和模板实例化。
- **动多态（Dynamic Polymorphism）**：通过继承和虚函数实现，在**运行期**通过虚函数表查找解决函数调用。

原理与推导：

| 对比维度 | 静多态（模板） | 动多态（虚函数+继承） |
| :--- | :--- | :--- |
| **实现机制** | 编译时生成具体类型代码（实例化） | 运行时通过虚函数表（vtable）动态绑定 |
| **时机** | **编译期** | **运行期** |
| **性能** | **高效**，无间接调用开销 | 存在一次额外的指针间接寻址开销 |
| **类型约束** | **非侵入式**，类型只需满足语法要求（鸭子类型） | **侵入式**，类型必须显式继承自特定基类 |
| **代码体积** | 不同参数实例化会生成多份代码，可能导致**代码膨胀** | 一份函数代码，通过基类指针/引用调用，体积小 |
| **灵活性** | 不同实例生成的类型相互独立，编译后不可变 | 可在运行时通过基类指针灵活替换不同派生类对象 |

【易错点】
模板是利用“看起来一样”的代码接口（鸭子类型）在编译期实现多态，而虚函数是利用“确实一样”的基类接口在运行期实现多态。

#### 成员函数模板（自学）
【定义】
成员函数模板是指在普通类或类模板中，将其成员函数定义为模板。这允许该成员函数拥有自己独立的、与类模板参数不同的模板参数，从而提供了更高维度的泛化能力。
```cpp
// 普通类中的成员函数模板
class normal_class {
public:
    int value;
    // 类内定义
    template<typename T> void set(T const& v) {
        value = int(v);
    }
    // 类内声明
    template<typename T> T get();
};
// 类外定义
template<typename T>
T normal_class::get() {
    return T(value);
}
```

原理与推导：
当一个类模板的成员函数自身也是模板时，类的模板参数和函数的模板参数是**两层独立的参数**。在类外定义时，必须分别提供两套 `template<...>` 声明，并且顺序固定：先类模板参数，再成员函数模板参数。
```cpp
template<typename T0> class A { 
    T0 value; 
public:
    template<typename T1> void set(T1 const& v); // 声明
};
// 类外定义
template<typename T0>  // 类模板参数
template<typename T1>  // 成员函数模板参数
void A<T0>::set(T1 const& v) { 
    value = T0(v); 
}
```
绝不能错误地合并成 `template<typename T0, typename T1>`，这会被理解为同时接受两个参数的单一模板。

【解题步骤】
1.  **分析层级**：区分模板参数属于类还是成员函数。
2.  **书写顺序**：在类外定义时，先写类模板的 `template<...>`，再写成员函数模板的 `template<...>`。
3.  **类型推导**：成员函数模板的参数能从调用实参推导，也可以显式指定无法推导的部分。
    ```cpp
    A<int> a;
    a.set(5);          // 自动推导T1为int
    double t = a.get<double>(); // 手动指定T1为double
    ```

【易错点】

| 错误写法 | 正确写法 | 解析 |
| :--- | :--- | :--- |
| `template<typename T0, typename T1>` <br> `T1 A<T0>::get(){...}` | `template<typename T0> template<typename T1>` <br> `T1 A<T0>::get(){...}` | 前者会被编译器理解为一个带有两个参数的单一模板，语法错误。必须分别声明。 |
| 将非类型参数传递为变量 | `int n=5; array<int, n> a;` | `const int n=5;` 或 `array<int, 5> a;` | 非类型模板参数要求编译期常量。 |
### 六、模板特化与重载 （自学）

#### 函数模板特化 【重点】

##### 1. 成员函数模板

**【定义】**
成员函数模板是指在类模板中定义的、自身带有额外独立模板参数的成员函数。这使得成员函数可以接受或返回与类模板参数不同类型的数据。

**【原理与推导】**
普通类模板的类型参数 `T0` 在实例化对象时确定，而成员函数模板的参数 `T1` 在调用该函数时独立推导或指定。编译器在编译成员函数模板时，会进行两次类型替换：
1. 实例化类模板，确定 `T0`。
2. 调用函数模板，推导或指定 `T1`，生成具体的成员函数。

这提供了极大的灵活性，允许在一个以 `int` 实例化的类对象中，调用接受 `double` 参数或返回 `string` 类型的成员函数。

**【解题步骤】**
编写或使用成员函数模板时：
1. **识别需求**：当成员函数的参数或返回值类型需要与类模板参数不同，但又希望保持泛型时。
2. **类内声明/定义**：
   ```cpp
   template<typename T0> class MyClass {
   public:
       template<typename T1> void func(T1 const& v); // 声明
   };
   ```
3. **类外定义（如有必要）**：需要写两重模板声明。
   ```cpp
   template<typename T0> // 类模板声明
   template<typename T1> // 函数模板声明
   void MyClass<T0>::func(T1 const& v) {
       // 实现
   }
   ```
4. **调用方式**：
   - **自动推导**：`obj.func(5.0);` 编译器从参数 `5.0` 自动推导 `T1 = double`。
   - **手动指定**：`obj.func<int>(5.0);` 显式指定 `T1 = int`。无法从参数推导时（如通过返回值类型推导）必须手动指定。

**【易错点】**
| 易混概念 | 错误理解 / 写法 | 正确做法 |
| :--- | :--- | :--- |
| **类外定义的语法** | `template<typename T0, typename T1>`<br>`void MyClass<T0>::func(T1 const& v)`<br>企图把两个模板参数合并在一行声明。 | **必须用两行独立模板声明**：<br>`template<typename T0>`<br>`template<typename T1>`<br>`void MyClass<T0>::func(T1 const& v)` |
| **调用时的推导** | `double val = obj.get();` <br>如果 `get` 模板参数 `T1` 只用于返回值，编译器无法从调用上下文推导 `T1`。 | **必须手动指定类型**：<br>`double val = obj.get<double>();` |

**【例题】**
- **题目**：指出下面类外定义成员函数模板的代码错在哪里，并改正。
  ```cpp
  template<typename T0> class Wrapper { 
      T0 data; 
  public:
      template<typename T1> void assign(T1 const& val);
  };
  
  template<typename T0, typename T1>
  void Wrapper<T0>::assign(T1 const& val) {
      data = T0(val);
  }
  ```
  （自编）

- **思路**：考查成员函数模板在类外定义时的语法规则。编译器需要知道哪个 `template` 属于类，哪个属于函数。

- **解答**：
  错误在于将类模板参数和函数模板参数合并在一个 `template` 声明中。
  正确写法是分离为两个独立的模板声明：
  ```cpp
  template<typename T0>
  template<typename T1>
  void Wrapper<T0>::assign(T1 const& val) {
      data = T0(val);
  }
  ```

- **技巧**：记住“两层模板，两层声明”。类模板的 `template` 在先，函数模板的 `template` 在后，次序不可颠倒。

##### 2. 函数模板的特化与重载辨析

**【定义】**
- **函数模板特化**：为特定类型提供不同于通用模板的特殊实现。
- **非模板重载函数**：一个普通函数，与函数模板同名，但参数列表不同或更具体。
- 三者调用优先级：**非模板重载函数 > 全特化函数模板 > 通用函数模板**。编译器总是优先选择最“具体”的那个版本。

**【原理与推导】**
重载决议时，编译器将候选函数分为两组：普通函数和从模板实例化出的函数。如果存在一个完全匹配的普通函数，它会被优先选择。函数模板特化不参与首次重载决议，只有当函数模板被选中后，编译器才会检查是否有适用于此类型的最佳特化版本。这一复杂规则是【易错点】的根源。

**【例题：待补充】**

#### 类模板特化

**【定义】**
类模板特化是指为某个类模板的特定类型参数组合，提供一份专门的实现。它可以完全偏离通用模板的定义，拥有完全不同的成员、方法和数据布局，以满足特定类型的需求或优化。

**【原理与推导】**
当编译器实例化一个类模板时，会首先查找是否存在与模板参数完全匹配的全部或部分特化版本。
- **全特化**：当所有模板参数都被具体的类型固定时。
- **偏特化（部分特化）**：当只有部分参数被固定，或对参数施加了额外约束（如类型变为指针、引用）时。

使用特化的典型场景包括：
1. **类型萃取（Type Traits）**：在 STL 中广泛使用，用于在编译期查询或修改类型属性。
2. **为特定类型提供更优实现**：如 `vector<bool>` 使用位压缩存储，显著节省空间。
3. **处理指针类型的特殊逻辑**：对 `T*` 进行偏特化，实现深拷贝等与值类型不同的行为。

**【解题步骤】**
1. **声明通用模板（Primary Template）**：
   ```cpp
   // 主模板
   template <class T>
   struct MyClass {
       static void info() { std::cout << "Generic" << std::endl; }
   };
   ```
2. **全特化**：为 `T = int` 实现一个完全不同的版本。
   ```cpp
   template <>
   struct MyClass<int> {
       static void info() { std::cout << "Specialized for int" << std::endl; }
   };
   ```
3. **偏特化**：为 `T` 是指针类型提供通用实现。
   ```cpp
   template <class T>
   struct MyClass<T*> {
       static void info() { std::cout << "Partial specialized for pointer" << std::endl; }
   };
   ```
4. **使用**：`MyClass<double>::info();` 输出 "Generic"；`MyClass<int>::info();` 输出 "Specialized for int"；`MyClass<double*>::info();` 输出 "Partial specialized for pointer"。

**【易错点】**
- **函数模板不能偏特化**：这是函数模板与类模板的关键区别。函数模板只能**重载**来实现类似偏特化的效果。
- **特化版本的声明与定义**
  | 易混概念 | 错误理解 | 正确做法 |
  | :--- | :--- | :--- |
  | **特化时机** | 在使用特化版本**之后**才声明或定义特化。 | 特化版本必须在所有使用该特化的源代码文件中提前声明（通常是放在头文件中，紧跟主模板之后），否则编译器可能会实例化主模板，导致 ODR 违例或非预期行为。 |
  | **偏特化函数** | `template<typename T> void func<T*>(T* p){}` | 这是**非法**的函数模板偏特化。正确做法是**函数模板重载**：<br>`template<typename T> void func(T* p){}` |

**【例题】**
- **题目**：阅读代码，判断输出结果。
  ```cpp
  #include <iostream>
  template <class T> class Container {
  public:
      void info() { std::cout << "Generic" << std::endl; }
  };
  // (1) 偏特化：T是指针
  template <class T> class Container<T*> {
  public:
      void info() { std::cout << "Pointer" << std::endl; }
  };
  // (2) 全特化：T = int*
  template <> class Container<int*> {
  public:
      void info() { std::cout << "int*" << std::endl; }
  };

  int main() {
      Container<char> c1;         c1.info();
      Container<char*> c2;        c2.info();
      Container<int*> c3;         c3.info();
      return 0;
  }
  ```

- **思路**：考查类模板特化选择的优先级：全特化 > 偏特化 > 主模板。编译器会为每个实例化找出最匹配的特化版本。

- **解答**：
  - `Container<char>` 无法匹配 (1) 和 (2)，选择**主模板**。输出：`Generic`
  - `Container<char*>` 能够匹配偏特化 (1) （T = char），并且没有针对 `char*` 的全特化，选择**偏特化**。输出：`Pointer`
  - `Container<int*>` 能够匹配偏特化 (1) （T = int）和 全特化 (2)，**全特化**是更佳匹配。输出：`int*`

- **技巧**：匹配规则可以概括为“普通版本是备胎，偏特化是升级版，全特化是定制版”。存在定制版时，编译器不会考虑升级版或备胎。

### 本章小结
本章知识点紧密围绕“类型安全、资源管理、多态机制”三条主线展开：I/O流和`stringstream`通过继承与模板实现格式化与类型安全转换；函数对象与智能指针是资源管理与可调用对象封装的典范；虚函数、纯虚函数、向上/向下转型构建了运行期多态的完整框架；模板则提供了编译期多态的泛型支持，两者互为补充。

**核心结论**：
1. 流对象不可复制，重载`operator<<`必须返回`ostream&`。
2. 继承体系中，通过基类指针`delete`对象时，基类析构必须为虚函数，否则资源泄漏。
3. `shared_ptr`控制块增减遵循“拷贝加1，析构/重置减1”；`unique_ptr`独占所有权，禁止复制。
4. 构造函数/析构函数中不会触发虚函数动态绑定；纯虚析构必须提供实现。
5. 模板实例化发生在编译期，必须头文件全定义；函数模板无偏特化，靠重载替代。

**必背公式**：
- `ostream& ostream::operator<<(ostream& (*fn)(ostream&));`  // 操纵算子原型
- `SmartPtr& operator=(const SmartPtr& rhs) { ++rhs.rp->count; if (--rp->count == 0) delete rp; rp = rhs.rp; return *this; }`  // 安全赋值
- `class Derived : public Base { void func() override final {} };`  // 最终覆盖
- `template<typename T0> template<typename T1> void A<T0>::func(T1 v) {}`  // 成员函数模板类外定义

【记忆】构建多态与泛型程序的两大支柱：运行时看虚函数表与动态转型，编译时靠模板实例化与类型推导。
