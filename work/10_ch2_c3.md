### 四、函数对象（Function Object）回顾

#### 函数指针与函数对象的概念统一 【高频】

**【定义】**
函数对象（Function Object，也称仿函数 Functor）是重载了 `operator()` 的类对象，使其可以像函数一样被调用。函数指针是指向函数的指针变量，同样可以用于调用目标函数。二者从调用方的角度来看，行为统一：都可以使用 `func(args)` 的形式进行调用。

**【原理与推导】**
在 C++ 中，任何可以以 `f(args)` 形式调用的实体，都可以看作一种可调用对象（Callable）。函数指针天然支持这种调用语法；函数对象则通过自定义的 `operator()` 成员函数实现相同的语法。当我们将可调用对象作为参数传递给算法（如 `sort`）时，不需要关心它到底是函数指针还是对象实例——只要它能通过 `()` 调用并返回期望类型即可。这种“只关心行为，不关心具体类型”的思想被称为鸭子类型（Duck Typing）。

**【解题步骤】**
- 识别场景：需要将一段行为（如比较规则、处理步骤）作为参数传递给模板函数或 STL 算法。
- 决定形式：
  1. 若逻辑简单且无状态，优先使用普通函数，传函数指针。
  2. 若需要维护状态（如计数器、阈值），定义函数对象类，重载 `operator()`。
  3. 若需要将多种不同形式的可调用对象统一存储或传递，使用 `std::function` 进行类型擦除。
- 使用：传递给模板参数（直接推导类型）或 `std::function` 对象。

**【易错点】**

| 易混概念 | 函数指针 | 函数对象 |
|---------|---------|----------|
| 本质 | 指向代码段的指针 | 类实例，可拥有数据成员 |
| 状态 | 无状态（除非指向函数内静态变量） | 可以有状态（通过成员变量） |
| 类型 | 编译期确定，函数签名必须匹配 | 编译期确定类型，但通过模板参数可泛化 |
| 赋值给 `std::function` | 支持 | 支持 |
| 存储于数组中 | 需要数组元素类型一致（函数指针类型严格） | 不同类型函数对象不能直接放入同一数组，除非使用类型擦除 |

- 常见误解：认为 `std::function` 能够直接存储“所有可调用对象”的类型而不发生转换——实际上 `std::function` 使用了类型擦除，运行时可能产生性能开销，不能完全替代模板。
- 错误用法：试图将一个有状态的函数对象赋值给普通函数指针（无法实现）。

**【例题】**

**例题**：
**题目**：以下代码试图将函数指针和函数对象统一存储到数组中，但编译失败。请说明原因，并给出修正方案。
```cpp
string readFromScreen() { ... }
class ReadFromFile {
public:
    string operator()() { ... }
};
auto readArr[] = {readFromScreen, ReadFromFile()}; // 编译错误
```
**思路**：`auto` 要求数组元素类型一致，但 `readFromScreen` 是函数指针类型 `string(*)()`，而 `ReadFromFile()` 是 `ReadFromFile` 类型，二者类型不同，无法推导共同类型。
**解答**：
引入 `std::function` 进行类型擦除，统一为 `function<string()>` 类型：
```cpp
#include <functional>
std::function<std::string()> readArr[] = {readFromScreen, ReadFromFile()};
```
此时数组元素均为 `std::function<std::string()>`，可存储函数指针或函数对象。
**技巧**：当需要将多种不同类型的可调用对象放入同一容器时，使用 `std::function` 或多态的 `std::function` 包装。

#### operator()重载与鸭子类型 【必背】

**【定义】**
鸭子类型（Duck Typing）是一种动态类型风格：如果一个对象能像函数一样调用（即支持 `obj(args)`），那么它就可以被当作函数使用，而不必关心它到底是函数指针还是某个类的实例。在 C++ 中，这一特性通过重载 `operator()` 并结合模板实现。

**【原理与推导】**
C++ 模板在编译期间会对传入的参数进行实例化，要求参数支持某种操作（如 `a(b)`）。如果传入的对象拥有 `operator()` 成员函数，模板代码就能通过编译。编译器并不会在模板定义处检查类型的具体实现，而是等到实例化时进行匹配。这恰好实现了鸭子类型：只要看起来像函数、能调用，就能作为函数使用。

函数对象重载 `operator()` 的基本形式：
```cpp
class AgeComp {
public:
    bool operator()(const People &a, const People &b) const {
        return a.age < b.age;
    }
};
```
使用时：
```cpp
AgeComp comp;
bool res = comp(p1, p2);        // 直接调用
sort(vec.begin(), vec.end(), AgeComp()); // 传递给 STL 算法
```

**【解题步骤】**
1. 设计一个类，其内部可能包含需要持久化的状态（如比较的基准值）。
2. 在类中声明 `返回类型 operator()(参数列表) const;`（常成员函数以避免副作用）。
3. 将该类的临时对象或实例作为参数传递给接受可调用对象的模板函数。

**【易错点】**

| 易混概念 | 函数调用 | 普通成员函数调用 |
|---------|---------|-----------------|
| 语法 | `obj(args)` | `obj.func(args)` |
| 本质 | 调用 `obj.operator()(args)` | 调用具名成员函数 |
| 鸭子类型适用 | 支持 | 不支持（除非模板要求具名成员） |

- 常见错误：在传递函数对象时忘记加括号，例如 `sort(... , AgeComp)` 传入的是类型名而非实例，应写成 `AgeComp()`。
- 注意 `operator()` 的 const 修饰：如果缺失，可能无法绑定到某些只接受常引用可调用对象的算法。

**【例题】**

**例题**：
**题目**：定义一个函数对象 `Threshold`，其构造函数接受一个阈值 `int t`，内部重载 `operator()(int x)` 返回 `x > t`。配合 `count_if` 统计一个 `vector<int>` 中大于 5 的元素个数。 [来源 L11-iostream&函数对象和智能指针-2026.pptx P45 改编]
**思路**：使用函数对象封装状态（阈值），并传递给 `count_if`。
**解答**：
```cpp
#include <vector>
#include <algorithm>
using namespace std;

class Threshold {
    int t;
public:
    Threshold(int thr) : t(thr) {}
    bool operator()(int x) const {
        return x > t;
    }
};

int main() {
    vector<int> v = {1, 8, 3, 6, 2};
    int cnt = count_if(v.begin(), v.end(), Threshold(5));
    // cnt == 2
}
```
**技巧**：与普通函数只能通过额外参数或全局变量传递阈值相比，函数对象可以将状态（如阈值）封装在内部，保持接口简洁。

#### 比较器在STL中的深度应用 【高频】

**【定义】**
STL 中的许多算法（如 `sort`、`lower_bound`、`priority_queue`）以及关联容器（`map`、`set`）都需要一种“比较关系”来确定元素顺序。比较器可以是函数指针、函数对象或 lambda 表达式，只要满足严格弱序要求。

**【原理与推导】**
以 `sort` 为例，默认使用 `operator<` 进行升序排序。如果要自定义排序规则，可提供第三个参数 `comp`，要求 `comp(a, b)` 在 `a` 应排在 `b` 之前时返回 `true`。STL 通过模板接收任意类型的比较器，在算法内部调用 `comp(*it1, *it2)`。实现比较器的方式有三种：
1. **重载 `operator<`**：对自定义类型的对象，直接定义默认的 `<` 比较。
2. **定义比较函数**：编写全局函数，传入函数指针。
3. **定义函数对象**：重载 `operator()` 的类，可带状态。
4. （C++11）**lambda 表达式**：直接内联写匿名函数。

**【解题步骤】**
- 确定排序或查找的规则。
- 选择合适的方式实现比较器：
  - 若规则固定且无额外状态，优先使用 lambda 或普通函数。
  - 若需要复用比较器并携带参数，用函数对象。
  - 若只需要改变某一次排序，使用 lambda。
- 传递给 STL 算法或容器（如 `map<Key, Value, Comp>`）。

**【易错点】**
- 严格弱序：比较函数必须满足非自反性、非对称性和传递性。例如 `comp(a,a)` 必须返回 `false`。常见错误：`return a <= b` 会导致 `a <= a` 返回 `true`，违反严格弱序。
- 函数对象作为模板参数：对于 `map` 等容器，需要在模板参数中传入比较器的类型，构造函数中传入实例。例如 `map<People, int, AgeComp> m;`，如果 `AgeComp` 是类名，默认构造即可。
- 忘记 `const`：`operator()` 必须声明为 `const` 成员，否则无法被只接受常引用比较器的算法使用。

**【例题】**

**例题**：
**题目**：分别用三种方法对 `vector<People>` 按年龄升序排序。 [来源 L11-iostream&函数对象和智能指针-2026.pptx P43-45]
**思路**：方法一：重载 `People::operator<`；方法二：定义比较函数 `compByAge`；方法三：定义函数对象 `AgeComp`。
**解答**：
方法一——重载 `operator<`：
```cpp
class People {
public:
    int age, weight;
    bool operator<(const People &b) const { return age < b.age; }
};
vector<People> vec = {{18,50}, {16,40}};
sort(vec.begin(), vec.end()); // 使用默认 <
```
方法二——函数指针：
```cpp
bool compByAge(const People &a, const People &b) { return a.age < b.age; }
sort(vec.begin(), vec.end(), compByAge);
```
方法三——函数对象：
```cpp
class AgeComp {
public:
    bool operator()(const People &a, const People &b) const { return a.age < b.age; }
};
sort(vec.begin(), vec.end(), AgeComp());
```
**技巧**：需要多套排序规则时，应避免独占 `operator<`，优先使用后两种方式，以防止默认规则冲突。

#### std::function类型擦除机制 【必背】

**【定义】**
`std::function` 是一个通用多态函数包装器，定义在 `<functional>` 头文件中。它可以存储、复制、调用任何满足指定签名的可调用目标：函数、函数对象、lambda 表达式、成员函数指针等。其实现依赖类型擦除（Type Erasure）技术。

**【原理与推导】**
类型擦除的核心思想是将具体类型的信息隐藏在一个统一的接口后面。`std::function<Ret(Args...)>` 内部通常会持有：
- 一个基类（如 `callable_base`），定义了虚函数 `invoke` 用于调用目标；
- 针对每种具体可调用对象类型，派生出一个模板子类，实现 `invoke`；
- 通过一个基类指针（或类似机制）指向实际的派生对象，在调用时进行动态分发。

这样，`std::function` 只需要知道返回值类型和参数列表，不需要知道被包装对象的静态类型，从而可以在运行时统一处理不同类型的可调用实体。

使用示例：
```cpp
#include <functional>
std::function<string()> f;
f = readFromScreen;        // 函数指针
f = ReadFromFile();        // 函数对象
string result = f();       // 调用
```

**【解题步骤】**
1. 确定可调用实体的签名：返回类型 `Ret` 和参数类型 `Args...`。
2. 声明 `std::function<Ret(Args...)>` 对象。
3. 通过赋值或构造函数将可调用对象存入其中。
4. 通过 `operator()` 触发调用。

**【易错点】**

| 易混概念 | 模板参数 | std::function |
|---------|---------|---------------|
| 确定时机 | 编译期 | 运行期（通过类型擦除） |
| 运行开销 | 无额外开销（编译期内联优化） | 有虚函数调用或间接调用开销 |
| 可统一种类的类型 | 同一模板实例化只能处理确定类型，但可通过模板参数泛化 | 可以接受任何匹配签名的可调用对象 |

- 错误：试图用 `std::function` 完全替代模板以获得灵活性，忽略了性能损失。
- 内存占用：`std::function` 可能使用堆分配来存储较大的可调用对象（若大于内部缓冲区，通常一个指针大小以上）。

**【例题】**

**例题**：
**题目**：解释为什么 `string(*readFunc2)(); readFunc2 = ReadFromFile();` 编译失败，而 `std::function<string()> readFunc; readFunc = ReadFromFile();` 成功。 [来源 L11-iostream&函数对象和智能指针-2026.pptx P53]
**思路**：函数指针类型必须严格匹配函数的静态类型，而 `ReadFromFile()` 是一个类的实例，不是函数。`std::function` 通过类型擦除抹去了具体类型差异。
**解答**：
函数指针 `string(*)(void)` 只能指向具有完全匹配签名的函数。`ReadFromFile` 是一个类类型，其对象不能隐式转换为函数指针。因此赋值非法。
`std::function<string()>` 实现了类型擦除，内部会构造一个适配器，使得 `ReadFromFile` 的 `operator()` 调用与 `string()` 接口匹配，因此赋值合法。
**技巧**：如需将不同的可调用对象统一保存（如存于容器、作为回调），使用 `std::function` 消除类型差异。

#### 模板、虚函数与std::function实现多态的对比 【必背】

**【定义】**
在 C++ 中实现多态有三种典型方式：
1. **虚函数多态**：通过基类指针或引用调用派生类重写的虚函数。
2. **模板（编译期多态）**：通过模板参数传递不同类型，在编译期间生成对应版本的函数。
3. **std::function 多态**：利用类型擦除，在运行期统一包装不同类型可调用对象。

**【原理与推导】**
- **虚函数**：基于继承和虚函数表（vtable），运行时通过查表决定调用哪个函数。需要预先定义基类接口，所有派生类必须继承自该基类。
- **模板**：在编译时根据传入参数的实际类型实例化函数或类，不同实例可调用对应类型的成员函数，无需共同基类。实现的是静态多态（鸭子类型）。
- **std::function**：通过类型擦除将调用逻辑封装在内部，向外暴露统一的函数签名接口。在运行时通过间接调用实现多态，无需使用者关心具体类型。

三者对比如下：

| 特性 | 虚函数 | 模板 | std::function |
|------|--------|------|---------------|
| 多态时机 | 运行期 | 编译期 | 运行期（类型擦除） |
| 是否需要公共基类 | 是 | 否（鸭子类型） | 否（只需签名匹配） |
| 运行时开销 | 一次虚函数查表 | 无（内联优化） | 间接调用，可能涉及堆分配 |
| 灵活性 | 受限于继承体系 | 非常高，但类型需在编译期确定 | 运行时灵活组合不同类型的可调用实体 |
| 代码膨胀 | 小 | 易致代码膨胀（大量实例化） | 较小 |

**【解题步骤】**
在需要回调或多态行为的场景，根据需求选择：
- 如果多态类型的数量在编译期就已固定，且对性能要求极高，使用模板。
- 如果多态类型需要运行时动态改变（如通过配置文件动态加载处理模块），且可以承担一定开销，使用 `std::function` 或虚函数。
- 如果已经有完善的类层次结构，且需要扩展性强，使用虚函数。

**【易错点】**
- 误以为 `std::function` 是模板，运行时零开销。实际上它通常会引入动态内存分配和间接调用。
- 模板在调用不同函数对象时，如果函数对象类型不同，会实例化出多份代码，可能导致可执行文件体积增大。
- 用 `std::function` 作为函数参数可能导致不必要的类型擦除开销，当不需要运行时多态时，应优先使用模板参数。

**【例题】**

**例题**：
**题目**：对比以下三种实现 `process` 函数的方式，说明各自的优缺点和使用场景。 [来源 L11-iostream&函数对象和智能指针-2026.pptx P47-55]
```cpp
// 方式一：虚函数
class CalculatorBase {
public:
    virtual string read() = 0;
    virtual string calculate(string) = 0;
    virtual void write(string) = 0;
    void process() {
        string data = read();
        string output = calculate(data);
        write(output);
    }
};

// 方式二：模板
template<class ReadFunc, class CalFunc, class WriteFunc>
void process(ReadFunc r, CalFunc c, WriteFunc w) {
    string data = r();
    string output = c(data);
    w(output);
}

// 方式三：std::function
void process(function<string()> r,
             function<string(string)> c,
             function<void(string)> w) {
    string data = r();
    string output = c(data);
    w(output);
}
```
**思路**：从多态时机、代码重用、性能、灵活性角度分析。
**解答**：
- **虚函数版**：需要继承体系，运行时多态。适合有一族具有相同接口的类，且运行时需要动态绑定。扩展需新增子类，不修改 `process`。
- **模板版**：编译期多态，无需继承，灵活支持任何具有对应调用能力的类型（函数指针、函数对象等）。零运行时开销，但会造成模板实例化膨胀，且不同类型不能放入统一容器。
- **std::function 版**：运行时类型擦除，兼具灵活性，可统一存储不同可调用对象，但引入了间接调用和内存分配开销。适合需要动态变换回调，但对性能不极敏感的场景。
**技巧**：先考虑模板实现高性能，若需动态存储多种回调则引入 `std::function`，若已有完善的类体系则用虚函数。
