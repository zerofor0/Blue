### 六、核心机制深度回顾

#### 重载（Overload）、重写覆盖（Override）与重写隐藏（Redefining）辨析 【必背】

##### 1. 三个概念的定义

**【定义】重载（Overload）**
在**同一作用域**（同一个类内，或同为全局函数），函数名相同，但**参数列表（类型或个数）必须不同**，返回值可以相同或不同。重载与虚函数无关，编译器通过参数类型在编译期决议。

**【定义】重写覆盖（Override）**
派生类重新定义基类中的**虚函数**，函数名、参数列表、返回值（一般情况下）**必须完全相同**。派生类虚函数表中原本指向基类虚函数的位置会被**覆盖**为派生类版本，从而通过基类指针/引用调用时实现多态。

**【定义】重写隐藏（Redefining）**
派生类定义了与基类**同名的函数**，但满足下列条件之一：
- 基类同名函数**不是虚函数**；
- 即使基类函数是虚函数，但**参数列表不同**。
此时派生类的函数把基类中**所有同名函数**（无论参数是否相同）都隐藏起来，在派生类对象上无法直接调用基类的同名函数，只能通过 `基类::函数名` 显式调用。

##### 2. 对比表

| 特性 | 重载 (Overload) | 重写覆盖 (Override) | 重写隐藏 (Redefining) |
|------|----------------|---------------------|----------------------|
| 作用域 | 同一作用域（同一个类/全局） | 基类-派生类之间 | 基类-派生类之间 |
| 基类函数是否为虚 | 无要求 | 必须是虚函数 | 不是虚函数，或参数不同 |
| 函数名 | 必须相同 | 必须相同 | 必须相同 |
| 参数列表 | 必须不同 | 必须相同（协变返回值除外） | 必须不同，或基类非虚 |
| 虚函数表影响 | 无关 | 派生类虚表中对应条目被覆盖 | 不覆盖，基类虚表项保留 |
| 派生类对象能否直接调用基类版本 | 视作用域而定 | 被隐藏，需 `Base::foo()` | 被隐藏，需 `Base::foo()` |
| 决议方式 | 编译期（静态绑定） | 运行期（动态绑定） | 编译期（静态绑定） |

##### 3. 原理与推导
- **重写覆盖改变虚函数表**：当派生类重写基类虚函数时，编译器将派生类虚函数表（vtable）中相应的条目替换为派生类函数地址；因此通过基类指针调用时，会查表找到派生类的版本。
- **重写隐藏不改变虚函数表**：隐藏发生时，派生类同名函数与基类的虚函数表无关，它只是在派生类作用域中“屏蔽”了基类的名字。基类的虚函数表中原有条目仍然指向基类版本；但如果用基类指针调用，依然可能调用到基类版本（除非派生类的同名非虚函数参数完全不同，且未覆盖虚表）。
- **隐蔽规则额外提醒**：一旦派生类中出现与基类同名的任何成员函数，基类中**所有同名函数**（无论是否为虚、无论参数）均在派生类作用域被隐藏，除非使用 `using` 声明把它们引入派生类作用域。

##### 4. 【解题步骤】如何识别重载、覆盖、隐藏
1. 检查作用域：如果在**同一个类**或同为自由函数，且函数名相同、参数不同 → 重载。
2. 如果在派生类和基类之间，检查基类同名函数是否为 `virtual`：
   - 若**不是**虚拟函数 → 一定为重写隐藏（即使参数相同）。
   - 若是虚拟函数，检查参数列表是否完全相同：
     - 参数**完全相同** → **重写覆盖**。
     - 参数**不同** → **重写隐藏**（即使基类函数是虚函数）。
3. 分析代码时，注意“隐藏”会导致派生类对象无法直接调用基类同名函数，而要通过 `Base::func()` 或基类指针/引用。

##### 5. 【易错点】
- **误将隐藏当作覆盖**：基类有虚函数，但派生类写错了参数类型（例如 `foo(float)` 覆写 `foo(int)`），这实际是重写隐藏，不会产生多态。使用 `override` 关键字可以让编译器主动报错。
- **隐藏的“全部屏蔽”效应**：在派生类中定义了与基类同名的函数后，基类所有同名重载版本都被隐藏。如果希望保留部分重载，需要用 `using Base::foo;` 引入。
- **返回值协变是覆盖的特例**：如果基类虚函数返回 `Base*`，派生类重写返回 `Derived*`，仍属于重写覆盖（协变返回类型），但覆盖条件仍要求参数相同。

##### 6. 例题

**例题**：辨析覆盖与隐藏，并测试 `override` 关键字效果  
**题目**：[L8-虚函数-2026.pptx P34-P37]
```cpp
class Base {
public:
    virtual void foo() { cout << "Base::foo()" << endl; }
    virtual void foo(int) { cout << "Base::foo(int)" << endl; }
    void bar() {}
};

class Derived1 : public Base {
public:
    void foo(int) { cout << "Derived1::foo(int)" << endl; }   // ①
};

class Derived2 : public Base {
public:
    void foo(float) { cout << "Derived2::foo(float)" << endl; } // ②
};

class Derived3 : public Base {
public:
    void foo(int) override { cout << "Derived3::foo(int)" << endl; } // ③
    // void foo(float) override {}  // 编译错误：参数不同不是重写
};
```
以及 main 中的调用：
```cpp
Derived1 d1; Derived2 d2;
Base* p1 = &d1; Base* p2 = &d2;
p1->foo();          // ④
p2->foo();          // ⑤
d1.foo(3);          // ⑥
d2.foo(3.0);        // ⑦
p1->foo(3);         // ⑧
p2->foo(3.0);       // ⑨
```
请判断 ①–③ 各属于重载、覆盖、隐藏中的哪一种？写出 ④–⑨ 的输出结果。

**思路**：先根据基类函数是否虚、参数是否相同判定①、②、③的性质，再结合虚函数表与隐藏规则分析多态调用和直接调用的行为。注意隐藏会导致派生类对象无法直接调用基类同名的无参版本 `foo()`。

**解答**：
- ① 基类 `foo(int)` 是虚函数，派生类参数完全相同 → **重写覆盖**。
- ② 基类 `foo(int)` 是虚函数，但派生类参数为 `float`，不同 → **重写隐藏**。且派生类没有定义其它 `foo`，所以隐藏了基类的所有 `foo`。
- ③ 使用 `override` 显式声明，与①相同 → 重写覆盖。

运行时：
- ④ `p1->foo()`：由于 `Derived1` 重写覆盖了 `foo(int)` 而未定义 `foo()`，虚函数表里 `foo()` 仍然是继承自 `Base` 的版本，所以输出 `Base::foo()`。
- ⑤ `p2->foo()`：`Derived2` 隐藏了基类所有 `foo`，但其对象内的虚函数表仍然保存了基类 `foo()` 的地址，通过基类指针调用 `foo()` 会查表得到 `Base::foo()`，输出 `Base::foo()`。
- ⑥ `d1.foo(3)`：直接调用派生类自己的 `foo(int)`，输出 `Derived1::foo(int)`。
- ⑦ `d2.foo(3.0)`：调用派生类的 `foo(float)`，输出 `Derived2::foo(float)`。
- ⑧ `p1->foo(3)`：基类指针调用虚函数，由于 `Derived1` 重写覆盖了 `foo(int)`，虚表中指向 `Derived1::foo(int)`，输出 `Derived1::foo(int)`。
- ⑨ `p2->foo(3.0)`：实参 `3.0` 是 `double`，但被隐式转换为 `int` 以匹配 `Base::foo(int)`；`Derived2` 并未覆盖虚表中 `foo(int)` 的条目，故调用 `Base::foo(int)`，输出 `Base::foo(int)`。

**技巧**：`override` 仅用于检查，不改变运行结果；判断隐藏/覆盖关键看“参数+virtual”是否完全匹配。基类指针访问虚函数只看虚表，不受派生类作用域隐藏影响。

#### 构造函数与析构函数的虚函数行为 【高频】

##### 1. 虚函数表（VPTR）与对象内存布局（预备）

**【定义】** 每个含有虚函数的类或派生类，编译器为其生成一个**虚函数表（vtable）**，其中存放该类所有虚函数的地址。每个对象内部隐含一个指向该表的指针 **VPTR**（通常占一个机器字长）。
**【必有】** 对象大小 = 非静态数据成员大小（加上内存对齐） + 一个 VPTR 的大小（若有虚函数）。多个虚函数只增加虚表条目，不增加对象大小，因为对象只存一个指针。

**示例**：[L8-虚函数-2026.pptx P21-P22]
```cpp
class NoVirtual  { int a; };                    // sizeof = 4（仅int）
class OneVirtual { int a; virtual void f1(); }; // sizeof = 12（int 4 + VPTR 8)
class TwoVirtual { int a; virtual void f1(); virtual int f2(); }; // sizeof = 12
```
在 64 位系统 `void*` 为 8 字节，加上对齐，出现 12 的结果。

##### 2. 构造函数与虚函数

**【定义】构造函数不能是虚函数，也不必是虚函数。**
- **不能**：创建对象时需要知道对象的完整类型以设置 VPTR，而 VPTR 的初始化发生在构造函数执行前（由编译器插入），若允许构造函数为虚，就需要在 VPTR 设置之前通过虚机制调度，这形成逻辑悖论。
- **不必**：构造函数的作用是初始化该具体类的数据成员，调用时类的类型是明确的，不需要多态。

**【核心规则】在构造函数（和析构函数）中调用虚函数时，虚机制不工作，实际调用的是当前类的“本地版本”。**
- **原理**：C++ 中基类子对象先于派生类部分初始化。执行基类构造函数时，派生类的数据成员尚未初始化，对象的动态类型尚不完整。为保证安全，编译器将在构造/析构期间将对象的动态类型“锁定”为正在构造/析构的类，因此虚函数调用被静态绑定到当前类的版本。
- **例子**：[L8-虚函数-2026.pptx P24]
```cpp
class Base {
public:
    virtual void foo() { cout << "Base::foo" << endl; }
    Base() { foo(); }   // 构造函数内调用虚函数
};
class Derived : public Base {
    int _num;
public:
    void foo() override { cout << "Derived::foo" << _num << endl; }
    Derived(int j) : _num(j) {}
};
```
执行 `Derived d(0);` 时，先执行 `Base` 构造，此时 `Derived::_num` 尚未初始化。`Base()` 中的 `foo()` 调用的是 `Base::foo`，而不是派生类版本——避免访问未初始化的 `_num`。

##### 3. 析构函数与虚函数

**【定义】析构函数可以是虚函数，且常常应该是虚函数。虚析构函数仍必须提供函数体。**
- **为什么要是虚函数**：当通过基类指针 `delete` 一个派生类对象时，如果基类析构函数不是虚函数，则只会调用基类的析构函数，派生类部分无法正确析构，造成资源泄漏。
- **示例**：[L8-虚函数-2026.pptx P27-P28]
```cpp
class Base1 { public: ~Base1() { cout << "~Base1()\n"; } };          // 非虚析构
class Derived1 : public Base1 { public: ~Derived1() { cout << "~Derived1()\n"; } };

class Base2 { public: virtual ~Base2() { cout << "~Base2()\n"; } };   // 虚析构
class Derived2 : public Base2 { public: ~Derived2() { cout << "~Derived2()\n"; } };
```
测试：
```cpp
Base1* bp = new Derived1; delete bp;   // 只输出 ~Base1()，Derived1 析构未调用
Base2* b2p = new Derived2; delete b2p; // 输出 ~Derived2() 再 ~Base2()
```
- **在析构函数中调用虚函数**：与构造函数类似，虚机制不工作。析构时派生类部分已先析构，对象的动态类型变为当前析构的类，故虚调用被静态绑定到当前类的版本，防止访问已销毁的派生类成员。

**【重要原则】总是将基类的析构函数声明为虚析构函数**，除非确定该类不会被用作多态基类。

##### 4. 【解题步骤】分析构造/析构中虚函数调用
1. 判断构造或析构执行时对象的“当前类”。
2. 记住虚机制在构造/析构中失效，调用的版本就是当前类中定义的函数（如果有），若无则按普通编译查找。
3. 析构顺序与构造相反：先析构派生类自身部分，然后成员对象，最后基类部分。在派生类析构函数体执行后，基类析构中不能再依赖派生类成员。
4. 对于 `delete` 基类指针，检查基类析构是否为虚，决定是否完整清理派生类资源。

##### 5. 【易错点】
- **构造函数误用为虚函数**：编译错误或概念错误。
- **忘记将基类析构函数设为虚函数**：导致 `delete` 基类指针时派生类析构未执行 → 内存泄漏（常见于多态容器）。
- **认为构造/析构中虚函数会多态**：实际上它们调用的是“当前类”版本，可能产生非预期结果，尤其当派生类重写了虚函数并依赖派生类成员时，会出现未初始化或已销毁成员的使用风险。
- **多重继承时构造顺序与虚表切换**：更复杂，但基本原则不变——每个基类子对象构造期间对象的动态类型为该基类。

##### 6. 例题

**例题**：虚析构的必要性辨析  
**题目**：[L8-虚函数-2026.pptx P29 选择题改编]
下列关于虚函数的说法不正确的是（ ）。  
A. 构造函数不可以是虚函数  
B. 析构函数可以是虚函数  
C. 类中的虚函数可能并不显式地带有 virtual 声明  
D. 一个类中的虚函数越多，其对象的大小就越大

**思路**：逐一分析选项，结合 VPTR 原理和虚函数表知识。C 项因为派生类重写虚函数时 virtual 可省略但仍为虚；D 项多个虚函数只增加虚表条目，对象大小不变。

**解答**：不正确的是 **D**。  
- A 正确，构造函数不能是虚。  
- B 正确，析构函数可以是虚，且应该是虚。  
- C 正确，派生类重写基类虚函数时可不写 `virtual` 关键字，但函数仍是虚函数（继承虚属性）。  
- D 错误，带虚函数的对象只多出一个 VPTR 的大小，与虚函数个数无关；多个虚函数对象大小与只有一个虚函数时相同（见 OneVirtual 和 TwoVirtual 的示例）。

**技巧**：对象大小只由非静态成员和 `#pragma pack` 对齐决定，外加一个可能的 VPTR；虚函数数量不影响 `sizeof`。

**例题**：构造/析构中虚函数调用陷阱  
**题目**：[L8-虚函数-2026.pptx P24 代码运行行为]
已知类定义：
```cpp
class Base {
public:
    virtual void foo() { cout << "Base::foo" << endl; }
    Base() { foo(); }
    void bar() { foo(); }
};
class Derived : public Base {
    int _num;
public:
    void foo() override { cout << "Derived::foo" << _num << endl; }
    Derived(int j) : _num(j) {}
};
```
写出如下代码的输出，并解释。
```cpp
Derived d(0);
Base &b = d;
b.bar();
b.foo();
```

**思路**：`Derived d(0)` 构造时先执行 `Base` 构造，调用 `foo()` 时虚机制不工作，调用 `Base::foo`。之后通过引用调用 `bar()`（非虚函数，但内部 `foo()` 是虚调用，此时对象完整，虚机制工作），以及直接调用虚函数 `b.foo()`。

**解答**：
- 构造 `d(0)` → `Base()` 内 `foo()` 输出 `Base::foo`。
- `b.bar()` → `bar()` 内 `this->foo()` 是虚调用，此时动态类型为 `Derived`，输出 `Derived::foo0`（构造时 `_num` 已由初始化列表初始化为 0）。
- `b.foo()` → 直接虚调用，输出 `Derived::foo0`。

最终输出：
```
Base::foo
Derived::foo0
Derived::foo0
```

**技巧**：构造函数体内的虚调用不“多态”；但构造结束后，对象的动态类型恢复正常。`_num` 由初始化列表在 `Base` 构造之前就完成初始化，所以 `Derived::foo` 输出时 `_num` 为合法值 0。
