# 复习笔记智能助手 - 系统设计方案

> 目标：用户上传一门课程的 PPTX / PDF / DOCX / 图片，助手自动提取全部文字，整理输出一份**结构清晰、零遗漏、带知识网络图和例题**的 Markdown 复习笔记。
>
> 第一性原则：**任何内容都不得因长度被截断或丢弃**。整个架构围绕这一原则设计。

---

## 〇、总体架构（Map-Reduce-Merge 流水线）

```
1. 文件内容提取（逐文件、逐页，带来源溯源）
   PPTX / PDF / DOCX / 图片  ->  统一 Record 列表

2. 内容组织与去重（建章节骨架 + 归并 + 近似去重）
   Records  ->  按章节分桶的 Records

3. 长文本分片（按章节切，超大章节递归再切，带 overlap）
   ->  Chunk 列表（每个 Chunk 附 token 预算）

4. 核心整理（Map：每个 Chunk 独立调用大模型，结构化输出）
   ->  每片输出 {concepts, examples} 的 JSON

5. 知识合并（Reduce：实体消解 + 图合并 + 例题去重）
   ->  统一的知识图谱 + 例题库

6. 最终输出（知识网络图 + 知识点详解 + 试题附录）
   ->  Markdown 复习笔记
```

**为什么是 Map-Reduce-Merge：**
- **Map**：分片后每片独立调模型，单片内容远小于上下文窗口，绝不会截断；
- **Reduce/Merge**：把各片结果拼回一个全局知识网络，损失由合并算法补齐；
- 整条流水线**先提取全部文本到磁盘**，再分片喂模型——提取阶段不受上下文限制，是"零遗漏"的根本保证。

## 一、统一数据结构（贯穿全流程）

所有提取出的内容都表示为带**来源溯源**的 `Record`：

```jsonc
{
  "id": "rec_000042",            // 全局唯一、有序
  "source_file": "操作系统.pptx",
  "source_type": "pptx",          // pptx | pdf | docx | image
  "page": 12,                     // 页码 / 幻灯片号 / 图片序号
  "chapter": "进程管理",          // 第②阶段推断出的章节（初始可为 null）
  "title": "进程的三种基本状态",   // 可为空
  "content": "进程在运行过程中...",// 正文文本
  "is_problem": false,            // 第①阶段做粗判，第④阶段细判
  "tokens": 86                    // 该条目的 token 数（用于预算）
}
```

> **关键设计：`source_file` + `page` 永远跟着内容走。** 它是第六阶段"试题附录"能精确追溯到原始资料的基石，也是用户验证"有没有遗漏"的抓手。

## 二、① 文件内容提取

针对四种格式分别处理，但输出都归一为上面的 `Record`。**提取阶段没有上下文限制，必须把全文落地，绝不截断。**

### 2.1 PPTX（推荐 `python-pptx`）
- 遍历每张幻灯片：`title_placeholder` -> `title`；`body_placeholders` 的文本框 -> `content`（按段落拼接）。
- **务必提取"演讲者备注（notes_slide）"**——备注里常常有正文没写的完整解释和例题。
- 提取表格、SmartArt 文本。
- 一页幻灯片 = 一条 `Record`（`page` = 幻灯片号）。

### 2.2 PDF（推荐 `PyMuPDF/fitz`，表格用 `pdfplumber`）
- 逐页 `page.get_text()`，一页文本 = 一条 `Record`。
- **标题层级探测**：用 `pdfplumber` 读字体大小，比正文大且加粗的行判为标题，用于辅助章节归并。
- **扫描版/图片型 PDF 兜底**：若某页 `get_text()` 返回空或字符极少（< 阈值），判定为扫描页，将该页渲染为图片走 2.4 的 OCR 分支。**这一步是 PDF"零遗漏"的关键防线。**

### 2.3 DOCX（推荐 `python-docx`）
- 遍历段落，按 `paragraph.style.name`（Heading 1/2/3...）建立**标题层级**，正文段落挂到最近的上级标题下。
- 提取表格（按行拼接为文本，保留表头语义）。
- 一段 = 一条 `Record`，`title` 字段记录其所属标题链（如 `"第三章 > 3.2 调度算法"`）。

### 2.4 图片（推荐 `PaddleOCR`，中文场景优于 Tesseract）
- 对每张图片先做"文字/纯插图"判定（文字框数量、置信度）。
- OCR 返回的文本块按**坐标位置重排**成阅读顺序（左上->右下，先列后行），再合并为该图的 `content`。
- 一图 = 一条 `Record`，`page` = 图片在材料包中的序号。

### 2.5 提取阶段的健壮性
- 文件加密 / 损坏：跳过并写入日志，**不中断**整批处理。
- token 计数用 `tiktoken`（或目标模型对应的 tokenizer）预计算到 `tokens` 字段，供第三阶段预算用。

## 三、② 内容组织与去重

### 3.1 建立章节骨架（以 PPT 为主）
1. **首选骨架**：汇总所有 PPTX 的幻灯片标题，做一次"合并同义标题 + 推断章节边界"的轻量大模型调用（输入只是标题列表，体积小，放得下）。
2. **次选**：若 PPT 无清晰结构，用 DOCX 的 Heading 层级。
3. **兜底**：都没有就调一次模型读全部 `Record` 的标题/首行，生成目录树。

### 3.2 把每条 Record 归入章节
- 用**标题关键词 + 内容 embedding 余弦相似度**把每条 `Record` 匹配到最贴近的章节；都低于阈值则进"未归类"桶（最后由模型再判一次）。

### 3.3 去重（三层，从快到准）
| 层级 | 方法 | 阈值 | 用途 |
|------|------|------|------|
| 精确 | 归一化（去空白/标点/全半角）后哈希 | 完全相等 | 抄录重复 |
| 近似 | MinHash / SimHash（Jaccard） | >= 0.85 | 改写、换序 |
| 语义 | 句向量余弦相似度 | >= 0.92 | 同义转述 |

- 命中重复时**保留信息最完整的一条**，合并各条的 `source_refs`（多来源恰好说明该点重要，可加权 `importance`）。

## 四、③ 长文本分片（核心：保证不超上下文、不遗漏）

**核心原则：按"章节"切，而不是按固定字符窗切。** 按章节切能让每片语义自洽、合并时边界清晰。

```
章节 token 预算 B = 上下文窗口 − 系统指令 − 输出预留 − 容差
（例如 200k 窗口，B 取 ~120k；8k 小模型，B 取 ~5k）
```

分片算法：
```
for chapter in chapters:
    if chapter.total_tokens <= B:
        生成 1 个 Chunk
    else:
        按子标题 / 幻灯片边界 二分递归切，直到每片 <= B
        相邻片之间保留 ~10% overlap（按段落对齐，不切断段落）
记录每个 Chunk -> 它覆盖的章节、起始/结束 Record 的全局序号
```

**为什么 overlap**：防止一个概念刚好被切在边界上、两片都只看到一半。

**逐片处理的"概念接续"技巧（关键）**：处理第 N 片时，把**前 N−1 片已提取出的概念名 + 别名清单**作为上下文一起喂给模型。这样模型在命名新概念时会**优先复用已有概念名**，大幅降低第五阶段"同义概念消解"的难度，从源头减少碎片化。

> **极端兜底**：即便某单个章节递归切到最小段落仍超 B（极罕见），退化为"按句滑窗"提取，先抽要点再合并——仍然不丢内容，只是该章多分几片。

## 五、④ 核心整理逻辑（给大模型的指令）

> **提示词已更新为"考试导向"版本**（见原型 `prompts.py`）。核心理念：**入口无损读取全部资料（绝不截断，否则无法判断什么考什么不考），出口按考试优先整理（复习资料不等于课程资料汇编）**。两者不矛盾，读完是前提，考什么是输出。
>
> 新版系统提示词围绕"考什么 / 怎么考 / 什么最重要 / 什么最易错 / 如何速记"组织，按 5 步流程（识别考试导向 -> 建框架 -> 提炼核心知识 -> 题型对应 -> 总结速查），并用固定排版标记（【定义】/【高频】/【重点】/【必背】/【易错点】/【真题】/【记忆】/【解题步骤】/ 公式编号）+ 学科适配（理工/计算机/经管/文史哲）。下面的 JSON Schema 为"完整版"的知识结构化方案；原型阶段为降低复杂度，采用 Markdown+标记输出并对标记做轻量解析以生成汇总。

每片调用大模型时，**强制结构化输出（JSON Schema）**，让后续合并可以机械化执行。

### 5.1 输出 Schema

```jsonc
{
  "concepts": [
    {
      "name": "进程",
      "aliases": ["process", "程序执行实例"],
      "definition": "程序在某一数据集合上的一次执行过程，是系统进行资源分配和调度的基本单位。",
      "chapter": "进程管理",
      "importance": 5,                 // 1-5（数值越大越重要）
      "importance_reason": "贯穿全章的基础概念，往年题高频",
      "relations": [
        {"target": "线程",   "type": "contains",    "desc": "一个进程包含至少一个线程"},
        {"target": "程序",   "type": "contrast",    "desc": "程序是静态的指令集合，进程是动态的执行过程"}
      ],
      "source_refs": [{"file": "操作系统.pptx", "page": 12}]
    }
  ],
  "examples": [
    {
      "concept": "进程状态转换",
      "question": "处于就绪态的进程能否直接转换为阻塞态？为什么？",
      "answer": "不能。就绪态缺少 CPU，只有获得 CPU 进入运行态后...",
      "is_original": false,           // false=来自原始资料，true=模型自编
      "source_refs": [{"file": "往年题.pdf", "page": 3}]
    }
  ]
}
```

- **关系类型枚举**（对应需求的四种关系）：
  `contains`(包含) / `depends_on`(依赖) / `parallel_to`(并列) / `contrast_with`(对比)
- **重要度评分细则**（写进 prompt，避免模型乱打分）：
  - 出现在往年题/例题中 -> 标【必背】
  - 是本章其他概念的前提/被多处引用 -> 提一级
  - 定义性、定义型基础概念 -> 基线标【重点】
  - 仅一笔带过的边缘名词 -> 仅在汇总里提及，不标重点
- **例题策略**：
  1. 先扫描该片中标记为"试题/例题/习题"的 `Record`，匹配到对应概念，`is_original=false`，并**原样保留**来源页码；
  2. 对核心概念（importance>=4）若资料里没有现成题，**自编一道**难度匹配的，`is_original=true`，并在 answer 里点明考察点。

### 5.2 Prompt 模板（精简版）

```
你是复习笔记整理专家。下面是《{课程名}》第"{章节}"章的一段原始学习资料（来自多个文件，已去重）。
请严格按给定 JSON Schema 输出，不要有任何额外文字。

要求：
1. 概念识别：找出本段所有核心概念，给出精确的一句话定义（不得照抄原文错漏处，需补全）。
2. 关系梳理：对每个概念给出它与其它概念的关系，type 只能取 contains/depends_on/parallel_to/contrast_with，并用一句话说明依据。
3. 重要度：按评分细则给 1-5 星并说明理由。
4. 例题：先用资料里现成的试题（标注来源页码）；核心概念若没有现成题则自编一道。
5. 命名一致性：下列概念名已被前文采用，若指同一对象请沿用同名：{已有概念名+别名清单}

[已有概念清单]
...

[本段资料]
{chunk_content}
```

## 六、⑤ 知识合并（Reduce）

把各片的 `{concepts, examples}` 合并为**全局唯一的**知识网络。

### 6.1 实体消解（同名/同义概念合并）
构造候选图，节点 = 概念名 + 所有别名。聚簇规则（满足任一即归为同一概念）：
1. 名称或别名**完全相同**（大小写/全半角归一后）；
2. 定义句向量**余弦相似度 >= 0.88**；
3. 同一别名被多个概念共用。

**簇合并规则**：
- 取最常出现/最清晰的名字为 `canonical_name`，其余进 `aliases`；
- 定义取最长最完整的一条（或调一次模型融合两三条）；
- `importance` 取 **max**，`source_refs` 取**并集**；
- `relations` 取**并集**（边指向别名时改指向 canonical）。

### 6.2 关系/图合并与去重
- 边 = `(source, target, type)`。完全相同的边只留一条，`desc` 取最详尽者。
- 同一对节点出现**冲突类型**（如既有 contains 又有 contrast）：多数表决，平票时调一次模型裁决。
- **跨片段关联补全**（可选但推荐）：对同一章节内仍未相连的概念对，做一轮轻量"是否相关"判别，补上被切片边界隔断的关系。

### 6.3 例题去重
- 按题干文本做近似去重（MinHash >= 0.8 视为重复），保留一条，`source_refs` 并集。
- 每个核心概念最终**最多保留 1 道主例题**（优先选来自真题的），其余进附录。

## 九、关键设计点回顾（为什么能"零遗漏"）

1. **提取与推理分离**：全文先落盘为 `Record`，提取不受上下文限制；模型只看分片。
2. **按章节分片 + 超大章节递归切 + overlap**：任何体积的材料都能被完整覆盖，边界不丢概念。
3. **逐片"概念接续"**：把已有概念名喂给后续片，从源头压低同名碎片。
4. **结构化 JSON 输出**：合并可机械化、可校验、可追溯，不依赖模型自由文本。
5. **来源溯源贯穿全程**：每个概念、每道题都能点回原始文件页码，用户可一键核验"有没有漏"。
6. **扫描页 OCR 兜底**：防止"看起来提取了实则空白"的隐性遗漏。
```
